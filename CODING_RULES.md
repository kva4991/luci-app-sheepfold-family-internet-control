# Правила кодинга Sheepfold

Этот документ обязателен для всех участников проекта: разработчиков, ревьюеров, контрибьюторов и AI-инструментов. Он применяется ко всему репозиторию, если в конкретном каталоге не появился более строгий локальный документ.

Правила сформированы на основе принятого стиля другого проекта и адаптированы к текущему стеку Sheepfold: OpenWrt shell, LuCI JavaScript, Android/Kotlin, конфигурация UCI, CSS, SQL и документация. Разделы PHP/Yii сохраняются как правила на случай появления такого кода в будущем.

## 1. Приоритеты и общий подход

1. Корректность, безопасность и понятность важнее сокращения кода.
2. Перед изменением файла необходимо изучить соседний код и продолжить его стиль, если он не нарушает безопасность и правила этого документа.
3. Старый файл не переписывается целиком ради современного синтаксиса, форматирования или личных предпочтений.
4. Рефакторинг, не требуемый задачей, не смешивается с функциональным исправлением.
5. Не изменять глобальные селекторы, публичные API, форматы конфигурации и структуру данных без явной необходимости и проверки совместимости.
6. Новая функциональность должна быть отделена от legacy-кода понятной границей, а не встраиваться случайными исключениями.
7. Имена должны описывать предметную сущность или действие. Не использовать абстрактные имена `a`, `data`, `item`, `model`, `tmp`, если можно дать точное имя.
8. Одноразовая переменная не создаётся, если она не улучшает читаемость, диагностику или безопасность.
9. Значение, используемое в нескольких проверках или логических шагах, сохраняется под конкретным именем, чтобы исключить расхождение вычислений.
10. При изменении поведения обновляются тесты и документация, если они описывают затронутую функцию.
11. Документация должна чётко различать:
    - реализованное поведение;
    - экспериментальную функцию;
    - план или идею;
    - известное ограничение.
12. Нельзя описывать несуществующий endpoint, экран, фоновую задачу или защиту как уже работающие.

## 2. Версия интерфейсных ресурсов

При изменении любого исходного `*.js` или `*.css` необходимо увеличить версию ресурсов.

В текущем проекте источником версии является:

```make
PKG_RELEASE:=117
SHEEPFOLD_UI_ASSET_VERSION:=$(PKG_VERSION)-$(PKG_RELEASE)
```

Поэтому при изменении `*.js` или `*.css` необходимо:

1. увеличить `PKG_RELEASE` в `package/luci-app-sheepfold-family-internet-control/Makefile`;
2. синхронно обновить значение `ui_asset_version` в `root/etc/config/sheepfold`;
3. не менять `PKG_VERSION`, если изменение не является отдельным продуктовым релизом;
4. не делать отдельное увеличение версии для нескольких JS/CSS-файлов одного логического изменения.

Если в будущем появится PHP-проект с `protected/config/params.php`, для него сохраняется исходное правило: при изменении JS/CSS увеличивать минорную версию, например `75.017 -> 75.018`.

## 3. Именование

### 3.1. Общие правила

- Переменные, функции, методы и ключи объектов писать в `camelCase`, если язык и окружающий код это допускают.
- Классы, интерфейсы, enum и data class писать в `PascalCase`.
- Константы писать в стиле, принятом языком и текущим файлом:
  - shell: `UPPER_SNAKE_CASE`;
  - Kotlin: `UPPER_SNAKE_CASE` внутри `companion object`;
  - JavaScript legacy: сохранять стиль текущего файла.
- Булевы значения называть как утверждение: `isEnabled`, `hasToken`, `canPair`, `shouldRefresh`.
- Коллекции называть во множественном числе: `devices`, `administratorSections`, `archiveFiles`.
- Переменная с экземпляром класса называется по имени класса в lowerCamelCase без лишнего суффикса `Model`:
  - `ExemptionFormSearchModel` -> `exemptionFormSearch`;
  - `RouterConnectionManager` -> `routerConnectionManager`.
- jQuery-объекты называть с префиксом `$`: `$countryId`, `$deviceRow`.
- В shell локальные рабочие переменные использовать в стиле текущего файла. Для глобальных путей и констант использовать точные имена: `TOKEN_STATE_DIR`, `MOUNT_POINT`, `PAIR_LOCK_DIR`.

### 3.2. Запрещённые абстрактные имена

Не использовать без необходимости:

```text
a, b, x, y, data, value, item, obj, model, result, temp
```

Допустимы короткие имена только в очевидной малой области:

- индекс цикла `i`, `j`;
- координаты;
- математическая формула;
- локальный callback на несколько строк, если смысл однозначен.

## 4. Комментарии и todo

1. Комментарии в проектном коде писать на русском языке.
2. После `//` или `#` ставить пробел.
3. Новый или исправляемый комментарий — одно понятное предложение с заглавной буквы без точки в конце.
4. Комментарий объясняет причину, условие, инвариант или следующий логический блок, а не повторяет синтаксис.
5. Перед нетривиальным условным блоком писать комментарий, обычно начиная с «Если».
6. Для простого guard clause, очевидной проверки аргумента или стандартной проверки результата отдельный комментарий не обязателен.
7. Для сложного многострочного условия допускаются комментарии внутри условия перед соответствующими группами проверок.
8. `todo` писать в нижнем регистре и обязательно с номером задачи:

```text
// todo: issue100 Добавить проверку миграции
# todo: issue100 Проверить поведение BusyBox tar
```

9. Не оставлять `todo`, который фактически скрывает критичную ошибку или небезопасный fallback.
10. Комментарии вида «временно», «потом исправить», «магия» без тикета запрещены.

## 5. Условия и управляющие конструкции

### 5.1. Общие правила

- Для `if`, `else`, `for`, `foreach`, `while`, `when`, `switch` использовать блочный стиль языка.
- Даже для одного оператора фигурные скобки обязательны в PHP, JavaScript и Kotlin.
- Однострочный `if` допустим только как короткий guard clause и только если это соответствует стилю файла.
- В одной цепочке `if/elseif/else` не смешивать однострочный и многострочный стиль.
- Части сложного условия группировать по смыслу, а не только по длине строки.
- Сравнения в составном условии заключать в скобки, когда это улучшает однозначность:

```php
if (($cityId > 0) && !is_null($city)) {
```

- Не добавлять лишние скобки вокруг простого булева значения.
- Не писать `== true`, `!= false`, `== 1`, если значение строго булево или нормализовано к `0/1`.
- Для булевого приведения использовать явное средство языка: в PHP `(bool)$value`, а не `!!$value`.
- Тернарный оператор применять только для короткого выбора значения, а не для побочных действий.
- В PHP тернарный оператор всегда оборачивать в скобки:

```php
(($condition) ? $valueA : $valueB)
```

- Если `if/else` только выбирает значение для `return` или одного присваивания, в PHP использовать тернарный оператор, если выражение остаётся читаемым.
- В Kotlin для выбора значения предпочтителен `if` как выражение или `when`.
- В shell сложные цепочки `&&` и `||` не должны скрывать важную ошибку. Для операций с изменением состояния использовать явный `if`.

## 6. Работа с null, пустыми значениями и типами

- `null` писать в нижнем регистре.
- В PHP проверять `null` только через `is_null($value)`.
- Для присваивания значения только при `null` в PHP использовать `??=`.
- В Kotlin использовать nullable-типы, safe call, Elvis и `requireNotNull` по назначению; не использовать `!!` без доказанного инварианта.
- Пустая строка, отсутствие значения, `null`, `0` и `false` не считаются взаимозаменяемыми без явной нормализации.
- В shell входные числовые значения проверять `case` до арифметики.
- Данные из UCI, HTTP, DHCP, ARP, sysfs и внешних команд всегда считать недоверенными.
- Если структура массива или объекта известна, описывать её типом, data class, интерфейсом или PHPDoc array shape.

## 7. Модификаторы доступа и наследование

- Модификаторы доступа делать максимально строгими: `private`, затем `protected`, затем `public` только при необходимости.
- Классы и методы помечать `final`, если наследование или переопределение не предполагается.
- Не ставить `final` на private-методы и внутри `final class`.
- Не открывать внутренний helper как публичный API ради удобства теста; тестировать через предусмотренную границу или вынести отдельный чистый компонент.
- Android `Context` не хранить дольше необходимого; если требуется длительное хранение, использовать `applicationContext`.

## 8. JavaScript и LuCI

1. Если файл старый и написан на `var`, callback и legacy LuCI API, не переписывать его целиком на `const`, `let`, class или стрелочные функции.
2. В новых небольших модулях допустим современный синтаксис, но стили внутри одного логического блока не смешиваются без причины.
3. В новом коде использовать строгие сравнения `===` и `!==`, если текущие данные не требуют legacy coercion.
4. Анонимная функция в legacy-стиле пишется с пробелом:

```javascript
function () {
```

5. Делегированные jQuery-обработчики писать через `.on()`:

```javascript
$('#comments').on('click', '.comment-delete', function () {
```

6. Инициализацию jQuery оформлять как в соседнем коде:

```javascript
$(function () {
```

или

```javascript
$(document).ready(function () {
```

7. AJAX в legacy jQuery-коде выполнять через принятые рядом `$.get()`, `$.post()`, `.done()`, `.fail()`.
8. В LuCI использовать существующие `rpc.declare`, `fs.exec`, `uci`, `ui`, `form` и view-модули вместо ручного обращения к внутренним URL, если для задачи есть стандартный LuCI механизм.
9. Прямой `fs.exec` допускается только для явно разрешённой команды из ACL и с валидированными аргументами.
10. Значения, попадающие из server-side кода в JavaScript, передавать через JSON-сериализацию, а не строковую конкатенацию.
11. DOM-узел не считать существующим без проверки, если он зависит от режима экрана или старой конфигурации.
12. Не полагаться на фиксированные индексы дочерних элементов при удалении или переносе полей, если можно использовать класс, `data-*` или стабильный контейнер.
13. События, таймеры и watchers должны иметь явный способ остановки и не создавать повторные параллельные циклы.
14. Генерация security-sensitive значения должна использовать криптографический источник случайности. Fallback на `Math.random()` для pairing-кода, токена или ключа запрещён.
15. Минифицированные `*.min.js` не читать и не использовать как источник. Минификацию выполнять только если это принято процессом сборки.
16. После изменения любого `*.js` выполнить правило версии ресурсов из раздела 2.

## 9. CSS

1. Не использовать сгенерированные каталоги и опубликованные assets как исходник.
2. Изменять исходный CSS-файл, который действительно подключён сборкой или LuCI view.
3. Новый CSS-блок оформлять многострочно:

```css
.sheepfold-card {
    display: flex;
    gap: 12px;
}
```

4. Старые компактные блоки не форматировать массово без задачи.
5. Секционные комментарии допускаются в стиле:

```css
/*** Настройки ИИ ***/
```

6. Селекторы, связанные с текущей HTML-структурой или Bootstrap/LuCI, менять осторожно.
7. Цвета, размеры, радиусы и отступы повторять из ближайшего контекста; не вводить новую дизайн-систему в рамках локальной задачи.
8. Для изображений использовать относительные пути от CSS-файла.
9. Не менять глобальные селекторы `a`, `img`, `input`, `button`, `input[type="checkbox"]` без явной необходимости.
10. Для анимаций использовать именованный `@keyframes` и проверять совпадение имени в `animation`.
11. После изменения любого `*.css` выполнить правило версии ресурсов из раздела 2.

## 10. Shell и OpenWrt

### 10.1. Базовый стиль

- Для новых исполняемых скриптов использовать `#!/bin/sh` и синтаксис POSIX/BusyBox.
- Не использовать bash-only конструкции: массивы Bash, `[[ ]]`, process substitution, `${var^^}` и другие расширения без гарантированной зависимости.
- Для скриптов, где остановка при ошибке безопасна, использовать:

```sh
set -eu
```

- Если ожидается частичная ошибка внешней команды, обрабатывать её явно, а не отключать `set -e` для всего файла.
- Все пути и значения в shell заключать в кавычки, кроме осознанного word splitting после строгой валидации.
- Не строить команды через `eval`.
- Не передавать непроверенный ввод в `sed`, `awk`, `grep`, `uci`, `mount`, `rm`, `tar`, `openssl` или имя файла.
- Временный файл создавать в том же filesystem, что и итоговый, затем применять `chmod` и атомарный `mv`.
- Для секретов устанавливать `umask 077` до создания файла.
- Для временных файлов и lock-каталогов использовать `trap` для очистки.
- Lock должен иметь ограничение ожидания и понятную ошибку.

### 10.2. UCI

- Читать UCI через `uci -q get` с явным безопасным значением по умолчанию.
- До числового сравнения валидировать значение.
- Не создавать административную сущность как побочный эффект проверки или pairing.
- Изменения нескольких связанных UCI-полей выполнять как одну логическую операцию и делать `uci commit` в контролируемой точке.
- При ошибке после commit предусматривать rollback, revoke или безопасный повтор.
- Не хранить plaintext Bearer-токены, API-ключи или пароли во временных логах и выводе команд.
- Не выводить `uci show` целиком в пользовательский журнал или AI-контекст.

### 10.3. OpenWrt package hooks

- Исполняемые права новых helper-скриптов проверять при установке.
- `postinst` должен одинаково корректно работать в staging root и на реальном роутере.
- При наличии `IPKG_INSTROOT` не запускать сервисы и не изменять host-систему сборщика.
- Изменения crontab делать управляемым маркированным блоком и удалять в `prerm`.
- Не считать `/etc/cron.d` поддерживаемым без подтверждения для целевой сборки OpenWrt.
- Пакетный release увеличивать для любого изменения, которое должно попасть в новый `ipk`.

### 10.4. Файловая система и устройства

- Перед форматированием, mount, swap и extroot повторно валидировать block device в backend, даже если UI уже проверил его.
- Нельзя принимать произвольный `/dev/...` как доказательство съёмного носителя.
- Нельзя форматировать накопитель, используемый для `/`, `/overlay` или системного swap.
- Нельзя размонтировать устройство, отличное от явно запрошенного.
- При ошибке шифрования plaintext не записывать под расширением `.enc`.
- Для destructive-команд требуются явная аутентификация, подтверждение в UI и server-side проверка.

## 11. Android и Kotlin

1. Имена классов — `PascalCase`, функций и свойств — `camelCase`.
2. Для моделей ответа использовать `data class` с конкретными полями, а не `Map<String, Any>`.
3. Nullable-поле отражает реальную возможность отсутствия значения; не делать всё nullable «на всякий случай».
4. Сетевой ответ проверять по HTTP-коду до чтения `inputStream`; для ошибки читать `errorStream`.
5. Успешным считать не просто `2xx`, а валидный ожидаемый JSON-контракт с идентификатором сервиса.
6. URL нормализовать и валидировать до сохранения.
7. Не разрешать произвольные схемы URL.
8. Redirect при pairing и administrative API должен быть запрещён или проверен, чтобы не отправить токен другому хосту.
9. Bearer-токен не хранить вместе с одноразовым pairing-кодом и не включать в QR.
10. Секреты хранить через безопасное Android-хранилище; обычные SharedPreferences/DataStore без шифрования не считать подходящими для долгоживущего административного токена.
11. `allowBackup` для приложения с локальными адресами, токенами или чувствительными настройками должен быть отключён, если не реализован безопасный backup policy.
12. `BroadcastReceiver.goAsync()` обязательно завершать через `finish()` в `finally`.
13. Периодический WorkManager не настраивать чаще минимально поддерживаемого интервала.
14. AlarmManager использовать только при реальной необходимости точного времени и с учётом разрешений целевой версии Android.
15. UI внутри `Scaffold` должен учитывать `innerPadding`.
16. Строки пользовательского интерфейса выносить в resources, кроме кратких технических маркеров и временного прототипа.
17. Не добавлять зависимость ради одного неиспользуемого экспериментального файла.
18. Не оставлять второй package с дублирующей темой, ресурсами или классами.
19. Перед merge должны собираться оба модуля:

```text
android
android-child
```

20. Cleartext HTTP не включать глобально без документированной модели угроз. Целевой путь — HTTPS или узко ограниченная локальная policy.

## 12. HTTP API, pairing и безопасность

- Публичный endpoint не должен создавать администратора, API-ключ, pairing-код или другое полномочие.
- Создание pairing-кода выполняется только из аутентифицированного LuCI или другой доверенной административной границы.
- `/pair` принимает только заранее выпущенный одноразовый код.
- Устройство из blocklist не может получить административные права.
- Проверку blocklist выполнять повторно непосредственно перед первой записью прав.
- Pairing-код потребляется только после успешной записи прав и хэша Bearer-токена.
- Plaintext Bearer возвращается только один раз успешному клиенту.
- На роутере хранится только криптографический хэш токена.
- Хэширование security-sensitive значения не должно иметь MD5 fallback.
- Rate limit идентифицирует устройство на стороне роутера, а не по переданному клиентом произвольному ID.
- Неизвестный MAC не считается доверенным.
- Обычный LAN-клиент не получает привилегированный AI-контекст, журналы, аккаунты и диагностику.
- API-ключ провайдера остаётся на роутере.
- CORS не включается глобально ради обхода ошибки клиента.
- Ошибки API возвращают стабильный machine-readable code и безопасное сообщение без секрета.

## 13. SQL и MySQL

Раздел применяется, если в проекте появится SQL или PHP-код с SQL.

- SQL-фрагменты в PHP писать через heredoc:

```php
<<< SQL
SELECT
    c.id
FROM city c
WHERE
    c.id = :id
SQL
```

- Если heredoc невозможен, использовать двойные кавычки.
- Одинарные кавычки для SQL-фрагмента в PHP не использовать.
- SQL-служебные слова писать в верхнем регистре: `SELECT`, `FROM`, `WHERE`, `LEFT JOIN`, `ORDER BY`.
- Таблицы, поля и алиасы не оборачивать в обратные кавычки без технической необходимости.
- Однострочный SQL не должен превышать 120 символов.
- При переносе каждый раздел начинать с новой строки.
- Если один раздел разбит на несколько строк, переносить запрос последовательно по разделам.
- В длинном `WHERE` каждое условие писать с новой строки.
- Операторы сравнения отделять пробелами: `c.id = :id`.
- При `JOIN` поля другой таблицы выводить отдельно с алиасом `table_name__field`.
- Алиас таблицы составлять из первых букв слов: `loyalty_card -> lc`.
- Отступ SQL внутри heredoc соответствует вложенности PHP-кода.
- Все внешние значения передавать через bind-параметры.
- Для boolean в MySQL использовать `tinyint(1)` со значениями `1` и `0`.
- Для процентов до 100 с дробной частью использовать `decimal(5, 2)`.
- Индексы именовать с типом, сокращением таблицы и полями:

```text
uq__spt__session_id__payment_terminal_id
idx__spt__payment_terminal_id
```

- Если имя слишком длинное, сокращать хвост через `others`.

## 14. PHP

Раздел обязателен, если PHP появится в проектном коде.

### 14.1. Теги и документация

- Использовать короткий тег `<?`, для вывода `<?=`.
- Не писать `<?php` в проектном коде.
- Каждый метод и функция должны иметь PHPDoc прямо перед объявлением без пустой строки.
- Для короткого описания использовать однострочный PHPDoc:

```php
/** Инициализация */
```

- Для параметров, return, throws, array shape или подробностей использовать многострочный PHPDoc.
- Типы указывать в сигнатуре:

```php
public function actionIndex(): void
```

- Если метод ничего не возвращает, указывать `: void`.
- Если тип нельзя точно выразить в сигнатуре, уточнять его в PHPDoc.
- Перед загрузкой объекта или массива объектов допустим формат:

```php
/** Город @var CityModel $city */
/** Детализации @var SessionModel[] $sessions */
```

### 14.2. Массивы и строки

- Массивы писать только коротким синтаксисом `[]`.
- В коротком массиве не ставить пробелы после `[` и перед `]`:

```php
['id' => $id]
```

- В многострочном массиве каждый элемент писать с новой строки и ставить запятую после последнего.
- В `rules()` допустимы длинные массивы в одну строку, если это соответствует стилю файла.
- Конкатенацию писать с пробелами вокруг точки:

```php
$a . $b
```

- Длинную конкатенацию переносить по смысловым границам.
- Для поиска подстроки использовать `str_contains()`, а не `strpos() !== false`.

## 15. Yii 1 и ActiveRecord

Раздел применяется только при появлении Yii 1.

- ActiveRecord-модель содержит только свойства полей таблицы.
- Свойства фильтров выносить в наследника вида `EventFormSearchModel`.
- Свойства форм создания и редактирования выносить в наследника вида `EventFormEditModel`.
- В `relations()` для `HAS_ONE` и `BELONGS_TO` проверять наличие связи через ключевое поле текущей модели, а не через объект связи.
- `tableName()`, `rules()`, `relations()`, `attributeLabels()` могут возвращать короткий массив сразу после открытия метода:

```php
public function rules(): array
{ return [ ... ]; }
```

- Для поиска использовать `CDbCriteria`, `compare()`, `addCondition()`, `CActiveDataProvider`.
- Для SQL использовать `Yii::app()->db->createCommand(<<< SQL ... SQL)`.
- Для входных данных использовать проектные helpers наподобие `AppHelper::getRequestFieldInt()`, `getRequestFieldArray()`, `getRequestFieldBool()`.
- Для ошибок, redirect и JSON использовать существующие helpers, а не дублировать обработку.
- Порядок в контроллере: входные данные, загрузка модели, проверки, заголовки/метаданные, `render()`.
- В admin controller `init()` сначала проверять доступ через `parent::checkAccess(...)`, затем вызывать `parent::init()`, если так принято рядом.
- `loadModel()` возвращает конкретный тип и обрабатывает отсутствие через проектный helper.
- Для стандартных описаний сущности использовать принятый паттерн `sd...()`.

### 15.1. Практические паттерны Yii-проекта

Если соответствующая архитектура будет перенесена или подключена:

- site controller: `ArticleController extends RenderableSiteController`;
- controller model: `ArticleControllerModel extends RenderableControllerModel`;
- admin controller: `CityController extends AdminPanelController`;
- типовые методы: `actionIndex`, `actionEdit`, `actionArchive`, `loadModel`;
- ActiveRecord: PHPDoc свойств, `model()`, `__get()`, `tableName()`, `rules()`, `relations()`, `attributeLabels()`, `sd...()`, методы данных.

## 16. PHP Views и HTML-шаблоны

Раздел применяется только при появлении PHP/Yii view.

- Использовать альтернативный синтаксис:

```php
<? if (...): ?>
<? foreach (...): ?>
<? endif; ?>
```

- Не использовать фигурные скобки для `if` и `foreach` внутри view-разметки.
- В начале view указывать PHPDoc с `$this` и переданными переменными.
- Порядок переменных в PHPDoc соответствует порядку передачи в `render()`.
- Для вывода использовать `<?= ... ?>`.
- Виджеты Yii вызывать через `$this->widget(...)`.
- Формы открывать `$this->beginActiveForm(...)`, закрывать `$this->endActiveForm()`.
- Хлебные крошки добавлять через `$this->addBreadcrumb(...)`.
- Сохранять принятые проектные widgets и helpers: `ExtendedGridView`, `FormatterHelper`, `ButtonColumn`, `ActiveForm`.
- Inline JavaScript допустим, если привязан к одной форме или странице и совпадает со стилем соседнего view.
- PHP-значения в JavaScript передавать через `json_encode()`.
- HTML-атрибуты и inline style не рефакторить массово без задачи.

## 17. Тестирование и CI

Перед завершением изменения выполнить применимые проверки:

### Shell

```sh
sh -n <script>
```

Для пакета проверять все helper-скрипты и CGI dispatcher.

### JavaScript

```sh
node --test tests/*.test.mjs
```

Если изменён LuCI view без прямого unit-теста, проверить хотя бы синтаксис и основные функции преобразования данных.

### Android

```sh
gradle -p android assembleDebug
gradle -p android-child assembleDebug
```

### Package

- проверить `postinst` со staging `IPKG_INSTROOT`;
- проверить повторную установку;
- проверить обновление поверх предыдущей версии;
- проверить `prerm`;
- проверить executable bit;
- проверить, что секреты и пользовательская конфигурация не перезаписываются.

### Негативные тесты

Для authentication, pairing, AI, USB и logs обязательны негативные сценарии:

- неправильный и истёкший pairing-код;
- неизвестный MAC;
- MAC из blocklist;
- повторное использование кода;
- ошибка записи токена;
- отсутствие OpenSSL;
- readonly filesystem;
- path traversal;
- неверный block device;
- mountpoint с другим устройством;
- превышение rate limit;
- запрос привилегированного контекста без Bearer.

## 18. Правила ревью

Ревьюер проверяет не только diff, но и следующие свойства:

1. Сохраняется ли обратная совместимость.
2. Не расширилась ли публичная поверхность API.
3. Нет ли нового fallback, который ухудшает безопасность.
4. Не появился ли секрет в логе, UCI, QR, exception или документации.
5. Совпадает ли документация с кодом.
6. Обновлена ли версия ресурсов при изменении JS/CSS.
7. Есть ли обработка частичного сбоя между несколькими записями.
8. Работает ли код в BusyBox/POSIX shell, а не только в Bash.
9. Собираются ли оба Android-модуля.
10. Не изменён ли сгенерированный файл вместо исходника.
11. Не смешан ли новый стиль с legacy-файлом без необходимости.
12. Есть ли понятное сообщение пользователю без раскрытия внутренних секретов.

## 19. Исключения

Исключение из правила допустимо только когда:

- этого требует совместимость с внешним API или существующим форматом;
- правило невозможно применить в конкретном языке;
- соблюдение правила ухудшает безопасность или корректность;
- рядом существует более строгий локальный стандарт.

Исключение должно быть минимальным и понятным из кода или commit message. Нельзя использовать исключение как основание для массового несогласованного стиля.
