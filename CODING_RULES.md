# Правила кодинга Sheepfold

Этот документ обязателен для разработчиков, ревьюеров, контрибьюторов и AI-инструментов. Он применяется ко всему репозиторию, если в конкретном каталоге нет более строгого локального стандарта.

Основной стек проекта: OpenWrt shell, LuCI JavaScript, Android/Kotlin, UCI, CSS и документация. Разделы PHP, Yii и SQL сохраняются на случай появления такого кода в будущем.

## 1. Общие принципы

1. Корректность, безопасность и понятность важнее краткости.
2. Перед изменением файла изучать соседний код и продолжать его стиль, если он не нарушает правила этого документа.
3. Не переписывать старый файл целиком ради нового синтаксиса, форматирования или личных предпочтений.
4. Не смешивать функциональное исправление с несвязанным рефакторингом.
5. Не менять публичные API, UCI-ключи, форматы файлов и сетевые контракты без проверки обратной совместимости.
6. Не описывать планируемую функцию как уже реализованную.
7. Документация должна явно различать:
   - готовую функцию;
   - экспериментальную функцию;
   - план;
   - известное ограничение.
8. При изменении поведения обновлять связанные тесты и документацию.
9. Не создавать отдельную переменную для значения, используемого один раз, если это не улучшает читаемость, диагностику или безопасность.
10. Не сокращать код ценой скрытого побочного эффекта или неочевидной обработки ошибки.

## 2. Версия пакета и интерфейсных ресурсов

При изменении поставляемого `*.js` или `*.css` необходимо увеличить версию ресурсов перед сборкой нового пакета.

В Sheepfold версия ресурсов формируется через:

```make
PKG_RELEASE:=<номер релиза>
SHEEPFOLD_UI_ASSET_VERSION:=$(PKG_VERSION)-$(PKG_RELEASE)
```

Правила:

1. Увеличивать `PKG_RELEASE` относительно последнего опубликованного или подготовленного пакета.
2. Синхронно обновлять `ui_asset_version` в `root/etc/config/sheepfold`.
3. Не увеличивать номер отдельно для каждого промежуточного коммита одного логического изменения.
4. `PKG_VERSION` менять только для отдельного продуктового релиза.
5. Для изменений, не затрагивающих поставляемые JS/CSS, повышение версии ресурсов не требуется.

Если в будущем появится PHP-проект с `protected/config/params.php`, для него сохраняется правило увеличения минорной версии при изменении JS/CSS, например `75.017 -> 75.018`.

## 3. Именование

### 3.1. Общие правила

- Переменные, функции и методы писать в `camelCase`, если это допускает язык и текущий файл.
- Классы, интерфейсы, enum и data class писать в `PascalCase`.
- Константы писать в стиле языка:
  - shell: `UPPER_SNAKE_CASE`;
  - Kotlin: `UPPER_SNAKE_CASE`;
  - legacy JavaScript: сохранять стиль файла.
- Булевы значения называть как утверждение: `isEnabled`, `hasToken`, `canPair`, `shouldRefresh`.
- Коллекции называть во множественном числе: `devices`, `archiveFiles`, `administratorSections`.
- Экземпляр класса называть по имени класса в lowerCamelCase без лишнего суффикса `Model`:

```text
ExemptionFormSearchModel -> exemptionFormSearch
RouterConnectionManager -> routerConnectionManager
```

- jQuery-объекты в legacy-коде называть с префиксом `$`.
- В shell глобальные пути и константы называть точно: `TOKEN_STATE_DIR`, `MOUNT_POINT`, `PAIR_LOCK_DIR`.

### 3.2. Внешние контракты

Правило `camelCase` не применяется к именам, определённым внешним контрактом.

Без необходимости не переименовывать:

- UCI-ключи;
- HTTP-параметры;
- JSON-поля публичного API;
- ключи внешнего провайдера;
- переменные окружения CGI;
- имена полей файлового формата;
- значения, необходимые для обратной совместимости.

### 3.3. Точность имён

Не использовать абстрактное имя, если доступно более точное предметное имя.

Плохо:

```text
data, item, model, value, result
```

Лучше:

```text
clientStatus, deviceEntry, administrator, pairingCode, encryptionResult
```

Имена `data`, `item`, `value`, `result` допустимы в небольших общих helpers и callbacks, когда их смысл однозначен из сигнатуры и контекста.

Короткие имена `i`, `j`, `x`, `y` допустимы для индексов, координат и математических выражений.

## 4. Комментарии и todo

1. Все комментарии в проектном коде писать на русском языке.
2. После `//`, `#` или другого маркера комментария ставить пробел.
3. Новый или исправляемый комментарий писать с заглавной буквы без точки в конце.
4. Комментарий должен объяснять причину, ограничение, инвариант или назначение блока, а не повторять синтаксис.
5. Перед нетривиальным условием писать комментарий, если без него неясно, почему проверка существует или что она защищает.
6. Очевидный guard clause и простая проверка входного значения не требуют комментария.
7. Для сложного многострочного условия допускаются комментарии внутри условия перед соответствующими группами проверок.
8. `todo` писать в нижнем регистре и с номером задачи:

```text
// todo: issue100 Добавить проверку миграции
# todo: issue100 Проверить поведение BusyBox tar
```

9. Не оставлять `todo`, скрывающий критичную ошибку или небезопасный fallback.
10. Не использовать комментарии «временно», «потом исправить» или «магия» без номера задачи.

## 5. Условия и управляющие конструкции

- Для `if`, `else`, `for`, `foreach`, `while`, `when`, `switch` использовать блочный стиль языка.
- В PHP, JavaScript и Kotlin фигурные скобки обязательны даже для одного оператора.
- Короткий guard clause допустим в одну строку, если это соответствует стилю текущего файла и не ухудшает читаемость.
- В одной цепочке `if/elseif/else` не смешивать однострочный и многострочный стиль.
- Части сложного условия группировать по смыслу.
- Не писать `== true`, `!= false` или `== 1`, если значение строго булево или уже нормализовано к `0/1`.
- Тернарный оператор использовать только для короткого читаемого выбора значения.
- Вложенные тернарные операторы не использовать.
- В Kotlin для выбора значения использовать `if` как выражение или `when`.
- В shell не скрывать важную ошибку длинной цепочкой `&&` и `||`; для изменения состояния использовать явный `if`.

## 6. Null, пустые значения и типы

- `null` писать в нижнем регистре.
- В PHP проверять `null` через `is_null($value)`.
- Для присваивания значения только при `null` в PHP использовать `??=`.
- В Kotlin использовать nullable-типы, safe call, Elvis и `requireNotNull` по назначению.
- Не использовать Kotlin `!!` без доказанного инварианта.
- Не считать взаимозаменяемыми пустую строку, отсутствие значения, `null`, `0` и `false` без явной нормализации.
- В shell проверять числовой ввод до арифметики.
- Данные из UCI, HTTP, DHCP, ARP, sysfs и внешних команд считать недоверенными.
- Известную структуру данных описывать типом, data class, интерфейсом или PHPDoc array shape.

## 7. Доступ и наследование

- Выбирать максимально строгий модификатор доступа: `private`, затем `protected`, затем `public`.
- Не проектировать наследование без необходимости.
- Использовать `final`, если это соответствует языку и не мешает framework, proxy или тестовым инструментам.
- Не открывать внутренний helper как публичный API только ради теста.
- Android `Context` не хранить дольше необходимого; для долгоживущих объектов использовать `applicationContext`.

## 8. JavaScript и LuCI

1. Если файл написан на `var`, callback и legacy LuCI API, не переписывать его целиком на современный синтаксис.
2. В новом небольшом модуле допустимы `const`, `let` и стрелочные функции.
3. Не смешивать legacy и modern стиль в одном логическом блоке без причины.
4. В новом коде использовать `===` и `!==`, если это не ломает совместимость с текущими значениями.
5. Анонимную legacy-функцию писать с пробелом:

```javascript
function () {
```

6. jQuery не добавлять в новый LuCI-код.
7. Правила jQuery применять только при исправлении уже существующего jQuery-файла:

```javascript
$(function () {
    $('#comments').on('click', '.comment-delete', function () {
        // Обрабатываем удаление комментария
    });
});
```

8. В LuCI использовать стандартные `rpc.declare`, `fs.exec`, `uci`, `ui`, `form` и view-модули, если они подходят задаче.
9. `fs.exec` разрешён только для команды, явно указанной в ACL, и только с валидированными аргументами.
10. Значения из server-side кода передавать в JavaScript через JSON-сериализацию.
11. Не считать DOM-узел существующим без проверки, если он зависит от режима экрана или старой конфигурации.
12. Не полагаться на фиксированный индекс дочернего элемента, если можно использовать класс, `data-*` или стабильный контейнер.
13. Таймеры, watchers и обработчики должны иметь понятный жизненный цикл и не создавать повторные параллельные циклы.
14. Для pairing-кодов, токенов и ключей использовать только криптографический источник случайности.
15. Не использовать `Math.random()` как fallback для security-sensitive значений.
16. Минифицированные `*.min.js` не использовать как исходник.
17. После изменения поставляемого JS выполнять правило версии ресурсов из раздела 2.

### 8.1. Модульность больших LuCI-view (по образцу podkop)

Ориентир — `itdoginfo/podkop` (`luci-app-podkop/htdocs/luci-static/resources/view/podkop/`): вместо одного файла на весь экран там есть тонкая точка входа (`podkop.js`, около 100 строк) и несколько сфокусированных модулей (`section.js`, `settings.js`, `dashboard.js`, `diagnostic.js`, общий `main.js`). Ниже — правила, которые из этого следуют.

18. Новый функционал в LuCI-view добавлять новым `require`-модулем, а не дописыванием в существующий большой файл, если функционал логически самостоятелен (отдельная вкладка/секция/переиспользуемая математика).
19. Модуль-строитель контента — обычный объект с именованными функциями (`function createXContent(section) {...}`), а не `view.extend()`, если это не отдельный маршрут LuCI-меню. `view.extend()` — только для файла, у которого есть собственный `menu.d`-роут.
20. Импорт — по схеме `"require view.<пакет>.<файл> as <алиас>"`, имя алиаса короткое и совпадает по смыслу с именем файла (как `main`, `section`, `settings` у podkop).
21. Общие для нескольких view вещи (стили, повторяющиеся виджеты, форматтеры) выносить в один общий модуль (аналог `main.js` у podkop) и подключать через `require`, а не копировать между файлами. Перед копированием куска кода в новый файл — сначала проверить, нет ли уже такой функции в общем модуле.
22. Самодостаточная математика/утилита без состояния view (кодеки, парсеры, генераторы наподобие QR/Reed-Solomon) всегда выносится в отдельный модуль без `view.extend()`, даже если пока используется только в одном месте — это единственный способ покрыть её `node --test` (раздел 17), не поднимая LuCI-рантайм.
23. Если правки делают файл view длиннее примерно 800–1000 строк — это сигнал завести новый модуль под конкретную вкладку/секцию, а не сигнал «дописать ещё один if». Существующий большой файл не обязательно переписывать целиком (правило 1), но новую независимую функциональность в него не добавлять.
24. Один экран может иметь только один явный источник переводимых строк на файл-модуль (раздел 8.2) — не заводить в новом модуле собственный словарь/копию `T()`.

### 8.2. Локализация: `_()` + `po/`, без самодельных словарей

25. Новый и переносимый код использует только штатный LuCI `_('...')` (gettext), не собственные функции-обёртки вроде локального словаря `T()`.
26. Строка внутри `_('...')` пишется на английском — это msgid и одновременно фолбэк, если перевода нет (как в podkop: `_("Connection Type")`, а не `_('Тип соединения')`).
27. Перевод строк живёт в `po/<lang>/sheepfold.po`, шаблон — в `po/templates/sheepfold.pot`; `.po` для английского не создаётся, английский — это сам исходный текст. Для генерации/обновления использовать `xgettext.sh` и `msgmerge.sh` (см. одноимённые скрипты в `itdoginfo/podkop`), а не редактировать `.po` вручную построчно, если можно перегенерировать.
28. `LUCI_LANGUAGES` и `LUCI_LANG.<код>` в `Makefile` обновлять при добавлении нового языка.
29. Самодельные словари наподобие `T()` в `overview.js` — переходное состояние, а не образец для нового кода; при правке файла, где такой словарь уже есть, не увеличивать его новыми ключами без необходимости — вместо этого заводить новые строки сразу через `_()`.

## 9. CSS

1. Не использовать сгенерированный или публикуемый каталог как исходник.
2. Изменять CSS-файл, который действительно подключён сборкой или LuCI view.
3. Новый CSS-блок оформлять многострочно:

```css
.sheepfold-card {
    display: flex;
    gap: 12px;
}
```

4. Не форматировать старые компактные блоки без задачи.
5. Секционные комментарии допустимы:

```css
/*** Настройки ИИ ***/
```

6. Селекторы, завязанные на существующую HTML-структуру, Bootstrap или LuCI, менять осторожно.
7. Цвета, размеры и отступы брать из ближайшего контекста.
8. Не вводить новую дизайн-систему в рамках локального исправления.
9. Для изображений использовать относительные пути от CSS-файла.
10. Не менять глобальные селекторы без явной необходимости.
11. Для анимации использовать именованный `@keyframes` и проверять совпадение имени в `animation`.
12. После изменения поставляемого CSS выполнять правило версии ресурсов из раздела 2.

## 10. Shell и OpenWrt

### 10.1. Базовый стиль

- Для новых скриптов использовать `#!/bin/sh` и POSIX/BusyBox-совместимый синтаксис.
- Не использовать bash-only конструкции без явной зависимости от Bash.
- Использовать `set -eu`, если остановка при ошибке безопасна для данного скрипта.
- Ожидаемую частичную ошибку внешней команды обрабатывать явно.
- Пути и значения заключать в кавычки, кроме осознанного word splitting после строгой валидации.
- Не использовать `eval`.
- Не передавать непроверенный ввод в `sed`, `awk`, `grep`, `uci`, `mount`, `rm`, `tar`, `openssl` и имя файла.
- Временный файл создавать в том же filesystem, что и итоговый, затем применять права и атомарный `mv`.
- Для секретов устанавливать `umask 077` до создания файла.
- Для временных файлов и lock-каталогов использовать `trap`.
- Lock должен иметь ограничение ожидания и понятную ошибку.

### 10.2. UCI

- Читать UCI через `uci -q get` с безопасным значением по умолчанию.
- Перед числовым сравнением проверять формат значения.
- Не создавать администратора или другую привилегированную сущность как побочный эффект проверки.
- Связанные UCI-поля изменять как одну логическую операцию.
- `uci commit` выполнять в контролируемой точке.
- Для частичного сбоя предусматривать rollback, revoke или безопасный повтор.
- Не хранить plaintext Bearer-токены, API-ключи и пароли в логах или выводе диагностической команды.
- Не передавать полный `uci show` в пользовательский журнал или AI-контекст.

### 10.3. Package hooks

- Проверять исполняемые права helper-скриптов при установке.
- `postinst` должен корректно работать со staging `IPKG_INSTROOT` и на реальном роутере.
- При наличии `IPKG_INSTROOT` не запускать сервисы и не изменять host-систему сборщика.
- Crontab изменять управляемым маркированным блоком и удалять его в `prerm`.
- Не считать `/etc/cron.d` поддерживаемым без проверки целевой сборки OpenWrt.
- Увеличивать release пакета для изменения, которое должно войти в новый `ipk`.

### 10.4. Устройства и файлы

- Перед format, mount, swap и extroot повторно проверять block device в backend.
- Не считать произвольный `/dev/...` доказательством съёмного носителя.
- Не форматировать накопитель, используемый для `/`, `/overlay` или системного swap.
- Не размонтировать устройство, отличное от явно запрошенного.
- При ошибке шифрования не записывать plaintext под расширением `.enc`.
- Для destructive-команды требуются аутентификация, подтверждение UI и server-side проверка.

## 11. Android и Kotlin

1. Имена классов писать в `PascalCase`, функций и свойств — в `camelCase`.
2. Модели ответа описывать `data class` с конкретными полями, а не `Map<String, Any>`.
3. Nullable-поле должно отражать реальную возможность отсутствия значения.
4. HTTP-ответ проверять по коду до чтения `inputStream`; для ошибки читать `errorStream`.
5. Успех определять по ожидаемому JSON-контракту, а не только по `2xx`.
6. URL нормализовать и валидировать до сохранения.
7. Не разрешать произвольные URL-схемы.
8. Redirect при pairing и административном API запрещать или проверять, чтобы не отправить токен другому хосту.
9. Не хранить Bearer-токен вместе с одноразовым pairing-кодом и не включать его в QR.
10. Долгоживущий административный токен хранить в защищённом Android-хранилище.
11. Обычные SharedPreferences или DataStore без шифрования не считать достаточным хранилищем для административного токена.
12. Для приложения с токенами и чувствительными настройками отключать backup либо реализовывать отдельную безопасную backup policy.
13. `BroadcastReceiver.goAsync()` завершать через `finish()` в `finally`.
14. WorkManager не запускать чаще минимально поддерживаемого интервала.
15. AlarmManager использовать только при реальной необходимости точного времени.
16. UI внутри `Scaffold` должен учитывать `innerPadding`.
17. Пользовательские строки выносить в resources, кроме кратких технических маркеров и временного прототипа.
18. Не добавлять зависимость ради неиспользуемого экспериментального файла.
19. Не оставлять дублирующий package с темой, ресурсами или классами.
20. Перед merge должны собираться оба модуля:

```text
android
android-child
```

21. Cleartext HTTP не включать глобально без документированной модели угроз.

## 12. Безопасность

Подробные инварианты pairing, Bearer-токенов, AI, USB и журналов хранятся в:

- `docs/security.md`;
- `docs/security.ru.md`;
- `docs/development-ideas-security.ru.md`.

`CODING_RULES.md` не дублирует эти документы. Любое изменение security-sensitive кода обязано соблюдать их требования.

Минимальные общие правила:

- не добавлять публичный endpoint, создающий полномочия;
- не хранить plaintext-токены на роутере;
- не использовать MD5 fallback для security-sensitive хэша;
- не доверять неизвестному MAC;
- не раскрывать привилегированный контекст без аутентификации;
- не включать глобальный CORS ради обхода ошибки клиента;
- не выводить секрет в тексте ошибки.

## 13. SQL и MySQL

Раздел применяется, если в проекте появится SQL.

- Многострочный SQL в PHP писать через heredoc или nowdoc.
- Короткий SQL допускается в обычной строке.
- Все внешние значения передавать bind-параметрами.
- SQL-служебные слова писать в верхнем регистре.
- Таблицы, поля и алиасы не оборачивать в обратные кавычки без технической необходимости.
- Однострочный SQL не должен быть длиннее 120 символов.
- При переносе начинать `SELECT`, `FROM`, `WHERE`, `JOIN`, `ORDER BY` с новой строки.
- В длинном `WHERE` писать каждое условие с новой строки.
- Операторы сравнения отделять пробелами.
- Алиасы таблиц должны быть короткими, понятными и не конфликтовать внутри запроса.
- Для boolean в MySQL использовать `tinyint(1)` со значениями `1` и `0`.
- Для процентов до 100 с дробной частью использовать `decimal(5, 2)`.
- Индексы именовать по типу, таблице и полям:

```text
uq__spt__session_id__payment_terminal_id
idx__spt__payment_terminal_id
```

- Если имя индекса слишком длинное, сокращать его без потери различимости.

## 14. PHP

Раздел применяется, если PHP появится в проектном коде.

### 14.1. Теги, типы и PHPDoc

- Для полноценного PHP-блока использовать `<?php`.
- Для вывода использовать `<?=`.
- Не использовать короткий тег `<?`.
- Каждая функция и каждый метод должны иметь PHPDoc непосредственно перед объявлением без пустой строки.
- Для короткого описания использовать однострочный PHPDoc:

```php
/** Инициализация */
```

- Для параметров, возвращаемого значения, исключений, array shape и подробностей использовать многострочный PHPDoc.
- Типы указывать в сигнатуре:

```php
/** Отображение страницы */
public function actionIndex(): void
```

- Если метод ничего не возвращает, указывать `: void`.
- Если тип нельзя выразить точно в сигнатуре, уточнять его в PHPDoc.
- Для объекта или массива объектов перед загрузкой использовать:

```php
/** Город @var CityModel $city */
/** Детализации @var SessionModel[] $sessions */
```

### 14.2. Массивы и строки

- Использовать только короткий синтаксис массивов `[]`.
- Не ставить пробелы после `[` и перед `]` в коротком массиве.
- В многострочном массиве писать один элемент на строку и ставить запятую после последнего.
- В `rules()` допустим длинный массив в одну строку, если это соответствует текущему стилю файла.
- Конкатенацию писать с пробелами вокруг точки.
- Длинную конкатенацию переносить по смысловым границам.
- Для поиска подстроки использовать `str_contains()`.
- Для булевого приведения использовать `(bool)$value`, а не `!!$value`.

## 15. Yii 1 и ActiveRecord

Раздел применяется только при появлении Yii 1.

- ActiveRecord-модель должна содержать только свойства полей таблицы.
- Поля фильтра выносить в `*FormSearchModel`.
- Поля формы создания или редактирования выносить в `*FormEditModel`.
- В `relations()` для `HAS_ONE` и `BELONGS_TO` проверять ключевое поле текущей модели.
- `tableName()`, `rules()`, `relations()`, `attributeLabels()` могут возвращать короткий массив сразу после открытия метода.
- Для поиска использовать `CDbCriteria`, `compare()`, `addCondition()`, `CActiveDataProvider`.
- Для входных данных и ошибок использовать существующие проектные helpers.
- В контроллере соблюдать порядок: входные данные, загрузка модели, проверки, метаданные, `render()`.
- `loadModel()` должен возвращать конкретный тип и обрабатывать отсутствие записи через проектный helper.
- Не переносить в Sheepfold имена классов, виджетов и архитектурные паттерны другого проекта без фактического подключения этой архитектуры.

## 16. PHP Views и HTML-шаблоны

Раздел применяется только при появлении PHP/Yii view.

- Использовать альтернативный синтаксис:

```php
<?php if (...): ?>
<?php foreach (...): ?>
<?php endif; ?>
```

- Не использовать фигурные скобки для `if` и `foreach` внутри view-разметки.
- В начале view указывать PHPDoc с `$this` и переданными переменными.
- Порядок PHPDoc-переменных должен соответствовать порядку передачи в `render()`.
- Для вывода использовать `<?= ... ?>`.
- Виджеты вызывать через `$this->widget(...)`.
- Формы открывать через `$this->beginActiveForm(...)`, закрывать через `$this->endActiveForm()`.
- Хлебные крошки добавлять через `$this->addBreadcrumb(...)`.
- PHP-значения в JavaScript передавать через `json_encode()`.
- Не рефакторить массово HTML-атрибуты и inline style без задачи.

- `prerm`;

### Негативные сценарии

Для authentication, pairing, AI, USB и logs проверять применимые негативные сценарии:

- неправильный или истёкший pairing-код;
- неизвестный MAC;
- MAC из blocklist;
- повторное использование кода;
- ошибка записи токена;
- отсутствие OpenSSL;
- readonly filesystem;
- path traversal;
- неверный block device;
- mountpoint с другим устройством;
- превышение rate limit;
- запрос привилегированного контекста без Bearer.

## 18. Правила ревью

Ревьюер проверяет:

1. Обратную совместимость.
2. Изменение публичной поверхности API.
3. Появление небезопасного fallback.
4. Утечку секрета в лог, UCI, QR, exception или документацию.
5. Соответствие документации коду.
6. Версию ресурсов при изменении поставляемого JS/CSS.
7. Обработку частичного сбоя между несколькими записями.
8. POSIX/BusyBox-совместимость shell-кода.
9. Сборку обоих Android-модулей.
10. Изменение исходника, а не сгенерированного файла.
11. Смешение legacy и modern стиля без необходимости.
12. Понятное сообщение пользователю без внутренних секретов.

## 19. Исключения

Исключение допустимо, если:

- этого требует внешний API или существующий формат;
- правило невозможно применить в конкретном языке;
- соблюдение правила ухудшает безопасность или корректность;
- рядом действует более строгий локальный стандарт.

Исключение должно быть минимальным и понятным из кода, документации или commit message.

## 20. Общие shell-библиотеки Sheepfold

В коде используются общие вспомогательные скрипты в `/usr/libexec/sheepfold`:

- `sheepfold-lib-device` — нормализация и валидация MAC, поиск UCI-секций устройств, работа со списками MAC (allowlist/blocklist).
- `sheepfold-lib-uci` — чтение `sheepfold.global.*`, создание и очистка UCI-секций списков, добавление/удаление MAC.
- `sheepfold-lib-json` — экранирование JSON, HTTP-заголовки для CGI-эндпоинтов, парсинг форм и `QUERY_STRING`.
- `sheepfold-lib-log` — делегирование в `sheepfold-log`, выбор пути RAM-лога по `sheepfold.global.log_cache_path`.

Новые shell-скрипты должны использовать эти библиотеки вместо локальных дублирующих helpers, если им нужна соответствующая функциональность.

## 21. Тесты для общих библиотек

Для общих библиотек добавляются отдельные shell-тесты в каталоге `tests`:

- `tests/test-lib-device.sh` — проверяет `normalize_mac`, `valid_mac`, создание/поиск UCI-секций устройств и работу allowlist без дубликатов.

Тесты работают в отдельном `UCI_CONFIG_DIR=/tmp/sheepfold-test-uci` и не изменяют реальный `/etc/config/sheepfold`. Перед изменением соответствующих helpers необходимо обновлять и прогонять эти тесты.
