### 3. Проверить реальное применение правил доступа

**Что сделать:** на отдельных тестовых клиентах проверить белый список, чёрный список, глобальное отключение, расписание разрешения/запрета, временный доступ, аварийно-полезные сайты и запрет доступа к LuCI/API/SSH.

**Почему:** красивое состояние UCI и интерфейса ещё не доказывает, что `nftables`, DHCP и DNS реально применяют правила. Это главный смысл проекта и главный риск слияния.

**Инварианты:**

- чёрный список сильнее групп, расписаний, временного доступа и автоопределения;
- устройство нельзя одновременно сохранить в белом и чёрном списке; интерфейс показывает красную ошибку и ничего самовольно не переносит;
- белый список обходит глобальную блокировку и расписания, но не создаёт административный доступ;
- устройства чёрного списка не открывают LuCI, SSH и Sheepfold API;
- отдельная настройка может оставить чёрному списку только аварийно-полезные сайты, но не интерфейс роутера;
- админское устройство удаляется из обычных групп и расписаний, добавляется в белый список и не может быть выбрано из чёрного списка;

### 4. Проверить HTTPS, сопряжение и смену порта на реальных устройствах

<!-- §j9wv3sz §z5ck8mv -->

**Что сделать:** выполнить QR-сопряжение и ручное сопряжение с Android 9 и современной версией Android. Проверить камеру, загрузку QR из файла, одноразовый код, отображение привязанного телефона у нужного администратора и повторное использование кода.

**Почему:** TLS handshake, Android Network Security и камера ведут себя на реальном телефоне иначе, чем компиляция и unit-тесты.

**Обязательно проверить:**

- QR не содержит root-пароль, cookie LuCI, bot token или AI API key;
- одноразовый код сжигается backend-частью только после успешного сопряжения и повторно отклоняется;

- ошибочный/чужой HTTPS-сервер не закрепляется как доверенный;
- после законной регенерации сертификата есть понятный сценарий повторного доверия, а не тупик;
- смена порта в LuCI доходит до уже сопряжённого APK безопасным способом; сейчас этот сквозной сценарий требуется отдельно подтвердить или доделать;

- детский `/client-status` возвращает только статус конкретного устройства, не раскрывает список семьи и не даёт административных действий;
- детский AI доступен только устройству из личной группы при разрешении родителя; иначе показывается понятный баннер.

### 5. Решить судьбу незавершённых разделов родительского APK

**Что сделать:** проверить вкладки «Расписания», «Группы», «Администраторы», «Wi-Fi» и «Журнал». Если это пока демонстрационные/обзорные экраны, либо реализовать полноценное редактирование через API, либо явно исключить их из релизной ветки и документации.

**Почему:** одинаковые пункты меню в LuCI и APK создают ожидание одинаковой работоспособности. Пустой или условный раздел выглядит как поломка, даже если сборка успешна.

**Мелкие UX-проверки:** системная кнопка «Назад», блокировка перехода до завершения защиты приложения, кнопки «Далее», контраст текста соглашения, квадратный QR-сканер, загрузка QR из файла, сообщения при мобильной сети, необязательное предупреждение о случайном MAC, пароль/PIN/биометрия, виджеты и уведомления о новых устройствах.

### 6. Устранить или явно зафиксировать все CI-ошибки GitHub Actions

**Что сделать:** после push открыть workflow `Validate Sheepfold` и убедиться, что зелёные все matrix jobs: `shell-and-node`, `android`, `android-child` для `push` и `pull_request`.

**Что сделать:** подготовить UCI с `D-0001`, `D-0012`, `A-0001`, отсутствующими ID, дубликатами и удалёнными устройствами. Запустить defaults/postinst и детектор несколько раз.

**Ожидаемый результат:** ID остаются стабильными, положительными и уникальными; следующий ID равен `max + 1`; формат `#1` создаёт только UI, а в UCI хранится число `1`.

**Что сделать:** проверить обратный порядок событий, ID устройства/администратора, отсутствие ложного «Интернет восстановлен» при первом запуске watchdog, RAM-хранение, очистку, лимит размера и экспорт за выбранный период.

**Почему:** журнал полезен только если события достоверны; ложная тревога быстро разрушает доверие. Секреты в журнале создают реальный риск компрометации.

**Не писать никогда:** пароли, bot/API tokens, session cookies, полные AI-запросы, банковские и медицинские данные. В обычном локальном журнале MAC/IP могут быть полными; при экспорте по умолчанию они маскируются, секреты всегда заменяются на `[secret]`.

## POLISH: мелочи, которые легко потерять

- Проверить водяной знак личной группы и разные пастельные цвета групп/Wi-Fi-карточек.
- Проверить размеры иконок типа устройства и ID, корону админского устройства и замок возле IP с постоянной DHCP-арендой.
- Убедиться, что таблицы и модалки не получают горизонтальную прокрутку на типичной ширине LuCI и на телефоне.
- Проверить сортировку устройств по ID, имени, IP, типу, группе и статусу; администраторов — по имени и логину.
- Проверить закрытие выпадающих списков кликом вне элемента и видимые рамки полей в фокусе.
- Проверить обе кнопки сохранения настроек, единое уведомление «Настройки успешно сохранены» и отсутствие отдельной непонятной кнопки «Применить».
- Проверить, что кнопки импорта/экспорта находятся только в «Настройки → Разное» и экспорт действительно включает клиентов.
- Проверить версию рядом с кнопкой обновления и cache-busting всех LuCI JS/CSS через одну версию ресурсов.
- Проверить QR Wi-Fi: реальный SSID/защита/пароль из UCI, большой QR, без показа технической строки `WIFI:...`.
- Проверить быстрый белый список: 30 секунд, пустая таблица вместо текста, список не уезжает из модалки, повторный запуск окна очищает кандидатов, а обновление таймера само по себе их не теряет.
- Проверить WPS/LED/Wi-Fi расписание на роутерах с разным числом LED, отсутствующей WPS-кнопкой и отдельными radio 2.4/5 GHz; неподдерживаемая функция должна быть отключена с объяснением.
- Проверить 10-секундный дисклеймер автоматического выключения Wi-Fi и доступность аварийного включения через WPS или активный мессенджер.
- Проверить Telegram-команды и русские алиасы, включая «помощь», `setMyCommands`, разрешённый user/chat ID и подтверждение опасных команд.
- VK оставить обозначенным как основной будущий мессенджер, но не показывать как готовый; MAX оставить экспериментальным и не реализовывать без стабильного API.
- Проверить, что updater не видит pre-release, не понижает версию и корректно переживает таймаут GitHub.
- Проверить Android 9 как минимальную поддерживаемую версию родительского APK; детское приложение не должно случайно получить административные экраны.
- Проверить, что артефакты `.apk`, `.ipk`, Gradle build, `.kotlin`, `__pycache__`, ключи и `local.properties` не попадают в Git.
- Проверить окончания строк LF и исполняемые биты после любых правок из Windows.
- После каждой правки LuCI, JS/CSS, manifest или package files поднять соответствующую версию; не менять версию только ради документации.
- В сложных местах оставить русские комментарии, объясняющие причину решения и защищаемый инвариант, но не пересказывать очевидный код.

## Команды контрольного прогона

Команды уточнять по актуальному `package.json` и workflow, но минимальный набор такой:

```powershell
npm test
sh -n install.sh
sh tests/runtime-hardening.sh
gradle -p android :app:assembleDebug
gradle -p android-child :app:assembleDebug
python scripts/build-test-ipk.py --out-dir artifacts
git diff --check
git status --short
```

На Linux/GitHub runner дополнительно проверить executable bit:

```sh
test -x install.sh
git ls-files --stage install.sh
```

## Порядок подготовки финального merge

1. Обновить `origin/main` и разрешить расхождения только в `editsByClaude`.
2. Пройти BLOCKER-пункты и записать результаты живых тестов в `docs/live-router-testing.ru.md`.
3. Исправить найденное, поднять версии только затронутых компонентов и снова прогнать весь набор тестов.
4. Собрать два APK и `.ipk`; положить пользовательские артефакты в `C:\Users\User\Downloads`, но не добавлять их в Git.
5. Запушить ветку и дождаться полностью зелёного GitHub Actions.
6. Создать PR с перечнем миграций, известных ограничений и инструкцией отката.
7. Сливать в `main` только после проверки резервного восстановления `/etc/config/sheepfold` и подтверждения пользователя.

## Ветка считается готовой к слиянию, когда

- все BLOCKER-пункты закрыты доказуемыми результатами;
- GitHub Actions полностью зелёный;
- чистая установка и обновление проверены на живом роутере;
- правила доступа реально применяются к тестовым устройствам;
- оба APK проходят ключевой сценарий на реальном Android 9+;
- в PR честно перечислены оставшиеся ограничения, а документация не называет заготовки готовыми функциями;
- `git status` чист, секретов и сборочных артефактов в коммите нет.
