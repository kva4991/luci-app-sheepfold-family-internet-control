# Проверка готовности крупных изменений к слиянию с `main`

<!-- §mrgready -->

Этот документ является рабочей памяткой для следующего разработчика или ИИ-агента. Он фиксирует не только крупные задачи, но и мелкие проверки, которые легко забыть после переключения между LuCI, OpenWrt backend и двумя Android-приложениями.

- Дата текущей контрольной точки: **19 июля 2026 года**.
- Фактическое состояние до оформления коммита: изменения находятся в рабочем дереве локального **`main`**, отдельной ветки `editsByClaude` в репозитории нет.
- После `git fetch origin` локальный `HEAD` и `origin/main` совпадали (`0/0`), поэтому скрытого расхождения с удалённой веткой на момент проверки не было.
- Большой незакоммиченный набор нужно сначала оформить в технической ветке, проверить staged-состав, затем вернуть в `main` обычным fast-forward merge. Это сохраняет понятную границу изменений и не делает вид, будто старая ветка всё ещё существует.

Этот документ различает **интеграционное слияние в `main`** и **публичный stable-релиз**. Для слияния нужны воспроизводимые локальные тесты, сборки, чистый состав коммита и честная документация. Аппаратная матрица, живые внешние API и первая полностью зелёная удалённая SDK-сборка остаются обязательными до публичного релиза, но не требуют держать проверенный код в вечной незакоммиченной ветке.

5. Оформить весь проверенный набор в отдельной технической ветке, проверить `git diff --cached --stat`, отсутствие секретов, локальных профилей, ключей и сборочных артефактов, затем слить ветку в `main` fast-forward.
6. После push проверить GitHub Actions. SDK-пакеты из этого запуска использовать для установки текущей версии на тестовый роутер и записать результат в `docs/live-router-testing.ru.md` до публичного stable-релиза.

**Особенность Windows:** если `npm test` из PowerShell блокируется через `npm.ps1`, использовать `npm.cmd test`. Если упаковочные Node-тесты получают `spawnSync(...).status === null`, повторить прогон вне ограниченной песочницы: им нужен запуск Python и запись во временный каталог. Не ослаблять системную Execution Policy ради этого.

## Куда особенно смотреть при проверке кода

<!-- §revspot -->

Проверять следует не только наличие функции, но и весь путь данных: ввод пользователя → LuCI/Android → API → UCI/backend → применение правила → повторное чтение фактического состояния → журнал и понятный ответ пользователю. Большинство опасных ошибок проекта возникает, когда первый шаг выглядит успешным, а состояние роутера не изменилось либо изменилось частично.

### OpenWrt package, установка и миграции

**Основные места:**

- `install.sh`;
- `package/luci-app-sheepfold-family-internet-control/Makefile`;
- `root/usr/share/sheepfold/sheepfold.uci.defaults`;
- `root/etc/init.d/sheepfold`;
- `update.sh` и `uninstall.sh`.

**На что смотреть:** shell должен работать в BusyBox `ash`, без неявной зависимости от Bash; пользовательские значения нельзя вставлять в команды без безопасного quoting; повторный `postinst` должен быть идемпотентным; обновление обязано добавлять недостающие настройки, но не перезаписывать пользовательские списки, пароли, группы и расписания; права CGI/libexec/init-файлов должны сохраняться; интерактивные вопросы задаются до `opkg`, а не внутри `postinst`.

**Почему:** ошибка здесь может оставить роутер без рабочей конфигурации, зависнуть во время установки или молча сбросить семейные правила. Особое внимание уделять `uci set/add_list/delete/commit`, обработке отсутствующих секций и миграции старых имён опций.

### Числовые ID и ссылки между сущностями

**Основные места:** `sheepfold-device-id`, defaults/postinst, детектор, pairing, Telegram, журнал, `/client-status` и загрузка администраторов в `overview.js`.

**На что смотреть:** пользовательский ID устройства хранится числом, никогда не переиспользуется и не уплотняется; миграция `D-0012` не должна превращать его в `2`; валидные существующие ID и ссылки расписаний остаются неизменными, а прежний строковый формат сохраняется как MAC-привязанный алиас. Администратор и группа связываются с MAC-секцией устройства, а не с позицией строки (§deviceid2).

**Почему:** разъехавшиеся ID могут дать права не тому телефону или сделать журнал недостоверным. Проверять не только генерацию нового ID, но и дубликаты, пустые значения и повторный запуск миграции.

### API, авторизация и выполнение команд

**Основные места:** `root/www/cgi-bin/sheepfold-api`, `sheepfold-api-legacy`, `sheepfold-router-control-legacy`, `sheepfold-token-common`, `sheepfold-pair-device`, `sheepfold-api-client-status`.

**На что смотреть:** каждый опасный route проверяет метод, размер тела, JSON-схему, bearer token, привязанный numeric device ID и MAC; действия blocklist/allowlist/admin нельзя вызвать через менее защищённый legacy route; код возврата и JSON должны отражать реальный результат UCI/firewall, а не только успешный запуск wrapper; все временные файлы создаются с безопасными правами и удаляются; rate limit нельзя обойти простой сменой необязательного заголовка.

**Почему:** фасад с именем `legacy` остаётся рабочей частью backend, поэтому его нельзя считать старым и безопасно игнорировать. Ищите случаи, где frontend показывает «сохранено» до повторного чтения UCI.

### HTTPS, сертификаты и сопряжение Android

**Основные места:** `SecureRouterConnectionManager.kt`, `RouterTlsPin.kt`, `ChildRouterHttps.kt`, `RouterAdminClient.kt`, discovery endpoint и `sheepfold-pair-device`.

**На что смотреть:** нет HTTP fallback; новый родительский `SF2` не падает обратно на TOFU, а детский pin сохраняется только после корректного ответа именно Sheepfold; изменившийся сертификат не принимается молча; pairing token короткоживущий и атомарно сжигается backend после успешной выдачи учётных данных; QR не содержит долговременных секретов; смена порта/сертификата имеет понятный сценарий восстановления.

**Почему:** «довериться любому первому TLS-серверу» без проверки Sheepfold позволяет закрепить чужой сертификат. Отдельно проверять гонку двух одновременных запросов с одним одноразовым кодом.

### LuCI frontend и сохранение UCI

**Основное место:** `htdocs/luci-static/resources/view/sheepfold/overview.js` и secure wrapper LuCI.

**На что смотреть:** вкладка и панель имеют одно устойчивое состояние; при открытии выделен правильный вложенный таб; модалки не закрываются из-за фонового refresh; таблицы не зависят от фиксированного номера столбца; обработчики не устанавливаются повторно при каждом render; все настройки применяются только по явной кнопке «Сохранить»; после backend/UCI-действия не остаются LuCI pending changes; интерфейс повторно читает backend и лишь после этого показывает успех.

**Почему:** `overview.js` большой и объединяет несколько подсистем, поэтому локальное исправление легко ломает соседнюю вкладку. При изменении JS/CSS обязательно поднять package release и проверить единый cache-busting `ui_asset_version`.

### Правила доступа и автоматическое определение устройств

**Основные места:** `sheepfold-device-detector`, classifier/mDNS/DHCP helpers, default groups и код применения firewall/DNS.

**На что смотреть:** имя/hostname не считается надёжным доказательством; для автоматического «Без ограничений» нужны независимые семейства признаков и включённая автонастройка; blocklist всегда сильнее автоматики; ручной тип и ручное удаление из группы не отменяются следующим проходом; периодический проход не создаёт дубликаты и не нагружает роутер постоянным сканированием портов.

**Почему:** ребёнок может подменить MAC и hostname, а слабая эвристика способна случайно дать безлимитный доступ обычному телефону или компьютеру. Любое новое правило классификатора проверять положительным, отрицательным и spoofing-тестом.

### Родительское Android-приложение

**Основные места:** `SafeRouterSetupScreen.kt`, `OperationalMainScreen.kt`, `AppProtectionStore.kt`, `AppUnlockScreen.kt`, widgets и `AndroidManifest.xml`.

**На что смотреть:** onboarding нельзя обойти системной кнопкой «Назад» или перезапуском процесса; пароль/PIN проверяются через медленный salted hash, а не простое сравнение; биометрия не считается настроенной, если устройство её не поддерживает; чувствительное состояние не попадает в логи/Bundle/clipboard; network-запросы не блокируют UI thread; coroutine и refresh не продолжают обновлять закрытую модалку; виджет показывает состояние backend, а не последнее нажатие пользователя.

**Почему:** успешный happy path не выявляет восстановление Activity, отказ в разрешениях, потерю сети и уничтожение процесса Android. Проверять Android 9 и современный Android отдельно.

### Детское Android-приложение

**Основные места:** `ClientStatusRepository.kt`, `ClientStatusResponse.kt`, `ChildRouterHttps.kt`, `AiRepository.kt`.

**На что смотреть:** приложение не получает административный токен и не умеет вызывать admin routes; `/client-status` определяет только текущий клиент на стороне роутера и не принимает произвольный MAC как истину; AI открывается только при согласии родителя и членстве в личной группе; ответ не раскрывает другие устройства, имена, журналы или настройки семьи.

**Почему:** endpoint намеренно доступен без обычной админской авторизации, поэтому минимальность ответа и надёжное определение клиента являются его границей безопасности.

### AI proxy, промпты и приватность

**Основные места:** `sheepfold-ai-gate`, `sheepfold-ai-handler`, versioned prompts, LuCI provider settings и Android AI clients.

**На что смотреть:** ключи провайдеров остаются на роутере и маскируются в UI/экспорте/логах; выбранная модель берётся из UCI; ошибки/таймауты/лимиты провайдера не превращаются в ложный успех; parent и child используют правильные версии промпта; отправка router-info или журнала требует preview и отдельного согласия; предложенное ИИ действие не выполняется без явного подтверждения.

**Почему:** тестовая строка без персональных данных может пройти, а реальный контекст случайно отправит MAC, имена детей или журнал. Проверять сформированный outbound JSON, а не только текст ответа модели.

### Журнал, мессенджеры и внешнее хранилище

**Основные места:** `sheepfold-activity-log`, общий log helper, Telegram bot, export, storage adapters и watchdog WAN.

**На что смотреть:** событие появляется только после реального изменения; первоначальный запуск watchdog не пишет ложное «Интернет восстановлен»; в локальном журнале есть нужные ID/IP/MAC, но экспорт маскирует их; секреты не пишутся никогда; Telegram принимает команды только от разрешённого ID и подтверждает опасные действия; external storage выключено по умолчанию и не отправляет незашифрованный полный архив без предупреждения.

**Почему:** ложные сообщения и фиктивные подтверждения быстро разрушают доверие пользователя, а утечка токена через диагностический лог компрометирует весь роутер.

### Тесты и признаки неполной реализации

**На что смотреть:** тест должен проверять поведение, а не только наличие нужной строки в файле; mocks не должны делать невозможное на реальном BusyBox/OpenWrt; исправление обязано иметь отрицательный сценарий; GitHub Actions запускает тот же важный тест, что и разработчик локально.

Полезный поиск перед ревью:

```powershell
rg -n "TODO|FIXME|placeholder|stub|not implemented|return true|catch \(_" package android android-child tests
rg -n "http://|trustAll|HostnameVerifier|cleartext|api[_-]?key|token|password" package android android-child
rg -n "uci (set|add_list|delete|commit)|ubus call|nft|firewall" package/luci-app-sheepfold-family-internet-control
rg -n "D-[0-9]|A-[0-9]|app_https_port|Овчарня" .
git diff --check origin/main...HEAD
```

**Красные флаги при ревью:** пустой `catch`; игнорирование exit code; успешный toast до чтения сохранённого состояния; shell-команда, собранная конкатенацией пользовательского текста; доверие одному hostname; секрет в QR/логе; HTTP fallback; автоматический перенос между белым и чёрным списком; функция, описанная в документации как готовая, но имеющая только UI-заглушку.

При исправлении тонкого места оставить короткий русский комментарий с объяснением защищаемого инварианта. Комментарий нужен не для пересказа строки кода, а чтобы следующий разработчик не «упростил» важную проверку.

Ниже у каждого пункта указан уровень. `MERGE` обязателен до локального слияния в `main`; `RELEASE` обязателен до публикации стабильного релиза и может выполняться уже из `main`, если незавершённость честно отражена в статусе реализации.

### 1. `[MERGE]` Повторно сравнить рабочее дерево с актуальным `origin/main`

**Что сделать:** выполнить `git fetch origin`, затем изучить `git rev-list --left-right --count HEAD...origin/main`, staged-состав и содержательные конфликты. **Почему:** `main` мог измениться после последнего анализа. Старый список из 31 расхождения перестаёт быть достаточным сразу после нового коммита в основной ветке.

### 2. `[RELEASE]` Проверить установку и обновление на живом OpenWrt-роутере

**Что сделать:** перед тестом сохранить `/etc/config/sheepfold`, список установленных пакетов и журналы `opkg`. Установить собранный пакет сначала как обновление существующей версии, затем проверить чистую установку на отдельном тестовом состоянии.

- обновление приложения видит только стабильный GitHub Release, сравнивает версии и показывает понятный русский результат;
- удаление пакета не удаляет список клиентов и сообщает об оставшихся настройках.

### 3. `[RELEASE]` Проверить реальное применение правил доступа

<!-- §xepq85j §h6mxq4c §w8dm3kv §d7vqm4s -->

**Что сделать:** на отдельных тестовых клиентах проверить белый список, чёрный список, глобальное отключение, расписание разрешения/запрета, временный доступ, аварийно-полезные сайты и запрет доступа к LuCI/API/SSH.

**Почему:** красивое состояние UCI и интерфейса ещё не доказывает, что `nftables`, DHCP и DNS реально применяют правила. Это главный смысл проекта и главный риск слияния.

**Инварианты:**

- чёрный список сильнее групп, расписаний, временного доступа и автоопределения;
- устройство нельзя одновременно сохранить в белом и чёрном списке; интерфейс показывает красную ошибку и ничего самовольно не переносит;
- белый список обходит глобальную блокировку и расписания, но не создаёт административный доступ;
- устройства чёрного списка не открывают LuCI, SSH и Sheepfold API;
- отдельная настройка может оставить чёрному списку только аварийно-полезные сайты, но не интерфейс роутера;
- админское устройство удаляется из обычных групп и расписаний, добавляется в белый список и не может быть выбрано из чёрного списка;
- инфраструктурное устройство попадает в «Без ограничений» только при включённой автонастройке и достаточной совокупности независимых признаков;
- ручной выбор типа прекращает последующее автоопределение; ручное удаление из «Без ограничений» не отменяется следующим проходом детектора.

### 4. `[RELEASE]` Проверить HTTPS, сопряжение и смену порта на реальных устройствах

<!-- §j9wv3sz §z5ck8mv -->

**Что сделать:** выполнить QR-сопряжение и ручное сопряжение с Android 9 и современной версией Android. Проверить камеру, загрузку QR из файла, одноразовый код, отображение привязанного телефона у нужного администратора и повторное использование кода.

**Почему:** TLS handshake, Android Network Security и камера ведут себя на реальном телефоне иначе, чем компиляция и unit-тесты.

**Обязательно проверить:**

- QR не содержит root-пароль, cookie LuCI, bot token или AI API key;

- детский `/client-status` возвращает только статус конкретного устройства, не раскрывает список семьи и не даёт административных действий;

**Что сделать:** проверить вкладки «Расписания», «Группы», «Администраторы», «Wi-Fi» и «Журнал». Если это пока демонстрационные/обзорные экраны, либо реализовать полноценное редактирование через API, либо явно исключить их из релизной ветки и документации.

### 10. Проверить миграцию числовых ID

**Что сделать:** подготовить UCI с `D-0001`, `D-0012`, `A-0001`, отсутствующими ID, дубликатами и удалёнными устройствами. Запустить defaults/postinst и детектор несколько раз.

**Почему:** ID попадают в журнал, UI, сопряжение и API. Повторная нумерация сделает историю непонятной и может привязать действие не к тому устройству.

**Ожидаемый результат:** ID положительные и уникальные; валидный старый номер сохраняется, удалённый номер не переиспользуется, поэтому разрывы и значение вроде `#109` при шести текущих карточках являются нормой. Новый ID берётся только из монотонного счётчика выше всех когда-либо выданных значений. Формат `#1` создаёт только UI, а в UCI хранится число `1`. Старый сопряжённый APK работает только с прежним ID, своим токеном и тем же MAC (§deviceid2).

### 11. Провести аудит журнала и приватности

**Что сделать:** проверить обратный порядок событий, ID устройства/администратора, отсутствие ложного «Интернет восстановлен» при первом запуске watchdog, RAM-хранение, очистку, лимит размера и экспорт за выбранный период.

**Почему:** журнал полезен только если события достоверны; ложная тревога быстро разрушает доверие. Секреты в журнале создают реальный риск компрометации.

**Не писать никогда:** пароли, bot/API tokens, session cookies, полные AI-запросы, банковские и медицинские данные. В обычном локальном журнале MAC/IP могут быть полными; при экспорте по умолчанию они маскируются, секреты всегда заменяются на `[secret]`.

- После каждой правки LuCI, JS/CSS, manifest или package files поднять соответствующую версию; не менять версию только ради документации.
- В сложных местах оставить русские комментарии, объясняющие причину решения и защищаемый инвариант, но не пересказывать очевидный код.

## Команды контрольного прогона

Команды уточнять по актуальному `package.json` и workflow, но минимальный набор такой:

```powershell
npm test
sh -n install.sh
# Полная команда shell/runtime-hardening приведена в docs/agent-environment.ru.md
android\gradlew.bat -p android :app:assembleDebug
```

На Linux/GitHub runner дополнительно проверить executable bit:

```sh
test -x install.sh
git ls-files --stage install.sh
```

## Порядок подготовки финального merge

1. Обновить `origin/main` и убедиться, что `HEAD` не отстаёт.
2. Завершить документационный аудит, выписать владельцу оставшиеся решения и не смешивать их с уже закрытыми вопросами.
3. Исправить найденное, поднять версии только затронутых компонентов и снова прогнать затронутые тесты; при изменении исполняемого кода повторить полный `npm.cmd test`.
4. Собрать оба APK и оба router-варианта локально. Явно запрошенные пользовательские артефакты положить в `C:\Users\User\Documents\pesochnica`, но не добавлять их в Git.
5. Создать техническую ветку из текущего `main`, добавить только проверенные исходники/документы, изучить staged diff и сделать один осмысленный коммит.
6. Вернуться в `main` и выполнить fast-forward merge. Не удалять техническую ветку до проверки итогового `git log` и `git status`.
7. После push дождаться зелёного GitHub Actions. Для публичного релиза дополнительно пройти живую установку SDK-пакета, runtime-матрицу и реальные Android-сценарии.

## Изменения готовы к интеграционному слиянию, когда

- актуальный `origin/main` не содержит неучтённых коммитов;
- полный локальный набор тестов зелёный, а каждый skip объяснён;
- оба APK компилируются, оба тестовых IPK упаковываются;
- документация честно отделяет работающие функции, ограничения и будущие планы;
- в staged-составе нет секретов, локальных профилей, ключей и сборочных артефактов;
- после коммита и fast-forward merge `git status` чист.
