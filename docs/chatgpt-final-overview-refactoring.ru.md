# Финальная карта изменений `overview.js`: все изменённые места и обоснование решений

Документ относится к кумулятивному проходу `chatGPT`, завершающему декомпозицию LuCI-файла
`package/luci-app-sheepfold-family-internet-control/htdocs/luci-static/resources/view/sheepfold/overview.js`.
Целевая версия OpenWrt-пакета после этого прохода — `0.1.0-r252`. Документ является отдельным журналом изменений, запрошенным владельцем проекта: для каждого существенного файла указано, что изменено, почему файл изменён и почему выбран именно такой способ.

## 0. Повторный аудит после первой финализации

После первоначального разреза `overview.js` был выполнен отдельный повторный аудит. Он не ограничивался проверкой числа строк и синтаксиса отдельных файлов. Проверялись фактический граф локальных LuCI-зависимостей, контракты `deps.*`, поведение UCI-клиента LuCI, частично успешные операции, конкурентные нажатия, Standard/AI stripping и ошибки, возникающие между сохранением UCI и применением runtime.

Первый набор `68/68` подтверждал только тот набор контрактов, который был заложен в тестовый загрузчик. Он **не являлся доказательством работоспособности полного LuCI runtime**: загрузчик разрешал часть отсутствующих модулей через заглушки, а статические проверки не моделировали глобальную staging-область LuCI UCI-клиента. Поэтому итоговая редакция документа не использует `68/68` как самостоятельный release gate.

### 0.1. Корневая причина пропущенных проблем

Основная причина состояла в сочетании четырёх факторов:

1. тестовый module loader был permissive и мог подставить пустой объект вместо обязательного локального модуля;
2. `node --check` проверял синтаксис каждого файла, но не существование импортируемого файла и не наличие ожидаемого метода;
3. прежний persistence-код ошибочно воспринимал `uci.save(config)` как сохранение одной конфигурации, хотя LuCI использует одну глобальную локальную staging-область, а `uci.save()` вызывается без аргументов;
4. часть тестов проверяла наличие строки или порядок вызовов в исходнике, но не выполняла эксперимент с ошибкой до commit, после commit, при параллельной мутации или при частично выполненном batch.

### 0.2. Обнаруженные дефекты, исправления и регрессионная защита

Ниже перечислены дефекты, найденные именно повторным аудитом. Для каждого пункта указано, почему предыдущая проверка его пропустила, как исправлено и каким видом проверки он теперь защищён.

#### 0.2.1. Отсутствующие обязательные модули `page/refresh.js` и `settings/save-flow.js`

- **Симптом:** `application.js` и Settings controller содержали `require` на файлы, отсутствовавшие в payload. Реальный LuCI loader завершил бы загрузку страницы ошибкой.
- **Корневая причина:** финальная сборка использовала wiring, подготовленный промежуточным проходом, но два файла не попали в cumulative overlay.
- **Почему предыдущая проверка пропустила:** тестовый loader возвращал пустой объект для неизвестного модуля, а `node --check application.js` не разрешает imports.
- **Как исправлено:** оба модуля добавлены в payload. `page/refresh.js` владеет локальной перерисовкой панелей, `settings/save-flow.js` — последовательностью validate → persist → special savers → runtime side effects → accept/partial result.
- **Регрессионная проверка:** строгий require-graph audit запрещает отсутствующие локальные `sheepfold.*` и `view.*` модули.

#### 0.2.2. Неполная dependency injection для custom device type control

- **Симптом:** `type-control.js` вызывал `deps.definitions()`, но composition root передавал только `byValue` и `icon`.
- **Корневая причина:** при переносе DOM-контрола в отдельный модуль был учтён не весь интерфейс исходной модели типов.
- **Почему предыдущая проверка пропустила:** форма редактора устройства не открывалась в первом smoke; создание модуля само по себе не вызывает `definitions()`.
- **Как исправлено:** `application.js` явно передаёт `deviceTypes.definitions`.
- **Регрессионная проверка:** targeted dependency-contract test и runtime-открытие type control.

#### 0.2.3. Вызов несуществующего `groupModel.colorForName()`

- **Симптом:** отображение или создание группы могло завершиться `TypeError`.
- **Корневая причина:** новое имя метода было придумано при переносе, но исходная модель экспортирует `hash()`, `automaticColor()` и `nextColor()`.
- **Почему предыдущая проверка пропустила:** синтаксис корректен; ошибка возникает только при вызове ветки выбора цвета.
- **Как исправлено:** `groups/naming.js` использует фактически экспортируемые `hash()` и palette, сохраняя прежний детерминированный выбор цвета.
- **Почему именно так:** добавлять второй почти идентичный API в модель ради одного caller не нужно; лучше использовать существующий стабильный контракт.
- **Регрессионная проверка:** module export/consumer contract test.

#### 0.2.4. Неверная модель `uci.save(config)` и `ui.changes.apply()`

- **Симптом:** разные операции могли смешать локально подготовленные изменения; код считал `ui.changes.apply()` ожидаемым Promise завершения, хотя это UI workflow.
- **Корневая причина:** LuCI UCI client ошибочно трактовался как набор независимых per-config транзакций. В действительности локальные `creates/changes/deletes/reorder` являются общей staging-областью, `uci.save()` не принимает имя конфигурации, а direct `uci.apply()` является ожидаемой persistence-примитивой.
- **Почему предыдущая проверка пропустила:** fake UCI принимал произвольные аргументы и возвращал удобный Promise; взаимодействие двух мутаций не моделировалось.
- **Как исправлено:** создана одна сериализованная граница `core/persistence/uci.js`. Она выполняет preflight локальных и remote pending changes, синхронный staging callback, проверку fingerprint, ровно один `uci.save()` без аргументов, затем напрямую вызывает низкоуровневые `uci.callApply(timeout, true)` и `uci.callConfirm()` и ожидает фактическое подтверждение. При ошибке выполняются remote revert и reload затронутых конфигураций.
- **Почему не использован публичный `uci.apply()`:** в проверенных LuCI ветках OpenWrt 24.10 и 25.12 этот метод запускает подтверждение через `setTimeout`, но не возвращает Promise отложенного confirm-процесса. Для Sheepfold это означало бы преждевременное сообщение об успехе. Поэтому application и secure wrapper используют те же RPC-примитивы `callApply/callConfirm`, но возвращают собственный ожидаемый Promise с ограниченным повтором подтверждения.
- **Почему именно так:** локальная staging-область LuCI глобальна; единственный надёжный способ не смешивать операции — сериализовать их до staging, вызывать save в том же turn и не считать операцию завершённой до подтверждения apply.
- **Регрессионная проверка:** zero-argument save audit, overlapping mutation tests, foreign-local-change tests, remote-change preflight, undefined-status rejection, confirm retry и apply-failure cleanup.

#### 0.2.5. Ошибочные сигнатуры UCI при создании именованных секций

- **Симптом:** часть кода пыталась создать `usb`, `cloud`, `gdrive` или `adguard` через трёхаргументный `uci.set()`, что не создаёт секцию.
- **Корневая причина:** создание секции и установка option были смешаны в helper-е Settings persistence.
- **Почему предыдущая проверка пропустила:** mock UCI разрешал сигнатуру, которой нет в настоящем LuCI API.
- **Как исправлено:** секции создаются только через `uci.add(config, type, name)`, после чего options устанавливаются четырёхаргументным `uci.set(config, section, option, value)`.
- **Почему именно так:** это соответствует публичному LuCI API и позволяет отдельно проверить конфликт имени с секцией другого типа.
- **Регрессионная проверка:** автоматический аудит всех вызовов `uci.add/set/unset/remove/save/apply` и runtime named-section test.

#### 0.2.6. Коллизии имён групп, расписаний и администраторов

- **Симптом:** два разных пользовательских объекта после slug-нормализации могли получить одно UCI section name или попасть поверх секции другого типа.
- **Корневая причина:** использовалось одно предпочтительное имя без полного collision loop.
- **Почему предыдущая проверка пропустила:** fixture содержал только один объект каждого типа.
- **Как исправлено:** generated name проверяется по всем секциям конфигурации; при занятом имени добавляются детерминированные суффиксы `_2`, `_3` и далее. Для администратора используется стабильный hash fragment, после чего всё равно выполняется collision loop. `ensureSection()` отдельно отклоняет совпадение имени с другим `.type`.
- **Почему именно так:** случайная строка усложнила бы backup/diff и не устранила бы необходимость проверки; детерминированный цикл повторяем и понятен при аудите.
- **Регрессионная проверка:** multiple-collision и wrong-type fixtures.

#### 0.2.7. Ошибка до commit оставляла локальное UCI staging

- **Симптом:** неудачная валидация или исключение в staging callback могло быть незаметно применено следующей операцией.
- **Корневая причина:** cleanup выполнялся только после ошибки сохранения или runtime.
- **Почему предыдущая проверка пропустила:** тесты начинали каждый сценарий с нового fake UCI и не выполняли следующую операцию после сбоя.
- **Как исправлено:** `mutate()` очищает принадлежащее операции staging при любой ошибке, в том числе до `uci.save()`, и сохраняет исходную ошибку. Cleanup error прикрепляется отдельно и не подменяет причину.
- **Регрессионная проверка:** stage-throws-then-next-mutation test.

#### 0.2.8. Ошибка после `uci.save()` могла оставить remote pending changes

- **Симптом:** если save отправил изменения в rpcd, а apply завершился ошибкой, следующая операция могла применить остаток.
- **Корневая причина:** локального `uci.unload()` недостаточно для очистки уже отправленного server-side staging.
- **Почему предыдущая проверка пропустила:** mock не разделял local staging и rpcd pending changes.
- **Как исправлено:** `application.js` объявляет узкий RPC `uci.revert(config)`, а UCI persistence вызывает его для принадлежащих операции конфигураций перед reload.
- **Почему именно так:** выполняется точечный revert только тех configs, которые были заявлены мутацией; общий unchecked revert чужих изменений не используется.
- **Регрессионная проверка:** apply failure → revert list → reload → original error identity.

#### 0.2.9. Wi-Fi editor мог навсегда остаться в `saving`

- **Симптом:** исключение при локальном staging происходило до Promise-chain и не попадало в `.finally()`.
- **Корневая причина:** `stageItem()` выполнялся до вызова persistence adapter.
- **Почему предыдущая проверка пропустила:** fake setters не бросали исключения.
- **Как исправлено:** весь Wi-Fi staging передан callback-ом внутрь сериализованной UCI mutation; `saving` снимается в `finally()` для precommit, commit и reload failures.
- **Регрессионная проверка:** throwing setter и повторное нажатие Save.

#### 0.2.10. Отключение static lease не удаляло старую DHCP host-секцию

- **Симптом:** флажок в UI становился выключенным, но dnsmasq продолжал выдавать прежний адрес.
- **Корневая причина:** persistence добавлял или обновлял lease, но не реализовывал обратную операцию.
- **Почему предыдущая проверка пропустила:** тестировалось только включение static lease.
- **Как исправлено:** adapter ищет существующую host-секцию по сохранённому section id или нормализованному MAC и удаляет её в той же Sheepfold+DHCP mutation.
- **Почему именно так:** изменение карточки устройства и DHCP должны быть одной пользовательской транзакцией, иначе UI и dnsmasq расходятся.
- **Регрессионная проверка:** staticLease true → false fixture с проверкой `uci.remove('dhcp', ...)`.

#### 0.2.11. Batch устройств скрывал частично выполненный результат

- **Симптом:** первые устройства могли быть сохранены, последующее падало, а UI сообщал общую ошибку без числа выполненных элементов.
- **Корневая причина:** batch строился как reduce-chain без explicit progress result.
- **Почему предыдущая проверка пропустила:** использовались одноэлементные наборы или полный успех.
- **Как исправлено:** controller учитывает `completedCount`, `totalCount`, `persisted`, `runtimeApplied` и `refreshFailed`; action key включает целевой статус и отсортированный список MAC.
- **Почему именно так:** уже committed UCI нельзя честно представить как полный rollback. Стабильный key одновременно предотвращает ошибочное coalescing разных batch.
- **Регрессионная проверка:** failure on Nth device и два параллельных batch с разным составом.

#### 0.2.12. Ошибка refresh после успешного commit маскировалась как ошибка сохранения

- **Симптом:** UCI и runtime были применены, но сбой перечитывания inventory показывал «не удалось сохранить».
- **Корневая причина:** refresh был последним `.then()` общей цепочки без отдельной классификации.
- **Почему предыдущая проверка пропустила:** reload mock всегда завершался успешно.
- **Как исправлено:** результат получает `persisted=true`, `runtimeApplied=true`, `refreshFailed=true`; UI рекомендует повторно открыть или обновить LuCI и не предлагает повторно сохранять данные.
- **Регрессионная проверка:** commit success + runtime success + reload failure.

#### 0.2.13. Pairing watcher создавал неполную синтетическую карточку раньше inventory reload

- **Симптом:** карточка могла получить временный ID и не содержать `configSection`, static lease и identity-признаки.
- **Корневая причина:** watcher сразу обновлял локальный массив по ответу pairing status.
- **Почему предыдущая проверка пропустила:** проверялось только появление карточки и уведомления.
- **Как исправлено:** после `paired=1` controller сначала перечитывает реальный UCI/device inventory и только при невозможности reload использует ограниченный fallback.
- **Почему именно так:** router-side UCI является источником постоянного ID и административной привязки.
- **Регрессионная проверка:** paired status + delayed real inventory fixture.

#### 0.2.14. Назначение admin device не очищало targets отдельных device-schedules

- **Симптом:** поля `schedule`/`schedules` device-секции очищались, но отдельная schedule-секция могла продолжать ссылаться на ID устройства.
- **Корневая причина:** поддерживались одновременно legacy и текущая модели расписаний, а cleanup охватывал только legacy-поля.
- **Почему предыдущая проверка пропустила:** fixture содержал только group-target schedule.
- **Как исправлено:** pairing persistence проходит schedule-секции с `target_type=device`, удаляет назначенные admin IDs из `targets` и сохраняет обновлённый list в той же mutation.
- **Регрессионная проверка:** mixed targets `[adminId, childId]` должны стать `[childId]`.

#### 0.2.15. Двойные кнопки Save в admin modal могли отправить операцию дважды

- **Симптом:** верхняя и нижняя кнопки использовали один callback, но не общий локальный `saving` guard.
- **Корневая причина:** две кнопки были добавлены для удобства длинной формы, а защита существовала только в backend runner.
- **Почему предыдущая проверка пропустила:** тест нажимал одну кнопку один раз.
- **Как исправлено:** editor хранит единый modal-scoped `saving`, синхронно блокирует обе кнопки и ожидает Promise controller-а.
- **Почему именно так:** backend coalescing не должен быть единственной защитой локального массива и DOM; guard ставится до первой асинхронной границы.
- **Регрессионная проверка:** два синхронных click на разные Save nodes дают один persist callback.

#### 0.2.16. Неодинаковая валидация administrator login

- **Симптом:** обычный overview и secure wrapper могли разрешить разные символы или длину.
- **Корневая причина:** regex был продублирован в двух местах.
- **Почему предыдущая проверка пропустила:** проверялось только наличие duplicate-login guard.
- **Как исправлено:** единый `validLogin()` находится в pairing persistence и передаётся обоим UI-путям.
- **Регрессионная проверка:** одинаковая таблица accepted/rejected login values.

#### 0.2.17. Runtime-only ошибка административной привязки имитировала rollback

- **Симптом:** UCI уже сохранял admin-device и allowlist, но controller возвращал старые локальные `deviceIds`.
- **Корневая причина:** все rejected Promise считались precommit failure.
- **Почему предыдущая проверка пропустила:** runtime всегда завершался успешно.
- **Как исправлено:** persistence маркирует postcommit failure полями `persisted` и `runtimeApplied`; controller перечитывает inventory и показывает предупреждение, не восстанавливая заведомо устаревшую локальную копию.
- **Регрессионная проверка:** commit success + schedule-sync failure.

#### 0.2.18. Группу можно было удалить при ссылке из schedule

- **Симптом:** правило оставалось с несуществующей целью.
- **Корневая причина:** deletion guard учитывал protected flag и assigned devices, но не schedule targets.
- **Почему предыдущая проверка пропустила:** fixture группы не содержал расписаний.
- **Как исправлено:** group persistence проверяет все enabled и disabled group-target schedules; при ссылке возвращает `group_referenced_by_schedule` до staging.
- **Почему именно так:** молча переписывать или удалять семейное расписание при удалении группы опаснее явного отказа.
- **Регрессионная проверка:** group-target schedule блокирует delete.

#### 0.2.19. Quick allowlist использовал IP или имя как резервный identity key

- **Симптом:** DHCP-смена или одинаковые имена могли объединить разные устройства.
- **Корневая причина:** UI стремился показать кандидата даже без MAC.
- **Почему предыдущая проверка пропустила:** fixtures всегда имели уникальный IP и имя.
- **Как исправлено:** candidate key существует только для валидного нормализованного MAC; кандидат без MAC не может быть автоматически сопоставлен.
- **Почему именно так:** IP и hostname являются наблюдениями, а не устойчивой идентичностью устройства.
- **Регрессионная проверка:** одинаковые names/IP без MAC не создают candidate identity.

#### 0.2.20. Action runner разрушал DOM иконок кнопки

- **Симптом:** временная замена `textContent` удаляла дочерний SVG; после операции возвращался текст, но не исходное дерево.
- **Корневая причина:** snapshot сохранял строку, а не child nodes.
- **Почему предыдущая проверка пропустила:** fake button содержал только текст.
- **Как исправлено:** runner сохраняет точный массив child nodes и восстанавливает его через `replaceChildren()` вместе с исходными `disabled`, class и `aria-busy`.
- **Регрессионная проверка:** SVG node identity до и после действия совпадает.

#### 0.2.21. Shell metadata могло включить произвольную первую строку backend stderr/stdout

- **Симптом:** чувствительное содержимое могло попасть в `actionMessage` структурированной ошибки.
- **Корневая причина:** wrapper использовал первую строку legacy output как удобное описание.
- **Почему предыдущая проверка пропустила:** тестовые backend-сообщения не содержали секретов.
- **Как исправлено:** metadata использует только фиксированные сообщения для известных stable error codes; полный legacy stdout/stderr остаётся отдельным результатом и не копируется в metadata.
- **Регрессионная проверка:** backend выводит `secret-value`, а `actionMessage` его не содержит.

#### 0.2.22. Settings side effects не везде проверяли exit code

- **Симптом:** cron, site policy, LED, IPv6, schedule, emergency или country profile могли завершиться ненулевым кодом, но save flow сообщал успех.
- **Корневая причина:** часть callbacks возвращала raw `fs.exec` result без `ensureOk()`.
- **Почему предыдущая проверка пропустила:** mocks отклоняли Promise на ошибке вместо возврата `{code:1}`.
- **Как исправлено:** каждый изменяющий backend result проходит `checkedResult`; `schedule-sync` объединён и вызывается максимум один раз для набора связанных options.
- **Регрессионная проверка:** для каждого side effect проверяются code 0, code != 0 и reject.

#### 0.2.23. Ошибки discovery JSON и service restart подавлялись

- **Симптом:** после изменения app port UI мог сообщить успех, хотя `/.well-known/sheepfold.json` или сервис оставались на старом порту.
- **Корневая причина:** `.catch(function () {})` использовался как best-effort.
- **Почему предыдущая проверка пропустила:** write/restart mocks не падали.
- **Как исправлено:** обе ошибки являются postcommit partial failures с отдельными пользовательскими сообщениями.
- **Почему именно так:** конфигурация порта без обновлённого discovery ломает восстановление Android endpoint; молчаливый best-effort здесь недопустим.
- **Регрессионная проверка:** write failure и restart failure после сохранённого UCI.

#### 0.2.24. Частично выполненные special Settings savers принимались только после полного успеха

- **Симптом:** первый saver мог реально сохранить данные, второй падал, а первый оставался визуально dirty.
- **Корневая причина:** `accept()` всех savers выполнялся общим финальным циклом.
- **Почему предыдущая проверка пропустила:** использовался один special saver.
- **Как исправлено:** каждый успешно завершившийся saver принимается немедленно; ошибка содержит `completedSaverCount` и `totalSaverCount`.
- **Регрессионная проверка:** первый saver success, второй failure, третий не запускается.

#### 0.2.25. Standard build сохранял остаточный AI-контракт в device persistence

- **Симптом:** после удаления AI-блоков могло остаться имя `activityLogEnabled`, хотя само UCI-поле отсутствовало.
- **Корневая причина:** variant markers охватывали запись option, но не всю подготовку значения.
- **Почему предыдущая проверка пропустила:** проверялось отсутствие `activity_log_enabled`, но не camelCase идентификатора.
- **Как исправлено:** весь AI-only вычислительный и staging-блок находится между markers; Standard stripped source не содержит ни snake_case, ни camelCase идентификатора.
- **Регрессионная проверка:** executable Standard stripping audit.

### 0.3. Почему исправления не возвращены обратно в `overview.js`

Найденные дефекты могли быть быстро исправлены локальными условиями в composition root. Такой способ отклонён, потому что он снова сделал бы `overview.js` или `application.js` владельцем предметных правил. Исправления размещены у фактического владельца:

- UCI concurrency и cleanup — `core/persistence/uci.js`;
- device/DHCP/list semantics — `features/devices/persistence.js`;
- batch UI — `features/devices/controller.js`;
- pairing/admin invariants — `features/pairing/persistence.js` и `features/administrators/controller.js`;
- group references — `features/groups/persistence.js`;
- Settings partial-save semantics — `features/settings/save-flow.js`;
- runtime side effects — `features/settings/side-effects.js`;
- button DOM preservation — `core/backend/actions.js`;
- safe shell metadata — `sheepfold-luci-action`.

Такое размещение важно: новый тест может создать конкретный модуль с fake dependencies и проверить один контракт без загрузки всего LuCI view.

### 0.6. Последняя перепроверка перед формированием архива (§ovaudit4)

После объединения всех промежуточных исправлений был выполнен ещё один независимый прогон, потому что само объединение нескольких аудированных вариантов способно вернуть уже устранённый дефект. Эта проверка действительно обнаружила такие возвраты и поэтому является частью финального результата, а не косметическим дополнением.

#### 0.6.1. Возврат запрета на удаление постоянной DHCP-аренды

- **Симптом:** persistence уже умел удалить старую `dhcp host`-секцию, однако редактор снова блокировал checkbox для существующей аренды и сообщал, что аренда может только обновляться.
- **Корневая причина:** при слиянии более позднего hardening-варианта был взят старый presentation-блок, не совпадавший с уже исправленным persistence-контрактом.
- **Исправление:** checkbox снова доступен; подсказка прямо говорит, что снятие флажка удалит существующую постоянную аренду. `devices/persistence.js` удаляет найденную host-секцию в той же сериализованной UCI-мутации.
- **Почему именно так:** отображаемое действие и фактический persistence-контракт должны совпадать. Скрывать поддерживаемое удаление означало бы оставлять в dnsmasq состояние, которое пользователь не может отменить из Sheepfold.
- **Регрессия:** `tests/overviewFinalExperimentalR252.test.mjs` и `tests/finalAuditHardening.test.mjs` проверяют одновременно доступность checkbox и наличие удаления в persistence.

#### 0.6.2. Возвращённые проверки stale section и collision

- **Симптом:** поздняя версия могла довериться устаревшему `device.configSection`, использовать занятую DHCP section, изменить несуществующее расписание, принять stale group reference либо создать дублирующую группу.
- **Корневая причина:** при переносе последних nullable/refresh исправлений были взяты упрощённые версии нескольких persistence-модулей и потеряны более ранние guard-функции.
- **Исправление:** восстановлены и объединены, а не заменены, следующие проверки: совместимость device section с MAC, collision loop DHCP host, `requireScheduleSection()`, проверка существования и типа редактируемой группы, duplicate group-name guard и duplicate administrator-login guard внутри сериализованной мутации.
- **Почему именно так:** UI-ссылка является подсказкой, а не источником истины. Перед записью persistence повторно читает текущие секции и отклоняет stale или wrong-type ссылку вместо молчаливого `uci.set()` в никуда.
- **Регрессия:** экспериментальный набор создаёт wrong-MAC, wrong-type, missing-section и duplicate-name fixtures.

#### 0.6.3. Неопределённый RPC status больше не считается успехом

- **Симптом:** выражение `Number(status || 0) === 0` превращало `undefined`, `null` и пустую строку в успешный ноль.
- **Корневая причина:** слишком широкая нормализация статуса была добавлена при поддержке числовых и строковых ответов ubus.
- **Исправление:** успехом считаются только точные значения `0` и `'0'`; все остальные значения превращаются в структурированную ошибку и запускают cleanup.
- **Почему именно так:** отсутствие ответа не эквивалентно подтверждённому apply. Fail-closed здесь важнее удобного fallback.
- **Регрессия:** `overviewFinalExperimentalR252.test.mjs` отдельно проверяет undefined apply и confirm status.

#### 0.6.4. Уникальность secure-администратора проверяется внутри mutation

- **Симптом:** две кнопки Create были защищены общим submission guard, но проверка существующего login всё ещё происходила до входа в сериализованную UCI-мутацию.
- **Корневая причина:** UI-валидация ошибочно рассматривалась как достаточный persistence guard.
- **Исправление:** непосредственно внутри `uciPersistence.mutate()` повторно перечисляются administrator-секции; совпадение login завершает операцию кодом `administrator_login_exists` до создания секции и сброса полей.
- **Почему именно так:** проверка, зависящая от изменяемого состояния, должна находиться в той же критической секции, что и запись.
- **Регрессия:** source-contract test подтверждает порядок проверки до `administratorSectionName(login)`; runtime test pairing persistence проверяет duplicate login.

#### 0.6.5. Итоговые экспериментальные результаты

- **Изолированный overlay:** 121 успешный тест; шесть test-файлов не стартуют без неизменённых base-файлов репозитория и поэтому не засчитываются как ошибки изменённого кода.
- **Контрактно полный синтетический fixture:** 162 из 162 тестов успешно. Fixture добавляет только минимальные неизменённые контракты base-файлов и никогда не включается в release payload.
- **Финальные hardening/experimental/refresh тесты:** 43 из 43 успешно после объединения guard-ов и последней проверки UCI/refresh-контрактов.
- **Структурный аудит:** 50 LuCI JavaScript-файлов, 82 локальных require, 32 create/dependency contracts и 108 UCI-вызовов; ошибок нет. Предупреждения относятся только к исходным модулям, отсутствующим в overlay-only каталоге.

Эти числа подтверждают модельные и структурные контракты, но по-прежнему не заменяют установленный LuCI/browser smoke, OpenWrt SDK build, реальный rpcd, nftables/DNS/Wi-Fi и физические Android-устройства.

## 1. Зачем потребовалось изменение

До рефакторинга `overview.js` одновременно выполнял слишком много независимых обязанностей:

1. загружал UCI-конфигурации и файлы состояния;
2. хранил изменяемое состояние страницы;
3. строил почти весь DOM LuCI;
4. открывал модальные окна;
5. валидировал формы;
6. изменял UCI для устройств, групп, расписаний, Wi-Fi и администраторов;
7. вызывал backend-команды;
8. применял firewall и DNS runtime;
9. создавал pairing-коды и QR payload;
10. импортировал и восстанавливал резервные копии;
11. управлял вкладками и deep-link;
12. отображал частичные ошибки после успешного сохранения UCI.

Такой файл сложно безопасно изменять. Исправление одной формы могло затронуть общий черновик Settings,
перерисовку списка устройств, pairing или применение firewall. Статический тест мог проверить отдельную
функцию, но не мог доказать отсутствие скрытой зависимости от глобальной переменной в другой части файла.

Главная цель прохода — не просто уменьшить число строк. Цель — сделать границы ответственности явными:

- `overview.js` только создаёт объекты и соединяет их;
- presentation-модули строят DOM, но не сохраняют UCI;
- persistence-модули сохраняют данные, но не знают DOM;
- controller-модули координируют один предметный раздел;
- общий page shell управляет жизненным циклом LuCI и вкладками;
- изменяемое состояние страницы находится в одном небольшом store;
- backend-действия проходят через единый runner;
- частично успешная операция не маскируется сообщением о полном откате.

## 2. Итоговая граница `overview.js`

Финальная граница состоит из двух уровней.

1. `view/sheepfold/overview.js` — стабильный четырёхстрочный LuCI entrypoint. Он загружает `features/overview/application.js` и возвращает уже собранный view. В entrypoint нет состояния, DOM, UCI, backend-команд, branch-specific логики и предметных функций.
2. `features/overview/application.js` — явный composition root размером менее 700 строк. Он создаёт store, environment, command runner, persistence-адаптеры и предметные контроллеры, после чего передаёт их в page shell. В нём нет прямых `uci.set()/unset()/remove()/save()/apply()`, `ui.showModal()/hideModal()/addNotification()` и `fs.exec()`.

Фактическая проверенная форма overlay до применения к полному checkout:

```text
overview.js:    4 строки, 117 байт
application.js: 627 строк
крупнейший предметный модуль: devices/controller.js, 667 строк
все новые controller/application-модули: менее 700 строк
```

Значения в пользовательском checkout перепроверяются установщиком; если лимиты нарушены, применение завершается ошибкой.

### Почему entrypoint сделан именно четырёхстрочным

Путь `view/sheepfold/overview.js` является публичной точкой загрузки LuCI и используется `overview-secure.js`, menu/ACL-контрактами, asset-versioning и существующими ссылками. Переименование этого пути создало бы ненужную миграцию. Поэтому путь сохранён, а реализация вынесена за него.

Четыре строки выбраны не ради рекорда, а ради архитектурного запрета: в entrypoint невозможно незаметно вернуть ещё одну форму, UCI-транзакцию или backend-команду. Любая новая функция должна иметь явного владельца в `features/`.

### Почему composition root оставлен отдельным явным файлом

Полностью убрать центральную сборку нельзя: кто-то должен создать общий store, связать устройства с администраторами, передать группам список устройств, соединить Settings draft с persistence и передать page shell все панели.

Эта сборка оставлена в `features/overview/application.js`, а не размазана по singleton-модулям. Так reviewer видит полный граф полномочий: какой модуль получает UCI, какой получает backend runner, какой может читать файлы, а какой строит только DOM. Это лучше service locator, скрытого global state и циклических импортов.

### Почему отклонён механический runtime-registry split

На промежуточной проверке был испытан автоматический разрез старого монолита на группы функций через runtime registry. Он действительно сокращал entrypoint, но сохранял старые неявные зависимости и мог механически перенести незавершённое wiring предыдущих проходов. В частности, такой разрез не исправлял бы отсутствующую связь country-aware Time/NTP и optional-`nmap` presentation.

Финальная версия использует явные human-readable контроллеры и persistence-адаптеры. Это немного больше кода в `application.js`, но существенно безопаснее: границы читаются без генератора, функции имеют предметных владельцев, а runtime smoke загружает настоящий граф модулей.

## 3. Новые и изменённые файлы финального прохода

Ниже перечислены все файлы, непосредственно добавленные или изменённые на финальном этапе декомпозиции.

### 3.1. `view/sheepfold/overview.js`

**Что изменено**

Файл заменён четырёхстрочным bootstrap:

```javascript
'use strict';
'require sheepfold.features.overview.application as overviewApplication';

return overviewApplication;
```

**Почему изменён этот файл**

Это исходный монолит и одновременно стабильный LuCI entrypoint. Пока в нём оставалась композиция или предметная логика, существовал простой путь снова превратить его в огромный файл.

**Почему выбран именно такой способ**

Путь оставлен прежним для совместимости с menu, ACL, asset cache busting и `overview-secure.js`. Реализация перемещена, а не продублирована. Bootstrap не создаёт второй экземпляр view и не меняет порядок инициализации: LuCI загружает application-модуль один раз и возвращает тот же объект.

**Риски и защита**

Главный риск — старые статические тесты могли продолжить читать только bootstrap. Установщик переводит их на `tools/quality/overviewApplicationSource.mjs`, а отдельный тест проверяет, что сам bootstrap остаётся маленьким.

### 3.2. `features/overview/application.js`

**Что изменено**

В этот файл без изменения публичного view-контракта перенесён явный composition root. Он подключает зависимости, создаёт store/environment/navigation, command runner, UCI/device/Wi-Fi/backup/pairing/schedule/group persistence, предметные контроллеры и возвращает `pageShellModel.create(...)`.

**Почему изменён этот файл**

После предметного разреза осталось около 627 строк явного wiring. Это уже не монолит поведения, но всё ещё слишком много для публичного entrypoint.

Composition root должен быть один и должен быть видимым. Его не стали дополнительно дробить на механические forwarder-модули: это ухудшило бы трассировку зависимостей. **Риски и защита**

Runtime-тест загружает bootstrap, application и фактические локальные LuCI-модули, вызывает `load()` и `render()`, а также проверяет `overview-secure.js`. Standard-вариант отдельно проверяется после удаления AI-блоков.

## 4. Почему использованы ленивые callbacks

Контроллеры естественно образуют несколько взаимных ссылок:

- устройство должно знать, является ли оно администраторским;
- администратор должен найти устройство по постоянному ID;
- группа использует device selector и schedule selector;
- device quick allowlist использует Wi-Fi QR;
- page refresh должен вызвать методы уже созданного LuCI view.

Прямые module imports привели бы к циклу `devices → administrators → devices`. Вместо этого composition root
передаёт функции вида:

```text
administrators: () => administratorController
deviceById: id => deviceController.byId(id)
schedules: () => scheduleController
wifi: () => wifiController
```

Callback вычисляется только в момент пользовательского действия, когда все контроллеры уже созданы.

Почему не service locator:

- зависимости остаются видимыми в конструкторе;
- тест может передать собственный callback;
- модуль не обращается к неявному глобальному registry;
- порядок инициализации документирован в одном месте.

## 5. Сохранение и runtime: почему это две разные стадии

Для OpenWrt операция часто состоит из двух независимых стадий:

1. UCI commit;
2. применение runtime: nftables, DNS filter, Wi-Fi reload или сервисы.

После успешного commit невозможно честно сказать пользователю «ничего не сохранено», если второй этап упал.
Поэтому persistence-модули маркируют ошибки:

```text
persisted = true
runtimeApplied = false
```

Controller в этом случае:

- перечитывает фактическую UCI-конфигурацию;
- обновляет видимую модель;
- не предлагает повторно сохранить уже сохранённое значение;
- показывает отдельное предупреждение о runtime.

Такой контракт применён к устройствам, группам, расписаниям, Wi-Fi и backup service refresh.

## 6. Почему command runner общий

Все изменяющие LuCI-действия проходят через `core/backend/actions.js`.

Он обеспечивает:

- один in-flight Promise на action key;
- блокировку связанных кнопок;
- восстановление их прежнего состояния;
- parsing shell metadata, JSON и key/value;
- единый error object;
- один порядок success callback, refresh и notification.

Это особенно важно после декомпозиции: без общего runner каждый controller снова реализовал бы собственную
защиту от двойного нажатия и несовместимую обработку ошибок.

## 7. Standard и AI Support

AI-only участки остаются внутри:

```text
/* SHEEPFOLD_AI_BEGIN */
...
/* SHEEPFOLD_AI_END */
```

При подготовке Standard-варианта блоки удаляются. Проверка компилирует stripped-версии всех новых модулей.

Почему не runtime `if (productVariant)`:

- Standard-пакет не должен содержать API-key UI и AI presentation;
- package boundary проверяется на содержимом реального архива;

## 13. Пофайловый журнал финального прохода с точными причинами

Этот раздел является исчерпывающим списком файлов, которые добавляет или изменяет финальный аудированный `r252`-проход поверх кумулятивного `r249`. Для файлов, созданных предыдущими проходами, здесь описано только финальное изменение их wiring; их исходная предметная реализация подробно описана выше.

### `tools/quality/overviewApplicationSource.mjs`

**Что изменено:** добавлен единый способ получить путь и текст реального composition root по стабильному пути `overview.js`.

**Почему изменён этот файл:** десятки исторических source-contract тестов читали `overview.js` напрямую. После превращения entrypoint в bootstrap они проверяли бы четыре строки вместо реализации.

**Почему выбран именно такой способ:** тесты по-прежнему начинают со стабильного `overviewPath`, а helper вычисляет соседний application path. Это не дублирует исходники и не создаёт тестовый «ложный монолит».

### `tests/overviewFinalDecomposition.test.mjs`

**Что изменено:** добавлена проверка четырёхстрочного bootstrap, bounded application, ограничений размеров контроллеров, полного LuCI module graph, secure-wrapper dispatch, навигации, global-internet ordering, Standard/AI boundary и наличия этого документа.

**Почему изменён этот файл:** статического `node --check` недостаточно; undefined wiring проявляется только при создании и рендеринге настоящего графа модулей.

**Почему выбран именно такой способ:** VM-loader использует реальные локальные Sheepfold-модули и заменяет только платформенные LuCI примитивы. Поэтому тест ловит пропущенный require/callback, не требуя браузера или роутера.

### `tests/overviewFinalization.test.mjs`

**Что изменено:** закреплены ownership-границы, лимит 700 строк, отсутствие UCI/DOM side effects в store/application, Standard syntax и LED acknowledgement contract.

**Почему изменён этот файл:** runtime smoke подтверждает сборку, но не объясняет, кому принадлежит логика. Нужен отдельный source-level архитектурный gate.

**Почему выбран именно такой способ:** тест проверяет свойства границ, а не полное форматирование файлов; это снижает хрупкость при обычном рефакторинге.

### `tests/categories.mjs`

**Что изменено:** оба финальных теста включаются в `smoke`, `luci`, `devices`, `ai`, `tooling` и `packaging`.

**Почему изменён этот файл:** тест, не назначенный категории, мог не попасть в выбранный quality gate.

**Почему выбран именно такой способ:** пересекающиеся категории отражают реальные риски: LuCI runtime, устройства, product variants, tooling и package boundary.

### `tests/productVariants.test.mjs`

**Что изменено:** packaged bootstrap проверяется отдельно, а AI/require/feature assertions читают packaged `features/overview/application.js`. Оба варианта обязаны содержать bootstrap и application.

**Почему изменён этот файл:** после разреза Standard bootstrap естественно не содержит AI-строк; проверять только его было бы ложноположительным тестом.

**Почему выбран именно такой способ:** проверяется фактический payload IPK, а не рабочее дерево. Standard должен не содержать AI wiring именно в application и downstream modules.

### Исторические tests, читавшие `overview.js`

**Что изменено:** установщик находит вызовы `readFileSync`, `read`, `readProjectFile` и `source`, которые читают стабильный overview path, и переводит их на `readOverviewApplication(...)`. Новые финальные тесты исключены из миграции, потому что они намеренно проверяют bootstrap.

**Почему изменены эти файлы:** их предметные утверждения относятся к composition root/wiring, а не к публичному четырёхстрочному загрузчику.

**Почему выбран именно такой способ:** миграция основана на фактическом выражении пути, а не на жёстком списке имён тестов. Поэтому она покрывает полный checkout и остаётся идемпотентной.

### `eslint.config.js`

**Что изменено:** добавляются readonly-глобалы для новых LuCI aliases, обнаруженных в application и controller-модулях.

**Почему изменён этот файл:** LuCI `require ... as alias` создаёт переменные в module wrapper, которые обычный ESLint иначе считает неопределёнными.

**Почему выбран именно такой способ:** aliases извлекаются из фактических файлов. Это исключает ручное рассогласование списка после следующего перемещения модуля.

**Почему изменён этот файл:** меняется устанавливаемый LuCI payload и asset graph; router package должен получить новый release identifier.

**Почему выбран именно такой способ:** `PKG_VERSION` остаётся `0.1.0`, поскольку продуктовая версия не меняется; инкремент release соответствует правилам OpenWrt.

## 14. Что считается «доделано до точки»

Для этой задачи критерий выполнен, когда одновременно истинны следующие условия:

1. `overview.js` — только стабильный bootstrap и не может расти незаметно.
2. реальная композиция находится в одном bounded application-файле;
3. ни application, ни новый контроллер не превышает 700 строк;
4. application не содержит прямых feature mutations и DOM-модалок;
5. каждый предметный раздел имеет controller/persistence/presentation владельца;
6. country-aware Time/NTP и optional nmap реально подключены в Settings wiring;
7. Standard и AI Support загружаются и синтаксически проверяются отдельно;
8. `overview-secure.js` продолжает работать через virtual dispatch;
9. исторические tests проверяют application, а новые tests отдельно проверяют bootstrap;
10. существует этот подробный пофайловый документ;
11. установщик идемпотентен и не коммитит/не пушит автоматически;
12. оставшиеся аппаратные и browser проверки перечислены честно, а не объявлены выполненными.

## 28. Финальный self-audit `r252` после объединения всех исправлений (§ovaudit5)

Последняя проверка выполнялась уже после повторного объединения UCI guard-ов, безопасных административных привязок, возможности снять существующую постоянную DHCP-аренду и точной проверки документа. Она отличается от предыдущих этапов тем, что проверяет не один тематический разрез, а совместимость всех кумулятивных изменений между собой.

### 28.1. Что было перепроверено

- `core/persistence/uci.js` использует глобальную семантику LuCI staging: один zero-argument `uci.save()`, прямой `callApply(timeout, true)`, повторяемый `callConfirm()` и fail-closed отказ, если нужные RPC-примитивы недоступны;
- отсутствие пустого или `undefined` RPC status больше не считается успехом: успешны только точные значения `0` и `"0"`;
- список конфигураций и UCI-list значения дедуплицируются перед операцией;
- stale device, DHCP, schedule, group и administrator sections проверяются до commit и не могут молча перезаписать секцию другого объекта или типа;
- повторная проверка уникальности administrator login выполняется внутри сериализованной мутации, а не только до открытия Promise-очереди;
- устройство, уже принадлежащее другому администратору, отклоняется с `administrator_device_already_bound`;
- существующую постоянную DHCP-аренду можно явно отключить, при этом старая host-секция действительно удаляется;
- secure administrator modal использует один submission guard для обеих визуальных кнопок создания;
- partial persistence, runtime failure и refresh failure остаются разными наблюдаемыми состояниями.

### 28.3. Почему эта проверка не называется аппаратной сертификацией

Синтетический fixture добавляет минимальные неизменённые base-контракты только для запуска source/runtime tests. Он не эмулирует реальный rpcd, браузерный LuCI loader, nftables, dnsmasq, Wi-Fi driver, OpenWrt package manager или физический Android. Поэтому перед merge по-прежнему обязательны официальный SDK build, installed-router browser audit, firewall/DNS/Wi-Fi matrix и физические Android-сценарии. Такое ограничение записано явно, чтобы зелёные модельные тесты не выдавались за доказательство поведения оборудования.
