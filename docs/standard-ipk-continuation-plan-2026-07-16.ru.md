## Что сделано в последнем проходе

1. В `sheepfold-updater` исправлено сохранение реального кода сетевой ошибки. Раньше `$?` читался после завершения `if`, из-за чего ошибка `uclient-fetch` могла превратиться в успешный код `0`.
2. Загрузка описания релиза и IPK ограничена 120 секундами и 64 МиБ. Добавлены последовательные клиенты `uclient-fetch`, `curl`, `wget`.
3. Telegram long poll и отправка сообщений ограничены 35 секундами; добавлены `curl`/`wget` fallback.
4. Загрузчик источников сайтов сохраняет рабочую копию крупного источника, если новая версия содержит менее 25% прежних доменов. Проверка включается от 1000 доменов.
5. Проверенный администратором уменьшившийся источник можно принять командой `sheepfold-site-lists update --accept-shrink`. Обычное обновление эту защиту не обходит.
6. После ранее отправленного предупреждения о трёх ошибках создаётся одно уведомление о восстановлении источника. Обычный успешный цикл уведомления не создаёт.
7. Добавлены тесты `tests/updateTransportSafety.test.mjs` и новые сценарии в `tests/siteListResilience.test.mjs`.
8. Updater принимает IPK только из официального stable-релиза нужного варианта Sheepfold и до `opkg` проверяет внешний архив, вложенные архивы, безопасные пути и поля `Package`, `Version`, `Architecture` (§updsafe).
9. При ошибке `opkg` или неподтверждённой установленной версии updater восстанавливает прежний UCI-конфиг. Удаление временной резервной копии перенесено из `postinst` в updater, чтобы пакет не уничтожил её до проверки результата. Бинарный откат пока не реализован и не заявляется.

11. На OpenWrt 25.12 результат helper совпал с независимым OpenSSL pipeline. `openssl-util` добавлен в зависимости обеих редакций; `PKG_RELEASE` поднят до `213`, Android до `0.1.41`/code 42. ## Куда смотреть при продолжении backend-аудита

1. **Обновление IPK.** На живом роутере проверить повреждённый пакет, нехватку места, ошибку `opkg install` и случай, когда `opkg` вернул успех, но новая версия не подтверждается. После каждого сценария проверить UCI-конфиг и состав установленных файлов. Конфиг восстанавливается автоматически, но для бинарного отката по-прежнему нужен сохранённый и проверенный предыдущий IPK (§updsafe).
2. **JSON GitHub Release.** Проверить устойчивость разбора JSON и выбор именно пакета текущего варианта. Не разрешать URL из другого репозитория или prerelease-канала.
3. **Списки сайтов.** Безопасный загрузчик подключён к транзакционной доменной политике. Встроенный движок использует локальный dnsmasq `nftset`; управляемый AdGuard Home получает отдельный фильтр Sheepfold через API и при ошибке уступает встроенному резерву. На живом роутере проверить четыре топологии, Basic-аутентификацию, недоступность API, смену IP клиента, сохранность пользовательских фильтров AdGuard Home и отсутствие обхода через DoH/VPN/прямой IP (§slstres, §dompol). Следующие проходы расширения API выполнять по отдельному плану и не превращать текущий флаг в разрешение менять весь AdGuard Home (§aghplan).
4. **LuCI списков.** Статусы, было/стало и десятисекундное подтверждение реализованы. Следующий UI-этап — заменить многострочные поля URL на отдельные редактируемые записи с включением/выключением без удаления и проверить всё на живом роутере.
5. **Уведомления.** Сейчас очередь общая. Проверить модель прочтения двумя родительскими APK и решить вопросы 14-16 и 37 в `owner-open-questions.ru.md` (§ownques).
6. **Firewall и приоритеты.** Белый/чёрный списки устройств, группа `Без ограничений` поверх расписания, чёрный список устройств поверх этой группы, одиночные/ночные/конфликтующие расписания и lifecycle временного доступа уже покрыты синтетическим live-runtime тестом. Продолжить матрицу для глобальной блокировки, переходов по реальным часам, firewall reload и оставшихся сочетаний правил; отдельно сверять status API с реально созданными nftables-наборами.
7. **Событийное автообнаружение и суточная страховка (§detlife1).** Проверить переход offline -> online, 20-секундную очередь, online-only проход после запуска и раз в сутки, блокировку параллельных запусков, отсутствие сканирования выключенных/чёрносписочных устройств, ручной тип и условие автоназначения групп только при включённой автонастройке.
8. **Импорт/экспорт.** Проверить откат после частично применённого UCI, сохранение секретов только в зашифрованном архиве и синхронное применение firewall без перезагрузки всей страницы.
9. **Wi-Fi/WPS/LED.** Не считать универсальными имена кнопок, radio и LED. Проверять обнаруженные аппаратные объекты и путь восстановления Wi-Fi на живом роутере.
10. **API.** Повторно проверить границы публичного `/client-status`, обязательную admin-аутентификацию остальных маршрутов, одноразовость pairing-токена и отсутствие root-пароля в QR. Для сопряжения отдельно снять настоящий QR `SF2` с LuCI, проверить успех на Android и обязательный отказ после подмены одного hex-символа `spki` (§tlspinv2).

## Точка завершения этого прохода

Локальная точка `0.1.0-192` достигнута: кроме прохода устойчивости AdGuard Home добавлено обратимое управление IPv6 для текущих режимов Podkop (§ipv6pod), включая применение после импорта и восстановление runtime при удалении без очистки `/etc/config/sheepfold`. Оба IPK собраны в `.build/ipv6-podkop-ipk`; последний полный результат до завершающих узких правок — `227/227`, более новые предметные результаты перечислены выше.

Следующий локальный проход уже не должен расширять AdGuard Home без новой причины. Практичные варианты: заменить многострочные поля источников отдельными редактируемыми строками с выключением без удаления либо переносить полноценные редакторы расписаний/групп/Wi-Fi из LuCI в родительский Android APK. Живые проверки firewall, DNS-пути, Telegram-подтверждений и четырёх топологий интеграций остаются отдельной обязательной задачей на физическом роутере.
