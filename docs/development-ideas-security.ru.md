# Идеи развития безопасности

Этот файл дополняет `docs/development-ideas.md`. Здесь собраны идеи, которые требуют отдельного проектирования и миграции, но уже должны учитываться в новых изменениях.

## Переход локального Android API с HTTP на HTTPS

### Проблема

Sheepfold обнаруживает роутер и обращается к нему внутри домашней сети. Сейчас прототип использует локальный HTTP. Даже в LAN это не защищает трафик от подмены точки доступа, заражённого устройства, ARP spoofing или ошибочно открытого API на внешнем интерфейсе.

Простое включение произвольного самоподписанного сертификата недостаточно: Android не знает, какому локальному сертификату доверять, а отключение проверки TLS уничтожает смысл HTTPS.

### Предлагаемая модель

1. При первой установке роутер генерирует уникальную пару ключей и сертификат Sheepfold.
2. Закрытый ключ хранится только на роутере с правами root и никогда не попадает в QR, UCI export или журнал.
3. LuCI показывает QR сопряжения с адресом API, одноразовым pairing-кодом и отпечатком публичного сертификата.
4. Android при первом соединении сверяет отпечаток сертификата с QR до отправки pairing-кода.
5. После успешного сопряжения приложение сохраняет pin сертификата вместе с адресом роутера и Bearer-токеном.
6. Все последующие запросы используют HTTPS и certificate pinning. Системное доверие к публичным CA можно использовать дополнительно, но оно не заменяет pin локального роутера.
7. При смене сертификата приложение не должно молча принять новый. Оно показывает понятное предупреждение и требует нового подтверждения в LuCI.

### Discovery

Публичный `/.well-known/sheepfold.json` может оставаться минимальным discovery endpoint, но должен сообщать только:

- идентификатор сервиса;
- HTTPS-порт и путь API;
- версию протокола;
- публичный fingerprint сертификата;
- признак необходимости миграции старого клиента.

Discovery не выдаёт pairing-код, Bearer-токен, список администраторов или диагностические данные.

### Переходный период

- Новые установки сразу предлагают HTTPS.
- Старые роутеры могут временно поддерживать HTTP только на LAN-интерфейсе и только для discovery/перехода.
- Административные операции и AI-контекст с журналами должны переходить на HTTPS первыми.
- Android не должен включать глобальный `usesCleartextTraffic=true`. Если временная совместимость необходима, разрешение cleartext ограничивается конкретной локальной конфигурацией и явно помечается как legacy.
- В интерфейсе нужно показывать статус: `HTTPS закреплён`, `старое HTTP-соединение`, `сертификат изменился`, `нужна повторная привязка`.
- После согласованного срока legacy HTTP удаляется из Android и роутера.

### Сертификат и резервные копии

Нужно заранее решить, переносится ли TLS-идентичность в зашифрованной полной резервной копии:

- перенос сохраняет привязку телефонов после восстановления того же роутера;
- генерация нового сертификата безопаснее при переносе конфигурации на другое устройство;
- рекомендуемый вариант — хранить идентичность в backup только при явной опции «восстановить идентичность роутера», иначе генерировать новую и требовать повторное сопряжение.

### Смена адреса роутера

Pin должен относиться к идентичности Sheepfold, а не только к IP. DHCP, смена подсети или hostname не должны автоматически ломать доверие, если сертификат и подтверждённый идентификатор те же. Однако новый адрес нельзя принимать по неподписанному redirect: приложение сначала проверяет сертификат на новом адресе.

### Что нельзя делать

- отключать hostname/certificate verification;
- доверять любому самоподписанному сертификату;
- помещать закрытый ключ или Bearer-токен в QR;
- автоматически принимать новый fingerprint после ошибки TLS;
- выставлять uhttpd/API на WAN ради удобства мобильного приложения;
- использовать один заводской сертификат или ключ для всех роутеров;
- считать VPN обязательным способом удалённого управления: удалённые сценарии Sheepfold остаются отдельным контролируемым каналом.

### Этапы реализации

1. Описать формат идентичности, fingerprint и QR v2.
2. Добавить генерацию сертификата и безопасное хранение ключа на OpenWRT.
3. Поднять HTTPS listener только на LAN и проверить firewall defaults.
4. Добавить pinning и экран подтверждения в Android.
5. Мигрировать `/pair`, Bearer API и AI endpoint.
6. Добавить ротацию, восстановление backup и обработку смены адреса.
7. Добавить телеметрию только локального состояния миграции без отправки секретов.
8. После периода совместимости удалить legacy HTTP.

### Критерии готовности

- MITM с другим сертификатом блокируется до отправки pairing-кода или Bearer-токена.
- Смена IP при сохранённой идентичности работает после безопасного discovery.
- Смена сертификата требует явного подтверждения в LuCI.
- API недоступен с WAN по умолчанию.
- Резервное копирование и восстановление имеют документированное поведение идентичности.
- Android не содержит универсального разрешения cleartext и не использует небезопасный TrustManager.
