# Тестирование на живом OpenWRT-роутере

Этот документ описывает будущую схему автотестов Sheepfold в связке с настоящим тестовым роутером. Такие тесты полезны, потому что часть ошибок проявляется только на реальном OpenWRT: `opkg`, `uci`, `ubus`, `rpcd`, LuCI, Wi-Fi, firewall, hotplug, права файлов и особенности конкретной прошивки.

Важно: тесты с живым роутером нельзя смешивать с обычными unit-тестами. Они должны запускаться явно, с понятным уровнем риска и с бэкапом конфигурации перед любыми изменениями.

## Тестовый роутер

Желательно иметь отдельный роутер, который не является единственным домашним шлюзом семьи. Идеальная схема:

- роутер на актуальном OpenWRT;
- включён SSH-доступ;
- задан root-пароль;
- LuCI доступен из локальной сети;
- Sheepfold установлен из тестового `.ipk`;
- есть хотя бы одно тестовое устройство в DHCP leases;
- желательно отдельно проверить сценарии с AdGuard Home, Podkop и без них.

Не использовать живой семейный роутер для write/hardware-тестов, если через него прямо сейчас работают работа, учёба, камеры, умный дом или мессенджер-бот.

## Переменные окружения

Тесты должны читать настройки подключения из переменных окружения, а не хранить адреса, пароли и токены в репозитории.

Пример для Windows PowerShell:

```powershell
$env:SHEEPFOLD_ROUTER_HOST="192.168.2.1"
$env:SHEEPFOLD_ROUTER_USER="root"
$env:SHEEPFOLD_ROUTER_SSH_PORT="22"
$env:SHEEPFOLD_ROUTER_API_URL="http://192.168.2.1/cgi-bin/sheepfold-api"
```

Для опасных тестов нужны отдельные флаги:

```powershell
$env:SHEEPFOLD_ROUTER_WRITE_TESTS="1"
$env:SHEEPFOLD_ROUTER_HARDWARE_TESTS="1"
```

Без этих флагов тесты не должны менять UCI, Wi-Fi, firewall, списки устройств, мессенджеры, LED, WPS или установленные пакеты.

## Уровень 1: read-only тесты

Эти тесты можно запускать чаще всего. Они ничего не меняют на роутере.

Команда будущего вида:

```powershell
node --test tests/router-readonly/*.test.mjs
```

Что проверять:

- роутер отвечает по SSH;
- `/cgi-bin/sheepfold-api` отдаёт JSON с `app=sheepfold`;
- `/.well-known/sheepfold.json` существует и содержит `service=sheepfold`, версию, имя роутера и API base URL;
- `uci -q get sheepfold.global.ui_asset_version` совпадает с установленной версией пакета;
- `uci show sheepfold` содержит обязательные секции `global`, `allowlist`, `blocklist`, `group_no_restrictions`, дефолтного администратора и базовые настройки;
- `sheepfold-router-control status` отвечает без ошибки;
- `sheepfold-router-control router-info` отдаёт модель роутера, версию OpenWRT, статус интернета, Wi-Fi-модули, наличие AdGuard Home и Podkop;
- CGI endpoint `/router-info` не отдаёт Wi-Fi-пароли, токены, API-ключи, LuCI cookie, MAC-адреса клиентов и журнал;
- LuCI asset version в UCI не пустой;
- после установки нет очевидных неприменённых UCI-изменений Sheepfold;
- исполняемые файлы имеют нужные права: CGI, `init.d`, `sheepfold-router-control`, `sheepfold-device-detector`, `sheepfold-telegram-bot`.

Read-only тесты должны падать с понятной ошибкой: например, не `ECONNRESET`, а `роутер недоступен по SSH`, `Sheepfold API не установлен`, `ui_asset_version пустой`.

## Уровень 2: осторожные write-тесты

Эти тесты можно запускать только при `SHEEPFOLD_ROUTER_WRITE_TESTS=1`. Перед запуском они обязаны сохранить бэкап:

```sh
uci export sheepfold > /tmp/sheepfold-test-backup.uci
```

После теста они должны восстановить настройки:

```sh
uci import sheepfold < /tmp/sheepfold-test-backup.uci
uci commit sheepfold
```

Что проверять:

- добавить тестовый MAC в белый список и удалить обратно;
- добавить тестовый MAC в чёрный список и удалить обратно;
- проверить, что один MAC нельзя одновременно добавить в белый и чёрный список;
- проверить, что админское устройство нельзя добавить в чёрный список;
- создать тестовую группу, поменять цвет, привязать тестовое устройство, удалить группу;
- проверить, что группу с устройствами нельзя удалить без понятного предупреждения;
- сохранить настройки мессенджера с тестовыми значениями без реального токена;
- сохранить `log_cache_path` и убедиться, что путь не выходит за безопасную область;
- изменить настройку `ui_asset_version` в рамках теста и вернуть обратно;
- проверить, что `uci commit sheepfold` и apply проходят без зависшей LuCI-плашки;
- проверить, что экспорт журнала маскирует чувствительные поля.

Write-тесты не должны:

- менять Wi-Fi SSID/пароль;
- выключать Wi-Fi;
- перезагружать роутер;
- запускать WPS;
- менять firewall на постоянной основе;
- удалять реальные устройства пользователя;
- чистить пользовательские списки без восстановления.

## Уровень 3: установка и обновление пакета

Эти тесты проверяют полный путь `.ipk -> opkg -> LuCI/backend`.

Запускать только на тестовом роутере:

```powershell
$env:SHEEPFOLD_ROUTER_WRITE_TESTS="1"
node --test tests/router-install/*.test.mjs
```

Что проверять:

- скопировать свежий `.ipk` на роутер в `/tmp/sheepfold-test.ipk`;
- выполнить `opkg install /tmp/sheepfold-test.ipk`;
- проверить, что `opkg` не сообщает `Malformed package file`;
- проверить, что `postinst` исполняемый и завершается без `Permission denied`;
- проверить, что default config создан, если его не было;
- проверить, что существующий `/etc/config/sheepfold` не затирается при обновлении;
- проверить, что новая версия записалась в `ui_asset_version`;
- проверить, что `/.well-known/sheepfold.json` обновился;
- проверить, что service/init script отвечает;
- проверить, что uninstall удаляет пакет без удаления пользовательских списков.

Перед тестом обновления нужно иметь backup `/etc/config/sheepfold`.

## Уровень 4: LuCI через браузер

Эти тесты можно делать Playwright-скриптом против реальной LuCI-страницы тестового роутера.

Что проверять:

- вход в LuCI с тестовым паролем;
- открытие страницы Sheepfold без `RPCError`;
- верхние вкладки растянуты и выделяются;
- таблица “Все устройства” помещается без горизонтальной прокрутки на стандартной ширине;
- кнопки “Интернет включен” и “Интернет отключен” отражают текущее состояние и не могут быть активными одновременно;
- настройки сохраняются только через кнопку `Сохранить`;
- после сохранения появляется центральное уведомление `Настройки успешно сохранены`;
- модалки администраторов, выбора устройств, групп и аварийно-полезных сайтов не выходят за экран;
- при изменении белого/чёрного списка не остаётся системная LuCI-плашка “не принятые изменения”;
- CSS/JS обновляются после поднятия `ui_asset_version`, без ручной очистки кэша.

Для этих тестов нужен отдельный тестовый пользователь/пароль LuCI или явное указание, что используется тестовый root-пароль. Пароль нельзя хранить в репозитории.

## Уровень 5: hardware-in-loop

Это самые рискованные тесты. Они трогают физическое состояние роутера и сети. Запускать только при `SHEEPFOLD_ROUTER_HARDWARE_TESTS=1`.

Что можно проверять:

- включение/выключение Wi-Fi по команде Sheepfold;
- дисклеймер и задержку подтверждения перед настройкой авто-выключения Wi-Fi;
- WPS-режим “дать подключиться к Wi-Fi”;
- WPS-режим “дать подключиться к Wi-Fi и добавить устройства в белый список” только на отдельной тестовой сети;
- LED-режимы: дефолтное поведение, отключение, сигнал о новом устройстве;
- возврат LED к дефолтному поведению после входа в LuCI или просмотра уведомления администратором;
- реакцию Sheepfold после `fw4 restart`, `network reload`, `rpcd restart`, `uhttpd restart`, reboot роутера.

Hardware-тесты должны всегда показывать предупреждение перед запуском. Нельзя запускать их автоматически в обычном CI.

## Тесты с Android-приложением

Android-тесты с живым роутером стоит делать отдельно от unit-тестов APK.

Что проверять:

- Android видит `/.well-known/sheepfold.json` в локальной сети;
- QR из LuCI успешно привязывает телефон к выбранному администратору;
- одноразовый код после успешной привязки больше не работает;
- телефон появляется в списке устройств как админское устройство;
- у устройства появляется корона;
- в таблице администраторов устройство появляется в списке “Админские устройства”;
- смена порта Sheepfold на роутере доходит до APK;
- Android не продолжает первичную настройку через мобильную сеть;
- Android показывает предупреждение про реальный MAC, но не блокирует настройку, если это текущая выбранная логика;
- AI-помощник получает router-info/log context только после явных галочек.

## Стоп-правила

Тест должен немедленно остановиться, если:

- роутер не отвечает по SSH;
- не удаётся создать backup UCI;
- router host не похож на ожидаемый тестовый адрес;
- в окружении не выставлен флаг для write/hardware-тестов;
- текущий default gateway компьютера совпадает с тестовым роутером, а тест собирается выключать Wi-Fi или перезагружать сеть;
- в `/etc/config/sheepfold` найдены реальные токены Telegram/VK/AI, а тест не read-only;
- невозможно восстановить backup после write-теста.

## Безопасность тестов

- Не хранить root-пароль, LuCI cookie, Telegram/VK token, AI API key и Wi-Fi-пароли в тестовых файлах.
- Не выводить секреты в stdout/stderr.
- Все тестовые MAC-адреса должны быть явно помечены как тестовые.
- Перед write-тестом сохранять backup, после теста восстанавливать.
- Для тестов списков использовать отдельный фиктивный MAC, которого нет среди реальных устройств.
- Для тестов Telegram/VK использовать отдельного тестового бота или mock-режим.
- Для тестов AI использовать fake provider/mock endpoint, если задача не проверяет реальный провайдер.

## Предлагаемая структура файлов

```text
tests/

  router-readonly/
    routerApi.test.mjs
    uciConfig.test.mjs
    filePermissions.test.mjs
  router-write/

```

## Порядок внедрения

1. Сделать read-only тесты SSH/API/UCI.
2. Добавить проверку прав файлов и структуры установленного пакета.
3. Добавить write-тесты белого/чёрного списка с backup/restore.
4. Добавить тесты сопряжения Android с одноразовым кодом.
5. Добавить Playwright-тесты LuCI на тестовом роутере.
6. Только после этого добавлять hardware-in-loop тесты Wi-Fi/WPS/LED.

Главная идея: живой роутер нужен не для того, чтобы заменить unit-тесты, а чтобы ловить реальные OpenWRT-ошибки, которые невозможно честно увидеть в обычном Node/Android тесте.
