# Тестирование на живом OpenWRT-роутере

Этот документ описывает полную матрицу автотестов Sheepfold в связке с настоящим тестовым роутером. Такие тесты полезны, потому что часть ошибок проявляется только на реальном OpenWrt: `opkg`/`apk`, `uci`, `ubus`, `rpcd`, LuCI, Wi-Fi, firewall, hotplug, права файлов и особенности конкретной прошивки.

<!-- §routerharness -->

Безопасная исполняемая основа уже находится в `tools/router-testing/`: она умеет подготовить SSH-ключ, определить менеджер пакетов, установить IPK либо SDK-собранный OpenWrt APK с backup, выполнить read-only и восстанавливаемые UCI-проверки, а также проверить LuCI через Playwright. Команды и точные границы описаны в [`live-router-automation.ru.md`](live-router-automation.ru.md). Остальные сценарии этого документа являются планом расширения и не должны считаться реализованными только потому, что они перечислены ниже.

Важно: тесты с живым роутером нельзя смешивать с обычными unit-тестами. Они должны запускаться явно, с понятным уровнем риска и с бэкапом конфигурации перед любыми изменениями.

## Тестовый роутер

Желательно иметь отдельный роутер, который не является единственным домашним шлюзом семьи. Идеальная схема:

- роутер на актуальном OpenWRT;
- включён SSH-доступ;
- задан root-пароль;
- LuCI доступен из локальной сети;
- Sheepfold установлен из пакета формата целевой версии OpenWrt;
- есть хотя бы одно тестовое устройство в DHCP leases;
- желательно отдельно проверить сценарии с AdGuard Home, Podkop и без них.

Не использовать живой семейный роутер для write/hardware-тестов, если через него прямо сейчас работают работа, учёба, камеры, умный дом или мессенджер-бот.

## Переменные окружения

Тесты должны читать настройки подключения из переменных окружения, а не хранить адреса, пароли и токены в репозитории.

Пример для Windows PowerShell:

```powershell
$env:SHEEPFOLD_ROUTER_HOST="192.168.2.1"
$env:SHEEPFOLD_ROUTER_USER="root"
$env:SHEEPFOLD_ROUTER_SSH_PORT="22"
$env:SHEEPFOLD_ROUTER_API_URL="https://192.168.2.1:5201/cgi-bin/sheepfold-api"
```

Для опасных тестов нужны отдельные флаги:

```powershell
$env:SHEEPFOLD_ROUTER_WRITE_TESTS="1"
$env:SHEEPFOLD_ROUTER_HARDWARE_TESTS="1"
```

Без этих флагов тесты не должны менять UCI, Wi-Fi, firewall, списки устройств, мессенджеры, LED, WPS или установленные пакеты.

## Уровень 1: read-only тесты

Эти тесты можно запускать чаще всего. Они ничего не меняют на роутере.

Исполняемая команда:

```powershell
npm.cmd run router:readOnly
```

Что проверять:

- роутер отвечает по SSH;
- `/cgi-bin/sheepfold-api` отдаёт JSON с `app=sheepfold`;
- `/.well-known/sheepfold.json` существует и содержит `service=sheepfold`, версию, имя роутера и API base URL;
- `uci -q get sheepfold.global.ui_asset_version` совпадает с установленной версией пакета;
- `uci show sheepfold` содержит обязательные секции `global`, `allowlist`, `blocklist`, `group_no_restrictions`, дефолтного администратора и базовые настройки;
- `sheepfold-router-control status` отвечает без ошибки;
- `sheepfold-router-control router-info` отдаёт модель роутера, версию OpenWRT, статус интернета, Wi-Fi-модули, наличие AdGuard Home и Podkop;
- CGI endpoint `/router-info` не отдаёт Wi-Fi-пароли, токены, API-ключи, LuCI cookie, MAC-адреса клиентов и журнал;
- LuCI asset version в UCI не пустой;
- после установки нет очевидных неприменённых UCI-изменений Sheepfold;
- исполняемые файлы имеют нужные права: CGI, `init.d`, `sheepfold-router-control`, `sheepfold-device-detector`, `sheepfold-telegram-bot`.

Read-only тесты должны падать с понятной ошибкой: например, не `ECONNRESET`, а `роутер недоступен по SSH`, `Sheepfold API не установлен`, `ui_asset_version пустой`.

## Уровень 2: осторожные write-тесты

Эти тесты можно запускать только при `SHEEPFOLD_ROUTER_WRITE_TESTS=1`. Перед запуском они обязаны сохранить бэкап:

```sh
uci export sheepfold > /tmp/sheepfold-test-backup.uci
```

После теста они должны восстановить настройки:

```sh
uci import sheepfold < /tmp/sheepfold-test-backup.uci
uci commit sheepfold
```

Что проверять:

- добавить тестовый MAC в белый список и удалить обратно;
- добавить тестовый MAC в чёрный список и удалить обратно;
- проверить, что один MAC нельзя одновременно добавить в белый и чёрный список;
- проверить, что админское устройство нельзя добавить в чёрный список;
- создать тестовую группу, поменять цвет, привязать тестовое устройство, удалить группу;
- проверить, что группу с устройствами нельзя удалить без понятного предупреждения;
- сохранить настройки мессенджера с тестовыми значениями без реального токена;
- сохранить `log_cache_path` и убедиться, что путь не выходит за безопасную область;
- изменить настройку `ui_asset_version` в рамках теста и вернуть обратно;
- проверить, что `uci commit sheepfold` и apply проходят без зависшей LuCI-плашки;
- проверить, что экспорт журнала маскирует чувствительные поля.

Write-тесты не должны:

- менять Wi-Fi SSID/пароль;
- выключать Wi-Fi;
- перезагружать роутер;
- запускать WPS;
- менять firewall на постоянной основе;
- удалять реальные устройства пользователя;
- чистить пользовательские списки без восстановления.

### Матрица firewall и интеграций

<!-- §v7x2k9p -->

На отдельном тестовом роутере пройти четыре профиля: `none`, `adguard`, `podkop`, `adguard_podkop`. Это один firewall-движок Sheepfold, а не четыре реализации правил. Для каждого профиля:

- проверить `Выключить IPv6 на роутере`: без Podkop значение по умолчанию `Нет`, с Podkop — автоматически `Да`; после ухода с Podkop должны восстановиться прежние kernel-значения (§ipv6pod);
- импортировать резервную копию с включённым и выключенным IPv6 и убедиться, что runtime меняется сразу; обновление IPK не должно кратковременно возвращать старое состояние, а удаление пакета должно восстановить исходные kernel-значения, сохранив `/etc/config/sheepfold`;

- выполнить `fw4 print` и убедиться, что package drop-in файлы Sheepfold дают валидный ruleset;
- проверить наличие только собственных sets/chains `sheepfold_*` в `table inet fw4`;
- добавить тестовый MAC в чёрный список устройств и проверить отсутствие интернета и доступа к LuCI/SSH/API;
- создать для тестового устройства два пересекающихся расписания одного уровня с противоположными действиями; при `schedule_conflict_internet=off` проверить блокировку, при `on` — разрешение, а в обоих случаях наличие предупреждения/API-флага и ровно одной новой записи конфликта в журнале;
- отдельно проверить ночной интервал, например понедельник `22:00-07:00`, до и после полуночи, затем дождаться минутной границы без ручного сохранения и убедиться, что nftables и `/client-status` изменились согласованно;
- включить глобальную блокировку и проверить доступ настоящего allowlist/админского устройства, а также блокировку обычного устройства даже при ранее выданном временном доступе;
- выполнить `/etc/init.d/firewall reload` и убедиться, что фоновый сервис восстановил элементы Sheepfold sets;
- удалить пакет и убедиться, что собственные sets очищены, а чужие правила продолжают работать.

Для профилей с Podkop сохранить до теста вывод `nft list table inet PodkopTable`, `ip rule` и относящиеся к Podkop маршруты/марки, а после каждого действия сравнить их с исходным состоянием. Sheepfold не должен выполнять `flush ruleset`, удалять таблицу Podkop, менять packet marks, `ip rule`, `ip route`, Dnsmasq или sing-box. Для профилей с AdGuard Home отдельно проверить DNS после blocklist/global-block и после их отключения.

### Матрица белых и чёрных списков сайтов

<!-- §dompol -->

Подробный runtime-контракт находится в [`site-domain-policy.ru.md`](site-domain-policy.ru.md). На живом роутере дополнительно проверить:

- при `site_filter_backend=sheepfold` во всех подходящих профилях создаётся валидный `/tmp/dnsmasq.d/sheepfold-domain-policy.conf`, dnsmasq отвечает, а нужные IPv4/IPv6 появляются только в наборах `sheepfold_site_*`;
- при `site_filter_backend=auto` без AdGuard Home используется встроенный backend, а при профиле `adguard`/`adguard_podkop` локальный файл удаляется только после подтверждения фильтра `Sheepfold family site policy` через API;
- при `site_filter_backend=adguard` проверить правильные и неправильные credentials, недоступный API, выключенную DNS-фильтрацию AdGuard Home и ответ с повреждённым JSON: во всех ошибочных случаях LuCI показывает причину и реально работает встроенный fallback;
- при `adguard_auto_manage=0` Sheepfold не вызывает `add_url`, `set_url` или `refresh`, удаляет собственную локальную политику только как осознанную ручную передачу и показывает жёлтый непроверенный статус;
- до и после refresh сравнить пользовательские фильтры и custom rules AdGuard Home: Sheepfold не изменяет их, не вызывает `filtering/set_rules` и управляет только своим URL-фильтром;
- переключить backend с AdGuard Home на Sheepfold и обратно; старый управляемый фильтр отключается, а ошибка отключения даёт предупреждение о возможном двойном применении без потери встроенной фильтрации;
- изменить DHCP-адрес тестового устройства и проверить обновление `$client=<IPv4>` после события аренды; устройство без актуального IPv4 вызывает fallback, а не ложный зелёный статус;
- получить локальную ленту правильным токеном с `127.0.0.1`, затем проверить отказ по неверному токену и с LAN-адреса; пароль и токен не появляются в status, системном журнале и безопасном экспорте;
- устройство строгой группы открывает разрешённый домен и аварийно-полезный сайт, но не открывает случайный сайт, прямой IP и произвольный не-web порт;
- каждый режим чёрного списка сайтов затрагивает именно нужные устройства; исключённое из чёрного списка сайтов устройство всё равно блокируется глобальной кнопкой;
- чёрный список устройств остаётся сильнее доменных правил и закрывает LuCI, SSH и Sheepfold API;
- при dnsmasq без `nftset` строгая группа не теряет весь интернет, локальная фильтрация не считается активной, а администратор получает одно уведомление;
- ошибка `dnsmasq --test`, ошибка перезапуска DNS и ошибка `sheepfold-firewall domain-reset` сохраняют прежний DNS-файл, прежний marker готовности и прежние nftables-наборы;
- параллельный update/tick не оставляет частично применённое состояние, а просроченный lock восстанавливается;
- неожиданно укороченный либо полностью плохой внешний источник не заменяет последнюю рабочую копию; после трёх неудачных циклов приходит уведомление, после восстановления — одно сообщение о восстановлении;
- перезапуск firewall и роутера восстанавливает политику после загрузки RAM-кэша без изменения таблиц и marks Podkop;
- общий CDN-адрес проверяется отдельно: возможная побочная блокировка фиксируется как известное ограничение, а не объявляется успешной точной фильтрацией;
- DoH, внешний DNS, VPN и прямой IP проверяются как известные пути обхода. Текущая версия не должна обещать их блокировку.

До успешного прохождения этой матрицы новый firewall-код нельзя считать готовым к слиянию, даже если локальные статические тесты зелёные.

## Уровень 3: установка и обновление пакета

Эти тесты проверяют оба полных пути: `.ipk -> opkg -> LuCI/backend` и `OpenWrt .apk -> apk v3 -> LuCI/backend` (§pkgmgr1).

Запускать только на тестовом роутере:

```powershell
$env:SHEEPFOLD_ROUTER_WRITE_TESTS="1"
node --test tests/router-install/*.test.mjs
```

Что проверять:

- определить менеджер пакетов и скопировать свежий IPK либо SDK-собранный OpenWrt APK в отдельный каталог `/tmp`;
- выполнить `opkg install` либо `apk add --allow-untrusted` через тестовый harness;
- проверить, что менеджер пакетов принимает контейнер, а внутренние имя, версия и архитектура совпадают;
- проверить, что `postinst` исполняемый и завершается без `Permission denied`;
- проверить, что default config создан, если его не было;
- проверить, что существующий `/etc/config/sheepfold` не затирается при обновлении;
- через штатную кнопку установить нормальный release asset и убедиться, что журнал показывает подтверждённую установленную версию;
- в отдельном контролируемом тесте подменить ответ загрузчика повреждённым файлом и убедиться, что установка не запускалась;
- в отдельных wrapper-тестах заставить `opkg` и `apk` вернуть ненулевой код после изменения копии конфига: updater должен вернуть тот же код и восстановить исходный файл;
- проверить, что новая версия записалась в `ui_asset_version`;
- проверить, что `/.well-known/sheepfold.json` обновился;
- проверить, что service/init script отвечает;
- проверить, что uninstall удаляет пакет без удаления пользовательских списков.

Перед тестом обновления нужно иметь внешний backup `/etc/config/sheepfold`. Встроенное восстановление UCI проверяется отдельно и не считается откатом бинарных файлов пакета (§updsafe).

## Уровень 4: LuCI через браузер

Эти тесты можно делать Playwright-скриптом против реальной LuCI-страницы тестового роутера.

Что проверять:

- вход в LuCI с тестовым паролем;
- открытие страницы Sheepfold без `RPCError`;
- верхние вкладки растянуты и выделяются;
- таблица “Все устройства” помещается без горизонтальной прокрутки на стандартной ширине;
- кнопки “Интернет включен” и “Интернет отключен” отражают текущее состояние и не могут быть активными одновременно;
- настройки сохраняются только через кнопку `Сохранить`;
- после сохранения появляется центральное уведомление `Настройки успешно сохранены`;
- модалки администраторов, выбора устройств, групп и аварийно-полезных сайтов не выходят за экран;
- при изменении белого/чёрного списка не остаётся системная LuCI-плашка “не принятые изменения”;
- CSS/JS обновляются после поднятия `ui_asset_version`, без ручной очистки кэша.

## Аварийно-полезные сайты

Подробная матрица и ограничения описаны в [`emergency-sites-runtime.ru.md`](emergency-sites-runtime.ru.md) (§emerg1). На живом роутере обязательно проверить сохранение UCI-карточек, dnsmasq с `nftset` и без него, IPv4/IPv6, глобальную блокировку, расписание и чёрный список устройств. Для чёрного списка устройств разрешённый сайт может открываться, но LuCI, SSH и Sheepfold API должны оставаться недоступными.

Для этих тестов нужен отдельный тестовый пользователь/пароль LuCI или явное указание, что используется тестовый root-пароль. Пароль нельзя хранить в репозитории.

## Уровень 5: hardware-in-loop

Это самые рискованные тесты. Они трогают физическое состояние роутера и сети. Запускать только при `SHEEPFOLD_ROUTER_HARDWARE_TESTS=1`.

Что можно проверять:

- включение/выключение Wi-Fi по команде Sheepfold;
- дисклеймер и задержку подтверждения перед настройкой авто-выключения Wi-Fi;
- WPS-режим “дать подключиться к Wi-Fi”;
- WPS-режим “дать подключиться к Wi-Fi и добавить устройства в белый список” только на отдельной тестовой сети;
- LED-режимы: дефолтное поведение, отключение, сигнал о новом устройстве;
- возврат LED к дефолтному поведению после входа в LuCI или просмотра уведомления администратором;
- реакцию Sheepfold после `fw4 restart`, `network reload`, `rpcd restart`, `uhttpd restart`, reboot роутера.

Hardware-тесты должны всегда показывать предупреждение перед запуском. Нельзя запускать их автоматически в обычном CI.

## Тесты с Android-приложением

Android-тесты с живым роутером стоит делать отдельно от unit-тестов APK.

Что проверять:

- Android видит `/.well-known/sheepfold.json` в локальной сети;
- QR из LuCI успешно привязывает телефон к выбранному администратору;
- одноразовый код после успешной привязки больше не работает;
- телефон появляется в списке устройств как админское устройство;
- у устройства появляется корона;
- в таблице администраторов устройство появляется в списке “Админские устройства”;
- смена порта Sheepfold на роутере доходит до APK;
- Android не продолжает первичную настройку через мобильную сеть;
- Android показывает предупреждение про реальный MAC, но не блокирует настройку, если это текущая выбранная логика;
- AI-помощник получает router-info/log context только после явных галочек.

## Импорт и резервные копии

Сценарии из [`settings-backup.ru.md`](settings-backup.ru.md) выполнять только после отдельной копии `/etc/config/sheepfold`, `/etc/config/dhcp` и `/etc/config/wireless` по SSH.

Проверить два маршрута:

- читаемый JSON сохраняет текущие совпадающие секреты, но не переносит их на чистый роутер;
- зашифрованная копия восстанавливает секреты только с правильным паролем.

После импорта сверить UCI, nftables-наборы, постоянные DHCP-аренды, SSID/каналы и повторное открытие LuCI после Wi-Fi reload. Повреждённый файл, неверный пароль и конфликт белого/чёрного списков устройств не должны менять конфиг. Bearer-токены Android не переносятся: приложения на другом роутере сопрягаются заново. (§cfgbak1)

## Стоп-правила

Тест должен немедленно остановиться, если:

- роутер не отвечает по SSH;
- не удаётся создать backup UCI;
- router host не похож на ожидаемый тестовый адрес;
- в окружении не выставлен флаг для write/hardware-тестов;
- текущий default gateway компьютера совпадает с тестовым роутером, а тест собирается выключать Wi-Fi или перезагружать сеть;
- в `/etc/config/sheepfold` найдены реальные токены Telegram/VK/AI, а тест не read-only;
- невозможно восстановить backup после write-теста.

## Безопасность тестов

- Не хранить root-пароль, LuCI cookie, Telegram/VK token, AI API key и Wi-Fi-пароли в тестовых файлах.
- Не выводить секреты в stdout/stderr.
- Все тестовые MAC-адреса должны быть явно помечены как тестовые.
- Перед write-тестом сохранять backup, после теста восстанавливать.
- Для тестов списков использовать отдельный фиктивный MAC, которого нет среди реальных устройств.
- Для тестов Telegram/VK использовать отдельного тестового бота или mock-режим.
- Для тестов AI использовать fake provider/mock endpoint, если задача не проверяет реальный провайдер.

## Порядок внедрения

1. Готово: read-only тесты SSH/UCI/backend, права файлов и структура установленного пакета.
2. Готово: установка IPK с внешним backup, SHA-256 и post-install проверкой.
3. Готово: write-тесты белого и чёрного списков устройств с фиктивным MAC и backup/restore.
4. Готово: read-only Playwright smoke-тест LuCI на desktop/mobile.
5. Следующий этап: pairing Android с одноразовым кодом и тестовым телефоном/эмулятором.
6. После отдельной разработки стоп-правил: hardware-in-loop тесты Wi-Fi/WPS/LED.

Главная идея: живой роутер нужен не для того, чтобы заменить unit-тесты, а чтобы ловить реальные OpenWRT-ошибки, которые невозможно честно увидеть в обычном Node/Android тесте.
