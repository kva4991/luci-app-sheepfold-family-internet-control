# Исследование автоопределения устройств

Итоговая фактическая архитектура и полный жизненный цикл собраны в
[device-passport-and-control.ru.md](device-passport-and-control.ru.md) (§devpas1). Здесь остаются
сравнение внешних проектов, обоснование выбранных решений и ещё не реализованные направления.

Этот документ дополняет рабочий контракт из `device-detection.ru.md`. Здесь собраны
полезные решения других проектов, лицензионные ограничения, направления развития и
вопросы, которые ещё требуют продуктового решения. Текущая реализация Sheepfold остаётся
локальной, событийной и рассчитанной на ограниченные ресурсы OpenWrt (§detlife1, §detload).

## Что изучено

### Home Assistant

Home Assistant разделяет устойчивые `identifiers` и сетевые `connections`. При наличии
обоих сначала ищется устойчивый идентификатор. IP, имя устройства, hostname, URL и имя
пользователя прямо перечислены как неподходящие уникальные идентификаторы. Реестр также
обнаруживает столкновение, когда один идентификатор или connection уже принадлежат другому
устройству.

Полезно для Sheepfold:

- разделять предполагаемое физическое устройство и отдельные сетевые подключения, но не
  создавать связь без решения родителя;
- считать serial/UUID идентификатором, а MAC сетевым подключением;
- явно обнаруживать коллизии устойчивых идентификаторов;
- не использовать IP или имя как доказательство личности;
- не давать автоматике перезаписывать ручное имя и ручной тип.

Источники: [Device registry](https://developers.home-assistant.io/docs/device_registry_index/),
[Entity registry](https://developers.home-assistant.io/docs/entity_registry_index/),
[исходный код device_registry.py](https://github.com/home-assistant/core/blob/dev/homeassistant/helpers/device_registry.py).
Лицензия Home Assistant: Apache-2.0. Архитектурный приём можно реализовать самостоятельно;
при прямом переносе кода потребуются сохранение лицензии и уведомлений.

### Fingerbank и PacketFence

Fingerbank сопоставляет не один признак, а комбинацию DHCP fingerprint, DHCP vendor,
MAC/OUI, User-Agent, mDNS-сервисов, сетевого поведения и других сигналов. Уверенность
зависит не только от найденных совпадений: конкурирующие признаки другого типа уменьшают
итоговый score. PacketFence умеет реагировать на смену класса устройства при прежнем MAC.

Полезно для Sheepfold:

- хранить баллы отдельно для каждого кандидата, а не только суммировать баллы победителя;
- уменьшать уверенность, если сильные признаки одновременно поддерживают разные типы;
- показывать родителю не только процент, но и понятную причину неопределённости;
- собирать порядок DHCP Option 55 и Vendor Class Option 60 локально без packet capture —
  это уже делает DHCP hotplug Sheepfold;
- разделять `тип устройства изменился` и `личность устройства не совпала`.

Источники: [Fingerbank API и scoring](https://api.fingerbank.org/api_doc/2/combinations/interrogate.html),
[локальная интеграция Fingerbank в PacketFence](https://www.packetfence.org/doc/PacketFence_Installation_Guide.html).
Публичный API нельзя включать по умолчанию: запрос может передавать внешнему оператору
MAC, DHCP fingerprint, mDNS и поведенческие данные. Возможная интеграция должна быть
отдельной, добровольной и с точным предварительным перечнем отправляемых полей.

### NetAlertX

NetAlertX принимает результаты ARP, DHCP, mDNS, Nmap, SNMP и импортёров через единый
plugin-контракт. Текущий снимок отделён от постоянного реестра устройств. Для полей
хранится источник значения; значения `USER` и `LOCKED` плагины не перезаписывают. Также
есть состояния для ожидаемого сна устройства и частого переключения online/offline.

Полезно для Sheepfold:

- ввести единый контракт коллектора: источник, семейство признака, значение, время,
  надёжность и срок годности;
- хранить происхождение автоматически определённых имени, типа, модели и группы;
- расширить ручную блокировку значения с типа на другие редактируемые поля;
- распознавать flapping и не создавать очередь/уведомление на каждое краткое исчезновение;
- различать `offline`, `временно спит` и `нестабильное подключение`, не меняя из-за этого
  политику доступа.

Источники: [NetAlertX](https://github.com/netalertx/NetAlertX),
[источники полей](https://docs.netalertx.com/DEVICE_SOURCE_FIELDS/),
[защита ручных значений](https://docs.netalertx.com/DEVICE_FIELD_LOCK/).
Лицензия NetAlertX: GPL-3.0. Код нельзя переносить в MIT-часть Sheepfold без изменения
лицензионной модели; идеи и независимо написанную реализацию использовать можно.

### WatchYourLAN

WatchYourLAN строит текущий MAC-снимок через `arp-scan`, сравнивает его с реестром,
записывает online/offline-историю и уведомляет о неизвестном MAC. Проект небольшой и
понятный, но считает MAC основной личностью и не решает подмену или объединение нескольких
MAC одного физического устройства.

Полезно для Sheepfold:

- простой переход состояний `предыдущий снимок -> текущий снимок`;
- отдельная история присутствия без повторной классификации;
- уведомление только при появлении действительно новой записи.

Источник: [scan-routine.go](https://github.com/aceberg/WatchYourLAN/blob/main/backend/internal/routines/scan-routine.go).
Лицензия MIT совместима с Sheepfold, но наша событийная реализация уже точнее и легче для
роутера, поэтому прямое копирование сейчас не даёт выигрыша.

### arpwatch

arpwatch хранит историю пар IP/MAC и различает `new station`, `changed ethernet address`,
`flip flop` и возврат давно известного адреса. Полезна сама идея называть повторяющиеся
переходы отдельным состоянием, а не рассылать одинаковые тревоги.

Источник: [arpwatch manual](https://man.archlinux.org/man/arpwatch.8.en).
Лицензия BSD совместима, однако один ARP не доказывает физическую личность устройства.

### ntopng и nDPI

ntopng объединяет ARP, SSDP, mDNS и SNMP; nDPI дополнительно извлекает DHCP, TCP и TLS
fingerprint. Это подтверждает ценность многопротокольного подхода, но полная система
требует значительно больше памяти и процессора, чем допустимо для базового IPK.

Источники: [Network Discovery](https://www.ntop.org/guides/ntopng/user_interface/network_interface/monitoring/network_discovery.html),
[описание активного обнаружения](https://www.ntop.org/network-device-discovery-part-1-active-discovery/).
Лицензия GPL-3.0. nDPI или ntopng допустимы только как будущая внешняя/опциональная
интеграция, а не как скопированная часть стандартного пакета.

### Nmap и MUD

Nmap полезен как ограниченный тяжёлый источник сведений об открытых сервисах, TCP/IP stack
и типе ОС. Частичный fingerprint может соответствовать множеству систем, поэтому результат
нельзя считать удостоверением личности. Sheepfold должен вызывать установленный `nmap`,
а не копировать его базу fingerprint.

Стандарт MUD позволяет IoT-устройству объявить через DHCP или LLDP HTTPS-ссылку на описание
назначения и необходимого сетевого доступа. Поддержка пока редкая, но MUD URL можно в
будущем учитывать как признак производителя и типа после строгой проверки источника.

Источники: [Nmap OS detection](https://nmap.org/book/man-os-detection.html),
[RFC 8520](https://www.rfc-editor.org/info/rfc8520/).

## Зачем Sheepfold нужны два отпечатка

Слово `отпечаток` используется для двух задач с разной ценой ошибки. Объединять их в один
hash нельзя (§devident1, §detlife1).

### 1. Классификационный отпечаток

Это дешёвый служебный cache key. Он отвечает только на вопрос:

> Изменились ли входные данные настолько, что тип устройства стоит определить заново?

Сейчас в материал входят версия правил, MAC, автоматически найденное и постоянное имя,
DHCP-сигналы, известные порты, mDNS/SSDP-профили и набор уже захешированных identity keys.
Из нормализованного материала вычисляется один hash и сохраняется как
`detection_fingerprint`.

Свойства:

- при любом значимом изменении весь hash может стать другим;
- старое значение разрешено перезаписать после успешного анализа;
- изменение `POLICY_VERSION` намеренно инвалидирует кэш;
- отсутствие совпадения не означает атаку, а лишь разрешает переклассификацию;
- криптографическая стойкость не обязательна: коллизия здесь грозит устаревшим типом, а не
  выдачей прав;
- ложное изменение стоит только ограниченного дополнительного анализа.

Включение identity keys в cache key не смешивает решения. Сначала отдельный identity
механизм решает, можно ли доверять подключению, и только затем классификационный hash
решает, нужен ли повторный расчёт типа.

### 2. Доверенный отпечаток личности

Это не один общий hash, а ограниченный набор элементов вида `семейство:хеш`. Он отвечает:

> Похоже ли текущее подключение на то физическое устройство, которому родитель уже дал
> эти права?

Сейчас сильными семействами считаются протокол-независимый device UUID, UPnP UUID/serial,
WS-Discovery endpoint UUID и serial/device UUID из mDNS TXT.
DHCP client ID, не являющийся MAC или `01+MAC`, и mDNS hostname считаются слабыми.
Сырые значения не записываются: после нормализации сохраняются не более 16 укороченных
HMAC-SHA-256 компонентов версии 2. Локальный случайный секрет не экспортируется и
сохраняется при `sysupgrade`.

Набор компонентов нужен вместо одного объединённого hash по важной причине: устройство
может сегодня не объявить один из своих сервисов. Сравнение по семействам позволяет
получить `совпадение`, `подозрительное противоречие` или `недостаточно данных`, а не считать
любое пропавшее объявление подменой.

Свойства:

- baseline создаётся по принципу trust on first use только при одном сильном или двух
  независимых слабых семействах;
- baseline нельзя автоматически заменить новым наблюдением;
- положительное сильное совпадение либо два слабых совпадения подтверждают устройство;
- противоречащий сильный UUID/serial либо два противоречащих слабых семейства включают
  карантин;
- отсутствие данных не подтверждает и не опровергает личность;
- сменить baseline можно только явным действием родителя `Доверять текущему подключению`.

### Разная цена ошибки

| Ситуация | Классификационный hash | Доверенный identity baseline |
|---|---|---|
| Изменилось имя | Можно пересчитать тип | Не доказательство подмены |
| Изменился IP | Обновить наблюдение | Никогда не идентификатор |
| Пропал mDNS | Возможно изменился кэш | `Недостаточно данных` |
| Изменился сильный UUID | Пересчитать тип после проверки | Подозрительное подключение |
| Обновились правила | Инвалидировать hash | Baseline не менять |
| Родитель выбрал тип | Классификацию больше не запускать | Identity проверять дальше |

Если оставить один отпечаток из всех полей, изменение IP или имени начнёт блокировать
настоящие устройства. Если оставить только устойчивый identity baseline, Sheepfold не
заметит, что появились новые сведения для определения типа. Поэтому два механизма нужны
не ради усложнения, а чтобы не смешивать кэш производительности с решением безопасности.

## Реализованное техническое усиление

Identity-компоненты больше не используют общий hash с резервным MD5. Для новых записей
обязателен HMAC-SHA-256 с отдельным случайным секретом роутера. Секрет создаётся атомарно,
имеет права `0600`, сохраняется при `sysupgrade` и не входит в экспорт настроек. Версия
baseline хранится отдельно.

Старые `family:digest` не заменяются вслепую: детектор сначала сравнивает текущее наблюдение
со старым форматом и только после положительного сильного/многофакторного совпадения
записывает HMAC-версию. Это предотвращает массовый ложный карантин после обновления.
HMAC не шифрует и не делает сетевое объявление криптографически достоверным, но не позволяет
сравнивать сохранённый digest с заранее перебранным словарём без локального секрета.

## Что перенято и что осталось

| Приём | Состояние в Sheepfold |
|---|---|
| Конкурирующие кандидаты Fingerbank | Реализован ограниченный штраф до 27% за признаки другого типа; причины сохраняются отдельно |
| Происхождение полей NetAlertX | Для имени, типа и группы сохраняется `user`/`router`/`detector`; ручные значения автоматика не перезаписывает |
| Anti-flapping | Реализован 90-секундный grace перед реальным offline и новым событием подключения |
| DHCP Option 55/60 | Уже собираются dnsmasq hotplug-скриптом в `/tmp` без сохранения пакетов |
| Индикатор identity | Зелёный значок означает сильный/многофакторный baseline, красный перечёркнутый — только MAC |
| HMAC identity baseline | Реализован HMAC-SHA-256 v2 с безопасной миграцией и сохранением секрета при `sysupgrade` |
| Новый MAC с прежними признаками | Только подсказка родителю; автоматической связи и переноса прав нет |
| Единый evidence-контракт всех коллекторов | Частично: DHCP, mDNS, SSDP/UPnP и WS-Discovery нормализуются detector в общие profile/identity/evidence поля; отдельный plugin API пока не нужен |
| Одновременная коллизия UUID | Реализовано: прежняя online-карточка не меняется, более новая получает бессрочный quarantine overlay до решения родителя |
| Ограниченное чтение UPnP `LOCATION` | Реализовано только в полном режиме: peer IP/LAN pinning, HTTP без DNS/redirect, 3 секунды, 64 КБ и не более 4 fetch за проход |
| WS-Discovery | Реализовано короткое пассивное окно и один bounded Probe только для нового устройства; `XAddrs` не открываются |

## Что не стоит переносить в стандартный IPK

- постоянный ARP/Nmap sweep всей подсети;
- публичный Fingerbank API без отдельного согласия;
- глубокий анализ всего трафика через Zeek/ntopng/nDPI на обычном роутере;
- базы или код GPL-проектов внутрь MIT-пакета;
- доверие имени, IP, OUI или открытому порту как личности устройства;
- автоматическое изменение trusted baseline вслед за текущим клиентом;
- блокировку только из-за исчезнувшего mDNS/SSDP-ответа.

## Принятые продуктовые решения

1. **Первый сильный UUID/serial.** Он принимается как исходный baseline по принципу TOFU
   (trust on first use). Повтор того же самообъявленного UUID во втором подключении почти не
   добавляет безопасности. Сам baseline не выдаёт права и не назначает `Без ограничений`.
2. **Автогруппа.** Для `Без ограничений` нужны не два подключения, а не менее двух
   независимых семейств классификационных доказательств и score не ниже 80. Один UUID,
   повторённый дважды, не превращается в два доказательства.
3. **Новый/случайный MAC.** Даже при совпавшем UUID Sheepfold только показывает, на какое
   прежнее устройство похожа запись. Родитель сам решает судьбу нового `#ID`; права не
   наследуются.
4. **Factory reset и прошивка.** Другой UUID не принимается молча. При прежнем MAC это
   подозрительное изменение и карантин; при новом MAC — новое устройство с подсказкой о
   возможном сходстве. Совпавшее имя разрешено показывать только как слабую подсказку.
5. **Видимость ограничения.** Во всех списках устройств слева от имени отображается зелёный
   identity-значок при устойчивом baseline или красный перечёркнутый при защите только по
   MAC.
6. **Карантин.** Он бессрочный до решения родителя или возвращения прежнего положительно
   совпавшего устройства. Нерешённое уведомление повторяется не чаще раза в сутки.
7. **Привилегии.** При подозрительной подмене приостанавливаются и белый список устройств,
   и `Без ограничений`. Исходные права не удаляются.
8. **Автогруппа стала ошибочной.** Если автоматически доверенное инфраструктурное устройство
   уверенно стало персональным, снимается только автоматическое назначение `Без ограничений`.
   Ручную группу Sheepfold не трогает.
9. **DHCP fingerprint.** Option 55/60 собираются локально без packet capture и являются
   признаками типа, а не личности.
10. **Чёрный список устройств и offline.** Для них активное определение не выполняется.
    Чёрный список устройств сохраняет только online/offline и журнал запретов доступа.
11. **Fingerbank.** Публичный API и тяжёлая локальная база не входят в стандартный IPK.
    Sheepfold использует независимо реализованную идею нескольких кандидатов и штрафа за
    конкурирующие признаки.
12. **Одновременный UUID.** При одном UUID у двух подтверждённо online-MAC прежняя карточка с
    меньшим `#ID` не меняется, а более новая получает бессрочный карантин до решения родителя.
13. **UPnP и WS-Discovery.** Оба источника разрешены только в полном режиме с жёсткими лимитами;
    URL `XAddrs` не открываются, а `LOCATION` может указывать только на ответивший LAN-peer.
14. **Объяснение результата.** В обычном интерфейсе и уведомлении о новом устройстве показываются
    IP и противоречащие признаки. Положительные внутренние баллы остаются диагностикой backend.
15. **SNMP.** Не реализуем его ни как зависимость, ни как фоновый модуль стандартного продукта.

## Термины, которые легко перепутать

### Админский ключ и доверенный сетевой отпечаток

Одноразовый код из QR нужен только для первого сопряжения. После него родительский APK
получает отдельный секретный Bearer-токен и предъявляет его каждой административной API-
команде. Подделав MAC телефона, ребёнок не получает этот токен и не сможет командовать API.

Но firewall принимает решения об обычном трафике по сетевому подключению, где Bearer-токена
нет. Без identity baseline поддельный MAC мог бы временно получить интернет-привилегии
белого списка устройств. Поэтому админский токен и сетевой отпечаток защищают разные слои.

### DHCP Option 55 и 60

- Option 55 — упорядоченный список настроек, которые DHCP-клиент просит у роутера. Разные
  сетевые стеки Android, Windows, Linux и простых IoT часто формируют его по-разному.
- Option 60 — строка класса производителя/клиента, если устройство её отправляет.

Оба поля легко подделать, поэтому Sheepfold использует их для классификации и конкурирующих
гипотез, но не для подтверждения физической личности. Hotplug хранит нормализованный снимок
в `/tmp`; полный пакет DHCP не записывается.

### UPnP `LOCATION`

SSDP-ответ может содержать LAN-URL XML-описания устройства. Там часто есть производитель,
модель, serial и список сервисов, поэтому чтение может улучшить распознавание телевизоров,
колонок, камер и медиасерверов. Однако URL сообщает само недоверенное устройство. Без
ограничений роутер можно заставить обратиться к постороннему или локальному служебному
адресу (SSRF).

Безопасный режим реализован только для нового, неуверенного или ожидающего безопасную автогруппу
online-устройства в полном профиле. Адрес назначения обязан совпадать с числовым IPv4 отправителя
и присутствовать в LAN neighbour; IP роутера, DNS, HTTPS, userinfo и redirect запрещены. Connect
timeout равен 2 секундам, общий timeout 3 секундам, ответ ограничен 64 КБ. Парсер извлекает только
тип, имя, производителя, модель, serial и UDN, не раскрывает сущности и не вызывает управляющие
URL. Не более четырёх описаний читается за проход; результат конкретного LOCATION кэшируется в
RAM на сутки.

Это всё равно самоописание клиента, а не удостоверение. UPnP используется как вспомогательный
сигнал классификации и как один компонент многофакторного сопоставления; одного ответа UPnP
недостаточно для административных прав, белого списка устройств или `Без ограничений`.

### WS-Discovery и SNMP

WS-Discovery — локальный multicast-протокол Windows/DPWS. Его используют Windows-узлы,
принтеры, камеры и NVR; сообщения `Hello`/`ProbeMatch` дают тип сервиса и endpoint без
сканирования портов. В Sheepfold реализованы короткое пассивное окно на LAN и один Probe только
при событии нового устройства, с десятью poll-циклами по 200 мс и пределом 96 сообщений. У каждого
Probe новый `MessageID`, поэтому устройства не принимают разные подключения за повтор одного
запроса. Endpoint UUID хешируется; `Types` и
`Scopes` помогают классификации; недоверенные `XAddrs` не открываются.

SNMP читает сведения управляемого сетевого оборудования, NAS и некоторых принтеров. Обычно
он выключен или требует community/учётные данные. По решению владельца проекта SNMP не
реализуем: выигрыш для семейного сценария не оправдывает дополнительный scanner и настройку.

### Google- и Яндекс-аккаунты

Пассивный роутерный трафик не даёт надёжного имени аккаунта. HTTPS/TLS и QUIC скрывают HTTP
заголовки, cookies, login/e-mail и содержимое; видимый домен означает только обращение к сервису.
Sheepfold не внедряет корневой сертификат, не выполняет MITM и не строит «личность» по догадке
о Google/Яндекс-аккаунте. Даже Android-приложение не должно перечислять чужие аккаунты без
разрешённого системой доступа. Если в будущем аккаунт понадобится как пользовательская функция,
его связывают только явным OAuth-согласием и не используют как доказательство LAN-устройства.

### Fingerbank API

Fingerbank может повысить полноту классификации большой общей базой, но запрос наружу может
раскрыть MAC/OUI, DHCP fingerprint, mDNS/UPnP и поведенческие признаки, а локальная база и
runtime слишком тяжелы для обычного роутера. Поэтому текущая рекомендация — локальная
классификация Sheepfold; добровольный Fingerbank допустим в будущем только через отдельный
модуль/внешний пользовательский worker с предварительным перечнем отправляемых полей.

## Принятые решения и оставшийся вопрос

- В каждой карточке хранятся три последних обезличенных снимка изменившихся evidence с датами,
  но без MAC, IP, имени, UUID и serial.
- В `tests/fixtures/deviceFingerprints.json` хранятся только обезличенные regression-примеры,
  вручную подтверждённые владельцем; реальные домашние идентификаторы туда не попадают.
- Похожий новый MAC остаётся отдельной карточкой до решения родителя. Будущая кнопка в блоке
  `Идентификация устройства` объединяет две записи полностью, сохраняет меньший постоянный `#ID`
  как основной, а второй как не переиспользуемый alias (§merge01).

Остаётся проверить на живых роутерах, дают ли HT/VHT/HE, supported rates и другие hostapd
capabilities достаточно пользы как слабый признак класса. Они не уникальны и не должны входить
в trusted identity либо самостоятельно влиять на права.
