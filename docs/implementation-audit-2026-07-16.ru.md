# Проверка реализованного функционала от 16 июля 2026 года

<!-- §implaudit -->

## Назначение

Этот документ отделяет подтверждённую реализацию от экранов-заготовок и проектных требований. Он не заменяет проверку на живом OpenWrt-роутере.

## Автоматически проверено

- полный набор Node/Python/shell-контрактов: `197/197` тестов прошли; IPK-зависимые тесты получили заранее собранные Standard/AI Support архивы `0.1.0-185`, чтобы Windows sandbox не подменял результат ошибкой вложенного запуска Python `EPERM`;
- родительский Android-код компилируется задачей `:app:compileDebugKotlin`;
- оба Android-проекта ранее собирались полностью, XML-ресурсы разбираются;
- shell-синтаксис install/init/hotplug/CGI/libexec проверен;
- runtime hardening, состав Standard/AI Support IPK и права исполняемых файлов проверены;
- `git diff --check` не нашёл ошибок пробелов; остаются только предупреждения о будущей нормализации окончаний строк.

## Исправлено во время проверки

1. **Локализация с кавычками.** Старый PO-to-JSON parser обрезал русские строки на экранированной кавычке. Парсер переведён на разбор PO-литералов, нечёткие записи пропускаются, каталог пересобран и проверен на оборванные значения.
2. **Атомарность сопряжения.** Админские права больше не подтверждаются до надёжного сохранения связанного Bearer-токена. При ошибке временные изменения откатываются, одноразовый код не создаёт частично привязанное устройство.
3. **Некорректное глобальное действие.** Backend принимает только явные значения включения и выключения глобальной блокировки. Неизвестное значение больше не трактуется как команда включить интернет.
4. **Смена порта после сопряжения.** Родительский APK при явной недоступности старого endpoint читает подписанный `/.well-known/sheepfold.json`, проверяет закреплённый TLS-сертификат, сохраняет новый порт и повторяет запрос. Неопределённый сетевой тайм-аут не вызывает автоматическое повторение команды.
5. **Применение списков доступа.** После сохранения белого/чёрного списка устройств и назначения админского устройства LuCI сразу синхронизирует nftables, не ожидая очередного фонового цикла.
6. **Временный доступ при глобальной блокировке.** Техническое добавление MAC для `+30 минут` больше не превращает устройство в постоянное nftables-исключение. Глобальное отключение действительно сильнее временного разрешения, как показывают LuCI и `/client-status`.
7. **Постоянное исключение не понижается до временного.** Backend отклоняет `+N минут` для админских устройств, белого списка устройств и группы «Без ограничений». Команда из API или бота больше не может заменить постоянный доступ временным статусом.
8. **Временные разрешения отделены от белого списка устройств.** Новый `+N минут` хранит только срок и временный статус. Фоновая очистка убирает технические allowlist-записи старых версий, поэтому родитель может заблокировать устройство немедленно, не ожидая окончания таймера.
9. **Постоянное решение отменяет таймер полностью.** При последующей блокировке, добавлении в белый список устройств или сохранении нового постоянного статуса backend удаляет срок и служебные поля временного доступа. Техническая запись старой схемы очищается, но явный выбор пользователя между белым и чёрным списками по-прежнему не переносится автоматически.
10. **WPS-белый список применяется сразу.** По завершении 30-секундного окна новые разрешённые MAC не только сохраняются в UCI, но и немедленно синхронизируются с nftables.
11. **Ограничение интернета отделено от чёрного списка устройств.** Расписания, обычный статус «Ограничено» и политика ограничения новых устройств используют отдельный forward-only nftables-набор. Только чёрный список устройств закрывает также LuCI, SSH и Sheepfold API.
12. **Политика новых устройств стала рабочей.** При значении `restrict` новый зарегистрированный MAC блокируется для выхода в интернет, `/client-status` сообщает ту же причину, а после DHCP-обнаружения firewall синхронизируется сразу.
13. **Сохранение правил не ждёт фонового цикла.** Изменение политики новых устройств, карточки устройства или состава группы запускает синхронизацию доступа после успешного UCI apply.
14. **Ложное редактирование приоритета отключено.** LuCI больше не позволяет сохранить порядок, который firewall не умеет исполнять. Показывается фактическая фиксированная цепочка, чёрный список устройств стоит выше автоматических назначений, а прежнее заводское значение `access_priority` мигрирует без перезаписи неизвестных пользовательских вариантов.
15. **Импорт и полный экспорт стали рабочими.** Новый формат сохраняет все Sheepfold-секции, постоянные DHCP-аренды и Wi-Fi. Читаемый JSON маскирует секреты, полный вариант шифруется AES-256-GCM, импорт проверяет конфликт списков и выполняет UCI apply с попыткой отката. Перезапуск сетевых служб отделён от ответа LuCI.
16. **Аварийно-полезные сайты перестали быть браузерной заглушкой.** Карточки сохраняются как UCI-секции общей кнопкой настроек. Backend наполняет собственные IPv4/IPv6 nftables-наборы через dnsmasq nftset либо резервное разрешение базового домена и открывает только web-порты, не снимая запрет LuCI/SSH/API для чёрного списка устройств (§emerg1).
17. **Загрузчик внешних списков больше не стирает защиту при сбое.** Источники проверяются независимо и с лимитами, отдельные плохие строки пропускаются, полностью плохой файл оставляет последний рабочий кэш, повтор назначается через сутки, а после трёх неудачных циклов создаётся дедуплицированное уведомление для администраторов. Сетевые попытки ограничены по времени; превышение числа источников или общего бюджета доменов также сохраняет прежний итог. Резкое сокращение крупного источника более чем на 75% требует отдельного принятия, после восстановления ранее проблемного источника приходит одно уведомление (§slstres).
18. **Standard IPK больше не запускает AI-задания.** Два задания ротации журнала активности создаются только в AI Support, где соответствующий backend действительно присутствует.
19. **Тестовый IPK не полагается на случайные пакеты роутера.** Его зависимости снова совпадают с основным Makefile и явно включают `uhttpd-mod-tls` и `curl` для HTTPS и сетевых интеграций.
20. **Updater больше не превращает сетевой сбой в успех.** Код загрузчика считывается внутри ветки ошибки, поэтому `uclient-fetch` с кодом `7` завершает проверку тем же кодом. Загрузка релиза и IPK ограничена 120 секундами и 64 МиБ; Telegram long poll ограничен 35 секундами и имеет `curl`/`wget` fallback.
21. **Повреждённый или чужой IPK не передаётся в `opkg`.** Updater принимает asset только из официального stable-релиза нужного варианта Sheepfold, проверяет структуру обоих tar-архивов, безопасные пути, имя пакета, версию и архитектуру. При ошибке установки или неподтверждённой установленной версии прежний UCI-конфиг восстанавливается. Это не является бинарным откатом пакета (§updsafe).
22. **Итоговые списки сайтов подключены к runtime.** В режимах `none`/`podkop` проверенный dnsmasq `nftset`-конфиг и собственные nftables-наборы Sheepfold переключаются транзакционно. В режимах `adguard`/`adguard_podkop` локальная DNS-политика снимается и явно делегируется AdGuard Home. Ошибка теста DNS, его перезапуска или синхронизации firewall восстанавливает прежнюю рабочую пару; до первой рабочей загрузки строгая группа не блокируется пустым списком (§dompol).
23. **Очистка доменных nftables-наборов стала атомарной.** Четыре IP-набора сайтов и MAC/интерфейсная политика теперь обновляются одним `nft -f` batch. Ошибка не может оставить очищенной только часть наборов. Потеря служебного хеша после успешной nft-транзакции не запускает опасный откат DNS: следующий sync просто повторит безопасную операцию (§dompol).
24. **Standard hardening проверяет всю цепочку уведомлений доменной политики.** Тестовый пакет требует не только `sheepfold-domain-policy`, но и исполняемый `sheepfold-admin-notification`; архивный тест читает фактический hardening script из собранного IPK.
25. **Выключение аварийного доступа для чёрного списка устройств теперь соблюдается при глобальной блокировке.** Индивидуальные аварийные MAC остаются выше запрета, но общая аварийная оговорка глобального режима перенесена ниже чёрного списка устройств. Поэтому устройство из чёрного списка получает аварийно-полезные сайты только при включённой соответствующей настройке (§emerg1).

## Подтверждено только контрактами, нужен живой роутер

- установка и обновление IPK на разных актуальных версиях OpenWrt;
- реальная блокировка nftables, запрет LuCI/SSH/API для чёрного списка устройств и восстановление после `fw4 reload`;
- переходы расписаний по времени, ночные интервалы и конфликтующие правила;
- совместная работа четырёх профилей: без интеграций, только AdGuard Home, только Podkop, AdGuard Home + Podkop;
- QR-сопряжение с реальным DHCP/neighbor состоянием телефона;
- смена порта `5201` на другой с уже сопряжённым физическим Android-устройством;
- WPS, LED и Wi-Fi automation на роутерах с разными кнопками/LED/sysfs-драйверами;
- blocked-page responder для браузеров и корректное поведение небраузерных клиентов.
- обычный и зашифрованный импорт на том же и на чистом роутере, включая сохранение секретов, откат, переподключение Wi-Fi и повторное сопряжение APK;
- аварийно-полезные сайты с обычным dnsmasq и dnsmasq nftset, AdGuard Home, Podkop, IPv4/IPv6, дополнительными доменами сайта и общими CDN-адресами;
- белые и чёрные списки сайтов во всех четырёх режимах, включая отсутствие `nftset`, восстановление после firewall reload/reboot, общий CDN и документированные обходы через DoH/VPN/прямой IP (§dompol);

## Честно незавершённые функции

- LuCI показывает фактически применяемый безопасный порядок правил доступа только для чтения; перестановка отключена, пока firewall, status API и расписания не научатся исполнять один произвольный порядок;
- автоматическая настройка AdGuard Home/Podkop не реализована;
- доменная политика списков сайтов реализована локально, но ещё не подтверждена живым роутером; она не анализирует URL/контент и не заявляет защиту от внешнего DNS, DoH, VPN, proxy или прямого IP (§slstres, §dompol, §uirunfx);
- Telegram остаётся ранним адаптером, VK/MAX не реализованы;
- в родительском APK рабочими являются сопряжение, основное управление, устройства, запросы, информация, feedback, защита и виджеты; расписания, группы, администраторы, Wi-Fi и журналы пока показывают экраны-заготовки вместо полных редакторов LuCI;
- долговременная архитектура сознания/памяти ИИ остаётся документацией будущей разработки и не входит в текущую проверку Standard IPK.

## Следующий практический проход

1. Установить IPK `0.1.0-185` на тестовый роутер с сохранением консольного вывода.
2. Пройти матрицу из [`live-router-testing.ru.md`](live-router-testing.ru.md), начиная с чёрного списка устройств, глобальной блокировки и расписаний.
3. Проверить физическим телефоном сопряжение, смену порта и восстановление endpoint.
4. Проверить аварийно-полезные сайты и доменную политику по матрицам из `docs/emergency-sites-runtime.ru.md` и `docs/site-domain-policy.ru.md`; после этого выбирать следующую реализацию, не объявляя единый настраиваемый runtime приоритета готовым только по виду LuCI.
