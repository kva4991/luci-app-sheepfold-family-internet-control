# Фоновые задания обслуживания Sheepfold

<!-- §maintjob1 -->

Документ фиксирует исполняемый контракт `sheepfold-maintenance`. Эти задания обслуживают уже сохранённые настройки, но не получают права автоматически менять семейную политику или устанавливать обновления.

## Запуск

`sheepfold-service` не чаще одного раза в пять минут вызывает:

```sh
/usr/libexec/sheepfold/sheepfold-maintenance run
```

Сам helper проверяет отдельный срок каждого задания и использует общий kernel `flock`. Повторный service tick не запускает параллельный cleanup, update probe или rotation.

Доступны диагностические команды:

```sh
sheepfold-maintenance force
sheepfold-maintenance rotate-log
sheepfold-maintenance cleanup-devices
sheepfold-maintenance check-updates
sheepfold-maintenance status
```

`force` игнорирует только расписание запуска. Он не ослабляет валидацию, не включает автоматическую установку пакета и не удаляет защищённые карточки.

## Ротация основного RAM-журнала

Используются:

- `sheepfold.global.log_retention`, например `3d`;
- `sheepfold.global.log_max_size_kb`, по умолчанию `1024`;
- безопасный путь `log_cache_path` только внутри `/tmp/sheepfold/`.

Журнал и rotator используют один `log-maintenance.flock`. Запись ограничивает одно сообщение 4096 символами. Rotation:

1. сохраняет строки за выбранное число локальных календарных дней;
2. сохраняет malformed/legacy-строки вместо их молчаливого удаления;
3. обрабатывает только ограниченный хвост `максимальный размер + 64 КиБ`, поэтому повреждённый производитель событий не заставляет сервис читать неограниченный файл;
4. затем оставляет самые новые полные строки в пределах размера;
5. заменяет файл атомарно и сохраняет режим `0600`.

Если год роутера явно неверен, age-pruning пропускается, но ограничение размера продолжает работать. Это предотвращает удаление журнала из-за часов `1970`.

USB и облачные архивы не считаются основным RAM-журналом. Их отдельная ротация не расширяется скрытно этим заданием.

## Очистка старых offline-карточек

Используется `offline_device_retention_days`; проход выполняется не чаще раза в сутки и только при правдоподобном времени роутера.

Карточка удаляется только если одновременно выполнены все условия:

- presence-снимок содержит подтверждённый `last_seen`, который старше лимита;
- устройство сейчас offline;
- статус карточки всё ещё пустой или `new`; старые решения `allow/blocked/restricted/scheduled` сохраняются даже до появления нового маркера `user_configured`;
- оно не является администраторским;
- MAC отсутствует в allowlist и blocklist;
- нет постоянной DHCP-аренды;
- нет расписания, группы, activity journal или активного временного доступа;
- нет ручного типа, `user_configured`, пользовательского исключения из автогруппы;
- нет legacy/merge alias или identity-quarantine состояния;
- нет сохранённой SIM-истории или истории новых Wi-Fi-сетей детского устройства.

Родительское сохранение карточки и Android-редактор групп ставят `user_configured=1`. За один проход просматривается не больше 512 и удаляется не больше 32 карточек, поэтому повреждённый большой конфиг не создаёт долгую блокировку сервиса. Удаление выполняется через изолированный `uci -t/-p`, снимок `/etc/config/sheepfold`, единый commit и проверку main UCI. При ошибке снимок восстанавливается. Счётчик `next_device_id` не уменьшается, поэтому удалённый постоянный `#ID` никогда не выдаётся повторно (§deviceid2).

## Периодическая проверка обновлений

`update_check_install_mode` задаёт только период проверки:

- `daily`;
- `weekly`;
- `monthly`;
- `never`.

Helper вызывает только:

```sh
sheepfold-updater probe
```

`probe` повторно использует проверку официального stable-release и сравнение версий updater, но не скачивает package asset и не вызывает `opkg`/`apk`. При новой версии создаётся дедуплицированное административное уведомление. Установка остаётся отдельным подтверждённым действием в LuCI или родительском APK (§updsafe).

Persistent state хранится в `/etc/sheepfold/maintenance/` с режимом `0600`; там нет URL с токенами, ключей или семейной конфигурации. Сетевой сбой записывается как `failed`, но не выдаётся за отсутствие обновления.

## Доказательства и границы

Локальные тесты подтверждают:

- календарную и size-ротацию временного журнала;
- сохранение групповых, админских, DHCP- и schedule-карточек;
- удаление только одной подходящей ненастроенной карточки;
- вызов `probe`, уведомление и отсутствие install/start-команд;
- подключение helper к service loop и общий log lock.

Они не доказывают длительную работу на конкретном роутере, поведение при реальном заполнении RAM, GitHub outage, package-manager runtime или последствия аварийного отключения питания. Перед stable-релизом нужен длительный live-router проход с backup/restore и проверкой отсутствия pending UCI (§routerharness).
