# Тест-кейсы Sheepfold

## Категории тестов

1. **Backend / роутер** — автоматические и ручные тесты API и логики на OpenWRT
2. **Android (родительский APK)** — UI, репозиторий, сетевой слой
3. **Детский APK** — статус, уведомления, обработка ошибок
4. **Интеграционные** — взаимодействие APK ↔ роутер в реальных условиях

---

## Backend: определение MAC по IP

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| B-01 | IP клиента есть в ARP/neighbor-таблице | MAC найден, нормализован в верхний регистр, возвращён статус устройства |
| B-02 | IP клиента есть только среди Wi-Fi клиентов (hostapd) | MAC найден через ubus/hostapd, статус возвращён корректно |
| B-03 | IP клиента отсутствует и в ARP, и в hostapd | `ok: false`, код ошибки `device_unknown`, понятное сообщение |
| B-04 | `REMOTE_ADDR` пустой или null | Безопасный возврат ошибки, роутер не падает |
| B-05 | MAC найден, но устройство не в списке Sheepfold | Статус `New`, `internetState` по умолчанию |

## Backend: расчёт статуса устройства

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| B-10 | Устройство в Blocked | `internetState: disabled`, независимо от расписания и глобального состояния |
| B-11 | Устройство в Allow, глобальный интернет выключен | Определяется продуктовым решением; результат должен быть явно зафиксирован и не меняться незаметно |
| B-12 | Устройство в Scheduled, сейчас в разрешённом окне | `internetState: enabled`, `accessMode: scheduled`, `accessEndsAt` = конец окна |
| B-13 | Устройство в Scheduled, сейчас вне окна | `internetState: disabled` |
| B-14 | Активное временное разрешение `+30m` | `internetState: enabled`, `accessMode: temporary`, `accessEndsAt` = startTime + 30m |
| B-15 | Временное разрешение истекло | `internetState` по базовому статусу устройства |
| B-16 | `until_bedtime`, время отбоя не задано | Безопасное поведение: либо fallback на конец дня, либо ошибка с понятным сообщением |

## Backend: endpoint /client-status

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| B-20 | Корректный запрос, MAC найден, статус рассчитан | `ok: true`, полный JSON с `deviceName`, `internetState`, `accessMode`, `accessEndsAt`, `minutesRemaining`, `serverTime`, `message` |
| B-21 | MAC не найден | `ok: false`, `error.code: device_unknown`, понятное сообщение на русском |
| B-22 | Запрос с параметром `?mac=...` в URL | Параметр игнорируется, MAC определяется только по IP |
| B-23 | `accessEndsAt` рассчитан корректно по расписанию | `minutesRemaining` соответствует разнице `accessEndsAt − serverTime` |

## Backend: временные разрешения

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| B-30 | POST device/action с `duration: 30m` | Временное разрешение записано, `accessEndsAt = now + 30m` |
| B-31 | POST device/action с `duration: end_of_day` | `accessEndsAt = 23:59:59` сегодня по локальному времени роутера |
| B-32 | Два активных временных разрешения для одного MAC | Берётся то, которое заканчивается позже (или более строгое — решение фиксируется явно) |
| B-33 | Временное разрешение истекает во время запроса `/client-status` | Возвращается актуальное состояние без expired-разрешения |

## Android: родительский APK

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| A-01 | Первый запуск, роутер не привязан | Отображается экран привязки (pairing) |
| A-02 | Тема по умолчанию | Применяется `ThemeMode.SYSTEM` |
| A-03 | Смена темы на `DARK` в настройках | Тема применяется немедленно, значение сохраняется в `ThemePreferenceStore` |
| A-04 | После перезапуска приложения | Сохранённая тема восстанавливается |
| A-05 | Нажатие «+30 мин» для устройства | POST `/device/action` с `actionType: temporary_grant`, `duration: 30m`; после успеха — список устройств обновляется |
| A-06 | Глобальное отключение интернета | POST `/internet/state`, UI отображает новое состояние |
| A-07 | Роутер недоступен (нет Wi-Fi или роутер выключен) | Показывается честное сообщение об ошибке, не выдуманный статус |
| A-08 | Ответ API содержит `ok: false` | Отображается `error.message` из ответа |

## Детский APK

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| C-01 | Запуск, роутер в сети, устройство в Allow | Показан статус «Интернет включён», таймер не отображается |
| C-02 | Запуск, устройство в Blocked | Показан статус «Интернет отключён», причина: родительский контроль |
| C-03 | Активное временное разрешение на 30 мин, осталось 27 мин | Показан таймер «Осталось 27 мин», уведомление запланировано на `accessEndsAt − 5 мин` |
| C-04 | До окончания доступа осталось менее 5 минут | Уведомление не ставится (поздно), состояние отображается корректно |
| C-05 | `accessEndsAt` изменился (новое +30 мин от родителя) | Предыдущее уведомление отменяется, планируется новое |
| C-06 | `accessEndsAt = null` | Таймер не отображается, уведомление не ставится |
| C-07 | Нет подключения к Wi-Fi / роутер недоступен | Отображается «Нет связи с роутером», а не выдуманный статус доступа |
| C-08 | Устройство не распознано роутером (`device_unknown`) | Отображается «Роутер не смог определить ваше устройство. Проверьте подключение к домашнему Wi-Fi.» |
| C-09 | Кнопка «Обновить» | Повторный запрос к `/client-status`, перепланирование уведомления |
| C-10 | Уведомление за 5 минут до конца | Уведомление показывается в нужное время, текст понятен ребёнку |

## Интеграционные тесты

| # | Сценарий | Ожидаемый результат |
|---|----------|--------------------|
| I-01 | Родитель блокирует устройство → детский APK обновляется | После обновления детский APK показывает `Blocked` |
| I-02 | Родитель выдаёт `+30 мин` → детский APK получает `accessEndsAt` | Таймер и уведомление появляются корректно |
| I-03 | Wi-Fi клиент (не проводной) — MAC определяется через hostapd | Статус возвращается корректно, не `unknown` |
| I-04 | Смена IP устройства (переподключение к Wi-Fi) | Новый IP правильно сопоставляется с тем же MAC |
| I-05 | Два разных устройства с одинаковым displayName | Статусы не перемешиваются, идентификация идёт по MAC |
