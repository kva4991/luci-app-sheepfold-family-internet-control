# План расширения управления AdGuard Home

<!-- §aghplan -->

Этот документ фиксирует, какие данные и настройки Sheepfold может безопасно передавать в AdGuard Home после уже реализованного URL-фильтра, а также порядок следующих проходов ИИ-разработчика. Это план, а не описание готовых функций.

## Короткое решение

Сейчас `adguard_auto_manage=1` означает только одно: Sheepfold может создать, обновить, включить или выключить **свой** URL-фильтр `Sheepfold family site policy` (§dompol).

Нельзя после обновления незаметно расширить смысл уже включённой настройки и начать менять глобальную защиту, DNS-серверы, DHCP, журналы или пользовательские правила AdGuard Home. Для каждого нового вида записи потребуется отдельное понятное действие или настройка, предпросмотр изменений и явное подтверждение владельца.

Большую часть следующего этапа следует строить не как передачу новых настроек, а как **чтение и проверку фактического состояния** AdGuard Home.

## Что Sheepfold уже передаёт

- адрес собственной token-protected ленты;
- имя, URL и состояние только собственного фильтра;
- сгенерированные доменные правила, ограниченные текущими IPv4-адресами нужных устройств;
- команду обновления URL-фильтров после изменения ленты.

Sheepfold не редактирует `AdGuardHome.yaml`, не заменяет custom rules через `filtering/set_rules`, не меняет пользовательские URL-фильтры и не переносит в AdGuard Home правила белого или чёрного списка **устройств**.

## Что стоит добавить в первую очередь

### 1. Расширенная диагностика без изменения AdGuard Home

Sheepfold может безопасно читать и показывать:

| Проверка | Официальный API | Что показывать пользователю |
| --- | --- | --- |
| Версия, запуск DNS-сервера, DNS-порт, глобальная защита | `GET /control/status` | `AdGuard Home работает`, версия, защита включена/приостановлена |
| Фильтрация, интервал и URL-фильтры | `GET /control/filtering/status` | найден ли фильтр Sheepfold, включён ли он, когда обновлён, сколько правил принято |
| DNS-конфигурация | `GET /control/dns_info` | проходит ли ожидаемая DNS-цепочка через AdGuard Home; без вывода чувствительных значений в журнал |
| Проверка конкретного правила | `GET /control/filtering/check_host` | действительно ли правило Sheepfold срабатывает для указанного IP клиента |
| Настройки журнала запросов | `GET /control/querylog/config` | журнал включён/выключен, срок, маскирование IP; содержимое запросов не читать |
| Настройки статистики | `GET /control/stats/config` | включена ли статистика и как долго она хранится |
| Safe Browsing, parental и Safe Search | соответствующие `*/status` | только фактический статус и ссылка на объяснение |
| Встроенный DHCP | `GET /control/dhcp/status` | кто раздаёт DHCP и можно ли вообще использовать MAC как идентификатор клиента |
| HTTPS/DNS encryption | `GET /control/tls/status` | настроено ли защищённое соединение, особенно для AdGuard Home на другом устройстве |

Эти данные нужны для честной статус-плашки. Одного ответа `HTTP 200` недостаточно. Интерфейс должен различать состояние самого API/фильтра и состояние реальной DNS-цепочки: `фильтр готов, путь клиентов не проверен` не равен полному зелёному статусу.

### 2. Контрольное правило, подтверждающее реальную работу

В собственную ленту можно добавить безвредные контрольные правила на зарезервированных тестовых доменах и после обновления проверить их через `filtering/check_host` с реальным IP устройства. Так Sheepfold отличит случаи:

- фильтр существует, но глобальная защита AdGuard Home выключена;
- фильтр скачан, но правило не принято;
- правило принято, но не совпадает идентификатор клиента;
- фильтр и клиентское правило действительно работают.

Контрольные домены не должны вести на реальный сайт. Точный набор правил сначала защищается unit-тестом, затем проверяется на живом роутере.

`filtering/check_host` проверяет решение движка AdGuard Home, но сам по себе не доказывает, что DNS-запросы телефона или другого LAN-клиента действительно доходят до AdGuard Home. Для сквозного зелёного статуса дополнительно нужна проверка DNS-пути; если она невозможна без тестового клиента, LuCI должен честно оставить отдельное состояние `путь не проверен`.

### 3. Понятные предложения исправления

Если Sheepfold обнаружил проблему, интерфейс должен показать причину и отдельное действие:

- `Включить защиту AdGuard Home`;
- `Включить DNS-фильтрацию`;
- `Восстановить фильтр Sheepfold`;
- `Повторить обновление фильтра`;
- `Проверить DNS-цепочку`.

Включение глобальной защиты через `POST /control/protection` и изменение глобальной фильтрации через `POST /control/filtering/config` нельзя выполнять молча. Кнопка должна показать текущее и новое значение, предупредить, что включатся и пользовательские фильтры AdGuard Home, дождаться подтверждения, применить изменение и перечитать состояние.

### 4. Очистка DNS-кэша только при доказанной необходимости

API предоставляет `POST /control/cache_clear`. Не следует вызывать его при каждом сохранении: это общий кэш AdGuard Home, а не объект Sheepfold. Сначала живой тест должен доказать, что после обновления собственного фильтра старый DNS-ответ продолжает обходить новое правило.

Если необходимость подтвердится, очистка разрешается только после успешного обновления фильтра Sheepfold, с ограничением частоты и записью понятного события в журнал. Ошибка очистки кэша не должна удалять рабочий фильтр.

### 5. Уведомления о деградации и восстановлении

Добавить отдельные дедуплицированные события:

- API AdGuard Home недоступен;
- авторизация отклонена;
- глобальная защита или фильтрация выключена;
- принадлежащий Sheepfold фильтр исчез или отключён;
- фильтр не может скачать локальную ленту;
- правило не прошло контрольную проверку;
- включён встроенный fallback Sheepfold;
- AdGuard Home снова работает и управляемый фильтр подтверждён.

Пароль, Basic Auth header, private feed token, полный URL с токеном и содержимое DNS-журнала в события не попадают.

## Что можно рассмотреть позже, но не включать автоматически

### Имена и постоянные клиенты

AdGuard Home умеет хранить постоянных клиентов и индивидуальные настройки. Теоретически Sheepfold мог бы передавать понятное имя устройства и его постоянный IP, чтобы статистика AdGuard Home была удобнее.

Пока это не следует включать по умолчанию:

- у объекта клиента нет надёжного отдельного поля владельца `Sheepfold`;
- имя может совпасть с уже созданным пользователем клиентом;
- динамический IP способен перейти другому устройству;
- MAC принимается как идентификатор только когда DHCP раздаёт сам AdGuard Home;
- изменение клиента может затронуть его пользовательские upstream, Safe Search, blocked services и настройки журналирования.

Допустим только будущий эксперимент с отдельным согласием: синхронизировать устройства с постоянной DHCP-арендой OpenWrt, предварительно показывать список, хранить точный реестр созданных Sheepfold объектов и никогда не обновлять найденного по одному имени клиента. До решения вопроса владения использовать текущие `$client=<IPv4>` правила.

### Safe Browsing, parental protection, Safe Search и blocked services

Эти функции полезны, но являются отдельным выбором семейной политики и могут затронуть все устройства. На первом этапе Sheepfold читает и объясняет их состояние. Позже можно сделать мастер одноразовой настройки с отдельным подтверждением каждого пункта.

Нельзя автоматически выводить эти значения из возраста ребёнка, названия группы или включения белого списка сайтов. Нельзя выключать функцию лишь потому, что пользователь выключил управление Sheepfold: она могла быть настроена владельцем прямо в AdGuard Home.

### DNS-цепочка с Podkop

В режиме `adguard_podkop` Sheepfold может:

1. прочитать текущие upstream DNS;
2. определить локальный адрес/порт, который рекомендует установленная версия Podkop;
3. проверить предлагаемую цепочку через `POST /control/test_upstream_dns` без сохранения;
4. показать точный diff и инструкцию;
5. только после отдельного подтверждения передать полную новую DNS-конфигурацию;
6. перечитать её и проверить DNS до фиксации успеха.

Этот мастер нельзя запускать одним фактом обнаружения Podkop. Ошибка в upstream способна отключить DNS всему дому. До реализации надёжного снимка и отката Sheepfold должен ограничиться диагностикой и ссылкой на официальную инструкцию Podkop.

### IPv6-клиенты

Текущая управляемая лента опирается на наблюдаемые IPv4. Следующий backend-этап должен научиться надёжно связывать IPv6-адрес с устройством, учитывать временные privacy-адреса и обновлять правила без бесконечного роста ленты. До этого LuCI обязан честно писать `проверено только для IPv4`, а не показывать полный зелёный статус для IPv6-сети.

## Что Sheepfold не должен передавать автоматически

| Настройка AdGuard Home | Почему нельзя менять автоматически |
| --- | --- |
| DHCP-сервер, диапазоны и аренды | конфликт с OpenWrt DHCP может оставить дом без сети |
| Общие upstream, bootstrap, fallback DNS | ошибка отключает DNS или ломает цепочку Podkop |
| Access lists клиентов | они закрывают только DNS и не заменяют чёрный список устройств Sheepfold, который блокирует интернет, LuCI, SSH и API |
| Custom filtering rules | нет безопасной границы владения; можно стереть правила пользователя |
| Пользовательские URL-фильтры и интервалы | глобальные изменения затрагивают чужие источники |
| DNS rewrites | могут подменить локальные сервисы и реальные домены |
| Query log, статистика и сроки хранения | это решение о приватности и памяти устройства |
| Содержимое query log | Standard IPK не собирает историю посещений; AI Support требует отдельного согласия и privacy-контракта |
| TLS-сертификаты и bind-адреса | ошибка способна закрыть доступ к панели и защищённому DNS |
| Обновление самого AdGuard Home | Sheepfold не владеет его бинарным релизом и откатом |

## Общий контракт любой будущей записи

Каждое действие за пределами собственного фильтра обязано пройти одну и ту же последовательность:

1. Проверить версию и наличие нужного API.
2. Прочитать текущее значение непосредственно перед изменением.
3. Показать пользователю точный diff простым языком.
4. Получить отдельное подтверждение; сохранённое ранее `adguard_auto_manage=1` не подходит.
5. Сохранить только тот предыдущий фрагмент состояния, который API действительно позволяет восстановить.
6. Отправить один минимальный запрос без секретов в командной строке и журнале.
7. Перечитать состояние и проверить фактический результат.
8. При ошибке восстановить предыдущий фрагмент, если откат доказуем; иначе остановиться и дать понятную инструкцию.
9. Не менять соседние поля, даже если API принимает большой объект целиком.
10. Записать в журнал кто подтвердил действие, что изменилось и чем закончилась проверка.

Если API не позволяет надёжно отличить объект Sheepfold от пользовательского или прочитать значение для отката, автоматическая запись запрещена.

## План следующих проходов ИИ-разработчика

Каждый проход завершается тестами, обновлением `docs/current-implementation-status.md` и записью реально проверенного результата. Не объединять несколько опасных проходов в одну большую правку.

### Проход 1. Контракт API и тестовый сервер

**Статус на 17 июля 2026 года: локально завершён.** Адаптер имеет явный allowlist из `filtering/status`, `filtering/add_url`, `filtering/set_url` и `filtering/refresh`; ограничивает время и размер ответа; удаляет старый response-файл до запроса; различает authentication, unsupported endpoint, throttling, rejected request, server error, timeout, invalid JSON/schema и чрезмерный ответ. Fake HTTP server подтверждает Basic Auth, полный цикл add/refresh/disable и точную форму JSON. Полный набор проекта прошёл `208/208` с IPK `0.1.0-188`. Живой роутер по-прежнему обязателен.

- вынести разрешённые read/write endpoints в явный allowlist адаптера;
- добавить fake AdGuard Home HTTP server для поведенческих тестов вместо одной проверки строк shell-скрипта;
- проверить Basic Auth, тайм-аут, `401`, `403`, `404`, `429`, `500`, невалидный JSON и несовместимую схему;
- доказать тестом, что адаптер не вызывает `filtering/set_rules`, `access/set`, `dns_config`, `clients/*`, `dhcp/*`, `rewrite/*` и TLS write endpoints;
- сохранить нынешний встроенный fallback.

### Проход 2. Диагностика и честный статус

**Статус на 17 июля 2026 года: локально завершён.** Адаптер читает и проверяет схему `/control/status`, `/control/dns_info` и `/control/filtering/status`. В status-файле сохраняются только безопасные агрегаты: запуск DNS, глобальная защита, версия, порт, количество DNS-адресов и количество upstream/fallback/bootstrap адресов. Сами адреса, логин, пароль и feed token не сохраняются. Отсутствие необязательного `dns_info` не отключает подтверждённый фильтр, но отображается как отдельная причина. Остановленный DNS-сервер или выключенная защита не считаются рабочим AdGuard backend и включают встроенный fallback; при этом adapter всё ещё может отключить только собственный фильтр, чтобы cleanup не создавал ложную ошибку. LuCI показывает жёлтый промежуточный статус `фильтр подтверждён, DNS-путь клиентов не проверен`, а не ложную зелёную плашку. Fake HTTP server подтвердил все эти состояния; полный набор проекта прошёл `212/212` с IPK `0.1.0-189`. Живой роутер по-прежнему обязателен.

- прочитать `/status`, `/filtering/status` и минимально необходимый `/dns_info`;
- сформировать единый machine-readable статус с кодом причины и временем проверки;
- разделить `API доступен`, `защита выключена`, `фильтрация выключена`, `фильтр отсутствует`, `лента не скачана`, `правило не подтверждено`, `fallback активен`;
- отдельно хранить результат проверки движка и результат проверки реального DNS-пути;
- показать эти состояния в LuCI без вывода логина, пароля и feed token.

### Проход 3. Сквозная контрольная проверка

**Статус на 17 июля 2026 года: локально завершён.** В ленту добавлены безвредные правила для зарезервированных доменов `.test`. После обновления адаптер вызывает `GET /control/filtering/check_host` для обычной блокировки, приоритетного аварийного разрешения и, при наличии строгой группы, одного её текущего IPv4-клиента. Успех требует точного совпадения текста правила и ID принадлежащего Sheepfold фильтра. Повреждённый ответ, чужой ID или неподтверждённое правило отклоняются; при первой неудачной установке новый фильтр выключается с обязательным перечитыванием состояния, а при обновлении восстанавливается прежняя лента. Число проверок ограничено двумя либо тремя. Проверка движка не меняет глобальные настройки AdGuard Home и не выдаётся за проверку DNS-пути LAN-клиента. Fake HTTP server подтверждает положительные и отрицательные сценарии; полный набор проекта прошёл `216/216` с двумя IPK `0.1.0-190`. Живой роутер по-прежнему обязателен.

- спроектировать безопасные тестовые домены и правила;
- после refresh вызвать `filtering/check_host` для представителя каждого применяемого класса политики;
- не считать весь backend рабочим, если хотя бы обязательный клиент не подтверждён;
- ограничить число запросов и повторов, чтобы не нагружать роутер.

### Проход 4. Устойчивость и уведомления

**Статус на 17 июля 2026 года: локально завершён.** Sheepfold проверяет не только кэшированный результат `check_host`, но и доступность локальной token-protected ленты через `uhttpd`; скачанное содержимое должно точно совпасть с активной лентой. Каждые пять минут фоновая служба повторяет применение только при выбранной цепочке AdGuard Home и реально включённой политике сайтов. После трёх последовательных ошибок создаётся одно уведомление администраторам, а после первого успешного восстановления — одно уведомление о восстановлении. Состояние серии хранится в RAM и не изнашивает flash. Удалённый фильтр создаётся заново, случайно переименованный возвращает каноническое имя, а несколько фильтров с одним точным URL считаются конфликтом и не выбираются произвольно. При смене токена прежний подтверждённо принадлежащий URL сначала выключается с перечитыванием API; сам URL с токеном хранится только в файле `0600` и не попадает в status, журнал или уведомления. Локальные тесты покрывают недоступный `uhttpd`, смену токена, дубликаты, переименование, три ошибки и восстановление; полный набор проекта прошёл `224/224` с двумя IPK `0.1.0-191`. Проверка перезапусков и DHCP/IP на живом роутере всё ещё обязательна.

- хранить последнюю подтверждённую причину ошибки и время;
- уведомлять после заданного числа последовательных ошибок, без спама;
- отправлять одно уведомление о восстановлении;
- проверить удаление/переименование/дублирование фильтра, смену токена, перезапуск uhttpd и AdGuard Home;
- проверить смену IPv4 после DHCP lease event.

## Зафиксированный остаток после локального прохода 4

Следующий разработчик не должен продолжать расширять AdGuard-адаптер только потому, что интеграция ещё не названа окончательно завершённой. Текущий локальный контракт собственного фильтра достаточен для переключения на другие функции Sheepfold. Остаток разделён по месту проверки:

### Обязательно проверить на живом роутере

- перезапуск `uhttpd`, AdGuard Home, `dnsmasq`, `firewall` и самого Sheepfold во время активной политики;
- удаление, отключение и переименование управляемого фильтра через настоящий интерфейс AdGuard Home;
- смену feed token и отсутствие старого URL в работающих фильтрах, журналах и списке процессов;
- смену IPv4 устройства после DHCP-события и удаление устаревшего адреса из правил;
- четыре схемы: без интеграций, только AdGuard Home, только Podkop, AdGuard Home вместе с Podkop;
- Basic Auth, неверный пароль, выключенную глобальную защиту, недоступный API и восстановление после каждой ошибки;
- сохранность пользовательских фильтров, custom rules, клиентов, rewrites, upstream DNS и настроек журналирования;
- реальный DNS-запрос LAN-клиента: зелёный статус движка `check_host` пока не доказывает, что клиент использует именно AdGuard Home;
- нагрузку, RAM, время обновления и поведение на роутере класса Xiaomi AX3000T.

### Следующие безопасные локальные доработки

- вывести отдельный read-only статус глобальной защиты, DNS-фильтрации и фактического DNS-пути без чтения query log;
- подготовить действия `Восстановить фильтр Sheepfold` и `Повторить проверку`, которые не меняют чужие объекты;
- не считать IPv6 покрытым, пока адреса с privacy rotation не связываются с устройством надёжно и с ограниченным сроком жизни;
- отклонять повреждённый локальный реестр владения URL и давать понятную инструкцию, не подбирая фильтр по одному имени.

### Только после отдельного проектирования и подтверждения пользователя

- включение глобальной защиты или DNS-фильтрации AdGuard Home;
- очистка общего DNS-кэша;
- изменение upstream DNS для цепочки с Podkop;
- синхронизация persistent clients;
- изменение Safe Browsing, Safe Search, parental protection, blocked services, DHCP, query log, статистики или TLS.

Эти действия не входят в смысл существующего `adguard_auto_manage=1`. До живых проверок и отдельного безопасного отката они остаются планом, а разработка продолжается в других частях Standard IPK.

### Проход 5. Явные действия исправления

- добавить отдельные кнопки для включения protection/filtering и восстановления только собственного фильтра;
- показать diff и влияние на пользовательские фильтры;
- повторно запросить статус после записи;
- реализовать откат только для реально прочитанного состояния;
- не добавлять общий режим `полностью управлять AdGuard Home`.

### Проход 6. Проверка кэша и DNS-цепочки

- на живом роутере измерить поведение до и после обновления фильтра;
- решить по доказательствам, нужен ли `cache_clear`;
- диагностировать, получают ли LAN-клиенты DNS именно от AdGuard Home;
- проверить четыре режима: без интеграций, только AdGuard Home, только Podkop, оба вместе.

### Проход 7. Необязательные клиенты и Podkop-мастер

- отдельно исследовать владение persistent clients и коллизии имён;
- если безопасной модели нет, официально отказаться от записи clients и оставить только диагностику;
- сделать read-only preview DNS-цепочки Podkop и тест предложенных upstream;
- запись upstream оставить выключенной до проверенного отката на физическом роутере.

### Проход 8. IPv6

- собрать реальные lease/neighbor-сценарии с privacy IPv6;
- ограничить срок и число адресов одного устройства;
- проверить смену адреса, отсутствие адреса и двойной стек;
- только после этого убрать предупреждение об IPv4-only подтверждении.

### Проход 9. Живой роутер и выпуск

- проверить локальный AdGuard Home без авторизации и с Basic Auth;
- проверить удалённый AdGuard Home только по HTTPS;
- сохранить пользовательские filters, custom rules, clients, rewrites, Safe Search, query log и upstream без изменений;
- проверить остановку сервиса, выключенную protection/filtering, плохой пароль, тайм-аут, перезагрузку и обновление IPK;
- измерить RAM, flash, время refresh и нагрузку на роутер уровня Xiaomi AX3000T;
- выпускать расширение только после записи матрицы результатов в `docs/live-router-testing.ru.md`.

## Критерии готовности

- пользовательские объекты AdGuard Home до и после обычной работы Sheepfold совпадают;
- текущий переключатель автоматического управления по-прежнему разрешает только собственный фильтр;
- статус показывает фактический исполнитель и fallback, а не выбранное значение формы;
- ошибка API не отключает проверенный встроенный backend;
- секреты отсутствуют в process list, LuCI status, syslog, уведомлениях и обычном экспорте;
- все новые write endpoints имеют отдельный тест подтверждения, перечитывания результата и отката;
- физический роутер подтверждает работу во всех четырёх топологиях.
