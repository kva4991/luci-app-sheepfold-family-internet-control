## Проверка документации

```powershell
npm.cmd run quality:docs
npm.cmd run quality:docs:all
```

Первая команда проверяет Markdown, изменённый относительно `origin/main`. Вторая проходит `AGENTS.md`, `CODING_RULES.md` и всё дерево `docs/`.

Проверяется:

- существование относительных ссылок на файлы и каталоги;
- регистрация конкретных §-тегов в `docs/dev/tag-map.md`;
- корректная обработка ссылок с anchor/query и путей в угловых скобках.

Не проверяется:

- доступность внешних HTTP/HTTPS-сайтов;
- истинность документа;
- существование конкретного Markdown-anchor внутри существующего файла;
- ссылки, формируемые программно.

Внешние URL намеренно исключены, чтобы тест не зависел от блокировок, DNS, GitHub rate limit и доступности иностранных доменов. Для цитируемого внешнего стандарта URL проверяется вручную при содержательном обновлении документа.

Коды:

- `BROKEN_LINK`: относительная цель отсутствует;
- `UNKNOWN_TAG`: конкретный §-тег не зарегистрирован.

Шаблонный `§xxxxxxx` разрешён только как пример. Прочие реальные теги надо сначала внести в карту с назначением и списком файлов.

## Структурный аудит

```powershell
npm.cmd run quality:structure
```

Инструмент берёт только изменённые исходники, сравнивает число строк с версией из `origin/main` и применяет порог по типу файла. Он показывает проблему роста, а не пытается автоматически дробить код.

Статусы:

- `ok`: файл ниже advisory-порога;
- `warning`: новый или выросший файл выше порога;
- `notice`: старый крупный файл затронут, но не вырос;
- `improved`: крупный файл стал меньше.

Размер сам по себе не является багом. Перед разделением нужно назвать обязанности, общий state и стабильные интерфейсы. Нельзя превращать один большой файл в десяток взаимозависимых обёрток без уменьшения сложности.

Порог меняется в `tools/quality/structureAudit.mjs`. Его повышают только после измерения устойчивого ложного шума, а не для сокрытия одного неудобного файла. Для OpenWrt helper без расширения действует отдельный порог, потому что shell-команды часто поставляются единым исполняемым файлом.

Текущий рефакторинг служит примером: `frontendSmoke.mjs` оставлен runner-оболочкой, а проверяемые действия страницы перенесены в `frontendAudit.mjs`. Ответственности стали различимы, и статический контракт читает оба файла.

## All-pairs матрица

```powershell
npm.cmd run quality:matrix
npm.cmd run quality:matrix -- --json
```

Модель находится в `runtimeCompatibilityModel.mjs`. Генератор покрывает каждую допустимую пару значений факторов и сохраняет обязательные строки. Текущая модель выбирает 15 сценариев вместо 384, покрывая 122 из 122 допустимых пар.

`requiredRuntimeRows` хранит не случайные примеры, а критические состояния:

- Standard без интеграций;
- AI Support с AdGuard Home и Podkop;
- встроенная фильтрация с Podkop;
- ручное управление установленным AdGuard Home;
- явно выбранный отсутствующий AdGuard Home.

Оптимизированный генератор один раз рассчитывает pair tokens каждой строки. Это уменьшает повторную работу на расширенной модели и не меняет детерминированный порядок.

Матрица не заменяет:

- security-инварианты и повреждённый ввод;
- migration и rollback;
- реальную установку IPK/OpenWrt APK;
- DNS/nftables на живом роутере;
- различия драйверов Wi-Fi и hardware-кнопок.

При добавлении фактора надо добавить минимум два значения, обновить ожидаемую runtime-модель и решить, нужен ли новый обязательный сценарий. Тест обязан подтвердить ноль `uncoveredPairs` и стабильность результата.

## Read-only LuCI-аудит

```powershell
$env:SHEEPFOLD_SCRIPT_SCRATCH_ROOT='C:\Users\User\Documents\pesochnica'
npm.cmd run router:frontend
```

Требования к SSH/DPAPI-профилю и тестовому роутеру описаны в [`../live-router-automation.ru.md`](../live-router-automation.ru.md). Runner устанавливает закреплённый `playwright-core` в игнорируемый `tools/local/`, использует системный Chromium/Chrome и не кладёт браузер в IPK.

`frontendSmoke.mjs` отвечает за окружение, два viewport и JSON. `frontendAudit.mjs` отвечает за вход и проверки. Viewport запускаются последовательно, чтобы не перегружать маломощный роутер двумя одновременными LuCI-сессиями.

Для быстрой проверки ещё не собранного CSS можно внедрить локальный файл только в изолированную browser-сессию:

```powershell
$env:SHEEPFOLD_FRONTEND_STYLE_PATH = (Resolve-Path 'package\luci-app-sheepfold-family-internet-control\htdocs\luci-static\resources\sheepfold\features\devices\responsive.css')
npm.cmd run router:frontend
Remove-Item Env:SHEEPFOLD_FRONTEND_STYLE_PATH
```

Режим не загружает файл на роутер, не меняет UCI и не доказывает упаковку asset в OpenWrt-пакет. Он нужен для короткого цикла responsive CSS; окончательное подтверждение требует SDK-пакета с поднятым `PKG_RELEASE` и повторного `router:frontend` уже без переменной.

Фатальные коды:

| Код | Что означает |
| --- | --- |
| `zeroSize` | видимый интерактивный элемент фактически не имеет площади |
| `missingName` | кнопка, ссылка, input, select или textarea не имеет доступного имени |
| `outsideViewport` | элемент вышел за горизонтальную границу viewport |

Предупреждения:

| Код | Что означает |
| --- | --- |
| `targetBelow24` | хотя бы одна сторона цели меньше 24 CSS px |
| `targetBelow32` | цель меньше проектного комфортного размера 32 CSS px |
| `clippedLabel` | текст интерактивного элемента обрезан скрытым overflow |

Предупреждения не валят тест автоматически, потому что существующий LuCI содержит компактные таблицы. Они сохраняются структурированно в `result.json` как `uxWarnings.byCode`; новый агент должен сравнить характер предупреждений, а не только общее число.

Дополнительно проверяются:

- отсутствие `RPCError`, `TypeError`, `[object HTML...]` и invalid constructor;
- чистые console/page/request errors после авторизации;
- переключение верхних и вторичных вкладок;
- наличие модели и версии прошивки вместо `unknown`;
- состав панели журнала;
- отсутствие горизонтального overflow;
- русский или английский набор основных табов.

Аудит не нажимает «Сохранить», интернет, Wi-Fi, WPS, reboot, delete и другие команды приложения. Поэтому зелёный `router:frontend` доказывает только read-only UI-контракт. Сохранение и runtime проверяются `router:writeSafe/fullSafe` с backup/restore.

Артефакты находятся в каталоге запуска внутри `SHEEPFOLD_SCRIPT_SCRATCH_ROOT`:

- `desktop.png`;
- `mobile.png`;
- `result.json`;
- `*-failure.png`, если viewport завершился ошибкой.

При ошибке сначала открыть JSON и соответствующий screenshot, затем проверить console/request. Не исправлять CSS наугад по одному общему числу предупреждений.
