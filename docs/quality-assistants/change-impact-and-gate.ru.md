# Карта влияния и единый quality gate

<!-- §qassist -->

## Зачем нужны два уровня

`review:impact` только анализирует. `quality:changed` использует тот же отчёт и выполняет безопасную локальную часть. Такое разделение позволяет сначала проверить план, не запуская долгие команды, а затем воспроизвести тот же выбор автоматически.

## Откуда берутся пути

По умолчанию база равна `origin/main`:

```powershell
npm.cmd run review:impact
node scripts/inspectChangeImpact.mjs --git origin/main
```

Поддерживаются три режима:

```powershell
node scripts/inspectChangeImpact.mjs --git main
node scripts/inspectChangeImpact.mjs --staged
node scripts/inspectChangeImpact.mjs --working-tree
```

- `--git <base>` включает все изменения ветки относительно базы, незакоммиченные изменения и untracked-файлы.
- `--staged` смотрит только индекс Git.
- `--working-tree` смотрит незакоммиченные tracked и untracked-файлы.
- Проверка `git diff --check` следует тому же режиму: база, только индекс или только рабочее дерево. Untracked-файлы во всех подходящих режимах дополнительно проверяет `whitespaceAudit.mjs`.
- Явные пути можно передать аргументами для проверки гипотезы.
- `--json` печатает машиночитаемый отчёт.

`gitChanges.mjs` использует `--name-status -z --find-renames`. Нулевой разделитель нужен для пробелов и национальных символов в имени. При rename анализируются старый и новый путь: перенос backend-файла в неизвестный каталог не должен скрыть прежнюю security-границу.

## Как определяется риск

| Уровень | Смысл |
| --- | --- |
| `low` | документация или локальный контракт без исполняемого поведения |
| `medium` | LuCI или тестовый helper с ограниченным радиусом |
| `high` | Android, backend, устройство, группа или административная граница |
| `critical` | общий API, UCI migration, firewall/DNS, package identity или updater |

Итоговый риск равен максимальному уровню совпавших правил. Неизвестный исходник получает `high`, неизвестный прочий путь получает `medium`. В строгом gate неизвестный путь является ошибкой: сначала надо решить, к какой области он относится, затем добавить точное правило и regression test.

Правила лежат только в `tools/quality/changeImpactRules.mjs`. Широкие совпадения по одному слову запрещены. Например, ADR с фразой `device-passport` не должен случайно запускать тесты детектора. Паттерн привязывается к реальному каталогу, исполняемому имени или конкретному профильному документу.

Есть одно узкое content-aware исключение: `gitChanges.mjs` помечает `Makefile` как `packageReleaseOnly`, только когда по сравнению с выбранной Git-поверхностью изменилась ровно одна строка `PKG_RELEASE:=...`, а всё остальное побайтно совпало. Тогда focused-запуск не раскрывает весь `security`, но package tests и обязательный полный gate сохраняются. Любая другая правка `Makefile`, ошибка чтения или явный путь без Git-контекста остаётся консервативной.

## Состав отчёта

- `changes`: status, новый путь и старый путь rename.
- `fileImpacts`: области, совпавшие для каждого файла.
- `areas`: агрегированные подсистемы.
- `categories`: категории Node-тестов.
- `directTests`: изменённые test-файлы, которые запускаются напрямую.
- `checks`: lint и внешние проверки.
- `reviews`: вопросы, которые нельзя надёжно автоматизировать.
- `unknown`: пути без правила.
- `risk`: максимальный риск.
- `fullTest`: требуется ли полный suite из-за общего контракта.

Изменённый test-файл не разворачивает все категории, в которые он входит. Он запускается напрямую, а категории определяются производственными путями. Это существенно сокращает обратную связь и не теряет проверку самого теста.

Категория `tooling` специально не содержит `openWrtVariantFeed`, `productVariants`, `testIpkI18n` и `testIpkPermissions`: они создают package-артефакты и остаются в `packaging`. Поэтому изменение локального анализатора не запускает дочерний Python и сборку IPK. Полный gate всё равно выполняет их перед push.

## Быстрый исполнитель

```powershell
npm.cmd run quality:changed
```

Последовательность:

1. Собрать Git-изменения и карту влияния.
2. Выполнить `git diff --check` относительно базы и отдельно проверить untracked-файлы на trailing whitespace и лишний/отсутствующий финальный перевод строки.
3. Проверить относительные ссылки и §-теги изменённых Markdown-файлов.
4. Запустить ESLint только для изменённых JS/MJS. Если изменён `eslint.config.js` или включён полный режим, проверить весь JS-контур.
5. При Android-изменении запустить Android Lint.
6. Объединить категории и прямые тесты без дублей и выполнить одним процессом.
7. Сохранить длительность, результат и невыполненные внешние команды в `.build/quality/last-run.json`.

Если `fullTest=true`, быстрый режим завершится как `partial`: это полезная промежуточная проверка, но не разрешение на push.

## Строгий gate

```powershell
npm.cmd run quality:gate
```

Он эквивалентен `--full --strict`:

- ESLint проверяет весь настроенный JS-контур;
- Node запускает каждый `tests/*.test.mjs` ровно один раз;
- неизвестные карте пути запрещены;
- Android Lint запускается, если затронут Android или общий API-контракт;
- ручные live/build проверки остаются в `pendingManualChecks`.

Флаг `--skip-android` существует для локальной диагностики среды. Результат становится `partial`; перед push Android-правки пропуск надо закрыть отдельным успешным `npm.cmd run lint:android`.

## Что не запускается автоматически

Живой роутер, сборка APK и GitHub SDK не стартуют из общего gate. Причины:

- живой router harness имеет backup/restore и отдельные уровни риска;
- Android build может скачивать Gradle/SDK и занимает существенно больше времени;
- GitHub workflow публикует артефакты и требует авторизованную внешнюю сессию;
- автоматический запуск таких операций по одному совпадению пути был бы неожиданным и потенциально опасным.

Команды выводятся рядом с причиной. Агент обязан либо выполнить их, либо явно назвать в финальном отчёте, почему они не выполнены.

## Отчёт и диагностика

Последний отчёт находится в `.build/quality/last-run.json` и не коммитится. Полезные поля:

```json
{
  "status": "passed | partial | failed",
  "durationMs": 1234,
  "pendingFull": false,
  "pendingManualChecks": [],
  "steps": [
    { "id": "eslint", "status": "passed", "durationMs": 240 },
    { "id": "focusedTests", "status": "passed", "durationMs": 900 }
  ]
}
```

Если внешний runner прервал процесс, сначала посмотреть последний завершённый step и наличие дочернего процесса. Тайм-аут долгого теста не переписывать как дефект продукта без отдельного воспроизведения.

## Как добавить правило

1. Выбрать точную границу пути, не слово из произвольного имени документа.
2. Назначить минимальные категории и риск.
3. Добавить только те внешние checks, которые реально нужны этой области.
4. Сформулировать вопрос review как проверяемый риск.
5. Добавить позитивный тест и соседний негативный путь в `tests/changeImpact.test.mjs`.
6. Запустить `quality:plan` и убедиться, что unrelated ADR или test fixture не расширяют набор.
7. Обновить этот документ, если появился новый тип проверки.

## Где искать ошибку инструмента

- Потерян rename: `tools/quality/gitChanges.mjs`, тест `parseNameStatus`.
- Лишняя область: слишком широкий `pattern` в `changeImpactRules.mjs`.
- Пропущена область: новый каталог не описан либо путь нормализован иначе.
- Тест запускается дважды: `testSelection.mjs` должен дедуплицировать имя.
- Полный suite запустился после документа: проверить `full: true` и пересечение паттернов.
- JSON говорит `partial`: проверить `pendingFull`, Android и `pendingManualChecks`, а не повторять все команды наугад.
