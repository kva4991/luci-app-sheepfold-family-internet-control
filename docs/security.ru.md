# Безопасность

Этот документ фиксирует обязательные инварианты backend, LuCI и Android-клиентов Sheepfold.

## Сопряжение администраторского телефона

### Разделение доверенных и публичных операций

Создание учётной записи администратора и выпуск одноразового кода — разные операции.

- Учётная запись создаётся только внутри аутентифицированного LuCI.
- Secure LuCI не выдаёт права устройству вручную. Кнопка ручной привязки скрыта; новый администратор создаётся без устройств.
- LuCI вызывает `sheepfold-router-control activate-admin-pairing-code` локально.
- Команда обязана найти уже существующую UCI-секцию `administrator` по логину.
- Публичного HTTP endpoint `/pair-token` нет.
- Единственная публичная операция сопряжения — `POST /cgi-bin/sheepfold-api/pair`.
- Android отправляет `X-Sheepfold-Client: android-admin-v1`.

### Последовательность

```text
Аутентифицированный LuCI
        |
        | существующий administrator + локальный выпуск кода
        v
Одноразовый code + expires_at
        |
        | QR: адрес, порт, login, code
        v
Android в LAN
        |
        | POST /pair + native marker
        v
REMOTE_ADDR -> MAC -> проверка blocklist
        |
        | проверка кода, срока и одноразовости
        | повторная проверка blocklist перед записью
        v
admin_device + allowlist + стабильный device_id
        |
        | код удаляется только после успешной записи
        v
Одноразовая выдача Bearer
        |
        | Android Keystore хранит Bearer
        | роутер хранит только SHA-256 hash
        v
Последующие запросы Authorization: Bearer
```

### Обязательные инварианты

1. Blocklisted-устройство не может стать административным.
2. Сопряжение никогда не удаляет MAC из blocklist автоматически.
3. Blocklist проверяется после определения MAC и непосредственно перед первой записью прав.
4. Код ограничен по времени, одноразовый и удаляется после успешного завершения.
5. Неуспешная попытка не выдаёт Bearer и не изменяет права.
6. Plaintext Bearer возвращается только в успешном ответе `/pair`.
7. Постоянный Bearer не принимается в query string.
8. Android хранит Bearer через Android Keystore; старое открытое значение мигрируется и удаляется.
9. Неопределённый MAC приводит к отказу.
10. Попытки `/pair` ограничиваются по фактическому LAN-устройству.
11. `deviceId` создаётся и возвращается сервером, а не выбирается клиентом.
12. Ручная привязка устройств в secure LuCI отключена; права выдаются через QR-поток.
13. Файл токена в `/etc/sheepfold/tokens/` хранит `login`, `device_id`, `mac`, `issued_at`, `expires_at` (plaintext Bearer на диске не пишется).
14. Admin API принимает Bearer только вместе с `X-Sheepfold-Device-Id` и `X-Sheepfold-Device-Mac`, совпадающими с записью токена и парным admin-устройством в UCI.

## ИИ-помощник

### Проверенная идентичность

Каждый AI-запрос содержит:

- `deviceId`;
- `clientRole`;
- `isAdministrator`.

Эти поля не считаются доверенными. Router-side gate повторно выполняет:

```text
REMOTE_ADDR -> MAC -> UCI device -> server deviceId/admin_device
```

Запрос отклоняется, если клиентские поля не совпадают с серверными.

Административный запрос дополнительно требует действительный Bearer и должен приходить с парного административного устройства.

### Детский режим

- роль обязана быть `child`;
- устройство не должно быть административным или находиться в blocklist;
- устройство должно состоять в личной группе; иначе вкладку можно показать, но отправка вопроса должна получить понятный отказ `personal_group_required`;
- требуется `child_ai_parental_consent=1`;
- версия согласия обязана совпадать с `child-ai-v1`;
- диагностика, журналы, аккаунт родителя, provider и model запрещены;
- используется отдельный детский system prompt;
- prompt не помогает обходить контроль или скрывать действия и направляет ребёнка к доверенному взрослому при серьёзном риске.

Доверенный child-safety flow пока не реализован. До отдельного threat model детский вопрос нельзя слепо пересылать родителю, добавлять в родительскую аналитику или обещать ребёнку абсолютную конфиденциальность. Родитель может быть источником угрозы; безопасный внутрисемейный получатель определяется будущим safety-протоколом, а не одной ролью `parent`. Sheepfold никогда не пересылает обращение государственным органам или экстренной службе: он только советует человеку обратиться самостоятельно (§aichild, §aigov0).

Внешний психолог/консультант не получает скрытый доступ к диалогу или памяти. Передача создаётся как отдельная подписанная задача после preview и согласия, проходит Privacy Proxy и содержит только выбранный минимум. Ответ консультанта считается рекомендацией, не применяет настройки и не становится фактом памяти автоматически (§aiescal, §aimask1).

Возможное правонарушение не создаёт скрытый канал в государственные структуры. Сетевой сигнал, категория сайта, один рассказ или совпадение выводов моделей не являются доказательством и не запускают сообщение полиции, органам опеки либо другому внешнему получателю. Будущий `legalRiskPolicy` различает непосредственную угрозу, тяжёлый правовой риск и небольшой ненасильственный проступок. В первом случае решение принимает safety-router, во втором предлагаются безопасный взрослый и квалифицированная местная юридическая помощь, в третьем — восстановительный разговор без унижения (§ailegal).

Sheepfold не помогает уничтожать доказательства, придумывать алиби, давить на свидетелей или скрывать продолжающийся тяжёлый вред. Этот запрет не разрешает передачу данных государству. Интеграции для автоматических звонков, заявлений, отчётов или отправки переписки государственным службам запрещены. Допускаются только проверенные справочные контакты и локальное открытие системного номеронабирателя/редактора сообщения без автоматической отправки (§ailegal, §aigov0).

Шаблон согласия находится в `docs/child-ai-parental-consent-template.ru.md`.

### Родительский режим

- используется отдельный родительский system prompt;
- диагностический контекст и журнал добавляются только после отдельных галочек;
- индивидуальный журнал требует `ai_individual_logs=1`;
- ключи провайдеров остаются на роутере.

### Маскирование AI-контекста

Полная страница информации остаётся доступной родителю в приложении. Внешнему AI передаётся отдельное сокращённое представление.

Из AI-контекста исключаются или обобщаются:

- модель роутера;
- firmware и OpenWrt release;
- kernel version;
- IP и MAC;
- serial/hostname;
- точные технические пути;
- секреты и токены.

Сохраняются только данные, способные объяснить ситуацию:

- доступность интернета и причина;
- качественная оценка ping;
- load и memory;
- число Wi-Fi-модулей;
- наличие Podkop и AdGuard Home.

### Внешние вычисления

- локальный Privacy Proxy работает до любого provider или worker, а не проверяет утечку постфактум (§aimask1);
- внешний worker получает только ограниченную по времени подписанную задачу с псевдонимами и минимальным payload;
- alias vault, реальные идентификаторы, provider keys, UCI и полные журналы остаются на роутере;
- соединение инициирует роутер по HTTPS; worker не получает входящий доступ к LAN, LuCI, SSH или Sheepfold API;
- ответ worker считается недоверенным, проверяется по schema/task ID/TTL и может создать только гипотезу либо кандидата подтверждения;
- отказ worker никогда не меняет правила доступа и не останавливает основные функции роутера (§aiexec1).

## Приватные архивы и USB

- Активный файл имеет формат `<timestamp>-<deviceId>-sheepfold.log`.
- Ротация выполняется под lock через staging-каталог.
- При ошибке архивирования staging возвращается, исходные записи не удаляются.
- Архивы именуются с epoch и не перезаписывают друг друга в один день.
- Нет безусловного удаления после семи дней; используется только `log_days`.
- Очистка выбирает только ожидаемые Sheepfold-файлы и не запускает `ls` без аргументов.
- Ключ выводится из двух случайных компонентов под общим lock первичной генерации.
- OpenSSL обязателен; plaintext под расширением `.enc` запрещён.
- Privacy notice копируется на уже смонтированный носитель из `/usr/share/sheepfold`.
- Автоматический extroot отключён до реализации копирования overlay, проверки и rollback.
- Swap проверяет свободное место, создаётся атомарно и восстанавливается при запуске Sheepfold.

## HTTPS и Android

Sheepfold поднимает отдельный uhttpd listener:

- HTTPS: единый порт приложения `5201` (Android parent/child и локальный Sheepfold API);
- Cleartext HTTP для Android API не используется.

Android admin (0.1.0-155+):

1. только HTTPS, `cleartextTrafficPermitted="false"`;
2. TLS pin (TOFU при первом успешном `/pair`, хранится в приложении);
3. Bearer-запросы с заголовками привязки устройства (см. инварианты 13–14).

Android child: только HTTPS, без admin Bearer.

API не должен открываться на WAN.

## Публичные маршруты

- `/pair-token` — всегда 404;
- `/pair` — публичный, но одноразовый и rate-limited;
- `/client-status` — доступен LAN-устройству и возвращает его собственный статус;
- `/router-info` — требует Bearer;
- `/pair-status` — требует Bearer;
- `/settings/save` legacy — отключён с `410 Gone`;
- Bearer в query string — отклоняется.

## Проверка изменений

Изменения pairing, токенов, AI, журналов, USB или транспорта должны сопровождаться:

- позитивными и негативными тестами;
- `sh -n` для всех helper-скриптов;
- Android debug build обоих приложений;
- проверкой cross-origin сценариев;
- повторным запуском runtime hardening;
- проверкой обновления без сброса ключевого материала;
- проверкой отсутствия секретов в логах, UCI, QR и notice;
- тестом HTTPS-only и проверкой закреплённого сертификата;
- тестом несовпадения deviceId/role/admin flag;
- тестом отказа детскому AI без актуального родительского согласия.
