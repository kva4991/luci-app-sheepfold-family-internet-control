# Безопасность

Этот документ фиксирует обязательные инварианты backend, LuCI и Android-клиентов Sheepfold. Он предназначен для разработчиков и проверяющих изменения безопасности.

## Сопряжение администраторского телефона

### Разделение доверенных и публичных операций

Создание учётной записи администратора и выпуск одноразового кода — разные операции с разными границами доверия.

- Учётная запись администратора создаётся и редактируется только внутри аутентифицированного LuCI.
- LuCI вызывает локальную команду `sheepfold-router-control activate-admin-pairing-code` через разрешённый RPC/exec-маршрут.
- Команда выпуска кода обязана найти уже существующую UCI-секцию типа `administrator` по логину. Если секции нет, операция завершается ошибкой и ничего не создаёт.
- Публичного HTTP endpoint `/pair-token` нет. Ни GET, ни POST не должны позволять LAN-клиенту создать администратора или выпустить код.
- Единственная публичная операция сопряжения — `POST /cgi-bin/sheepfold-api/pair`. Она принимает только уже выпущенные `login` и `code`.

### Последовательность

```text
Аутентифицированный LuCI
        |
        | 1. Администратор уже существует в UCI
        | 2. Локальный вызов activate-admin-pairing-code
        v
Одноразовый code + expires_at
        |
        | 3. QR содержит адрес, порт, login и code
        |    Постоянного Bearer-токена в QR нет
        v
Android в той же LAN
        |
        | 4. POST /pair
        v
Ограничитель попыток по MAC/IP
        |
        | 5. Проверка существования администратора
        | 6. Проверка кода, срока действия и одноразовости
        | 7. REMOTE_ADDR -> MAC через DHCP/ARP
        | 8. Проверка blocklist
        | 9. Повторная проверка blocklist непосредственно перед записью прав
        v
Запись admin_device + allowlist
        |
        | 10. MAC никогда не удаляется из blocklist автоматически
        | 11. Код удаляется после успешной записи
        v
Выдача Bearer-токена один раз
        |
        | 12. Android сохраняет Bearer, но не pairing-code
        | 13. На роутере хранится только хэш Bearer-токена
        v
Последующие административные запросы с Authorization: Bearer ...
```

### Обязательные инварианты

1. Устройство из `blocklist` не может стать администраторским ни при каких группах, расписаниях или наличии корректного кода.
2. Сопряжение не переносит устройство между чёрным и белым списками автоматически. Конфликт должен завершаться отказом.
3. Проверка `blocklist` выполняется дважды: после определения MAC и непосредственно перед первой записью административных прав. Это уменьшает окно между проверкой и использованием результата.
4. Код ограничен по времени, используется один раз и удаляется только после успешного завершения записи.
5. Неуспешная попытка не выдаёт Bearer-токен и не изменяет права устройства.
6. Plaintext Bearer-токен возвращается только в успешном ответе `/pair`. На роутере хранится хэш; Android хранит Bearer отдельно от одноразового кода.
7. Повторные попытки `/pair` ограничиваются на уровне роутера. Значения по умолчанию: 10 попыток за 600 секунд на разрешённый идентификатор устройства.
8. Неопределённый MAC не считается безопасным значением по умолчанию: сопряжение завершается ошибкой.
9. Нельзя восстанавливать `/pair-token` как «удобный» публичный endpoint. Это снова объединит создание полномочий и их использование.

### Негативные тесты

- Несуществующий логин передан в локальную команду выпуска кода: администратор не создаётся.
- Несуществующий логин передан в `/pair`: отказ без изменений UCI.
- Истёкший, неправильный или повторно использованный код: отказ без Bearer-токена.
- MAC отсутствует в DHCP/ARP: отказ.
- MAC находится в `blocklist`: отказ; `blocklist` остаётся неизменным.
- MAC попал в `blocklist` между первой проверкой и записью: повторная проверка блокирует операцию.
- Превышен лимит попыток: HTTP 429 и `Retry-After`.
- Успешное сопряжение: код удалён, устройство отмечено как администраторское, Bearer выдан один раз, повторное использование кода невозможно.

## ИИ-помощник

ИИ-помощник может быть доступен обычным устройствам LAN, но не является анонимным общим прокси к платному API.

- Помощник работает только при `sheepfold.global.ai_enabled=1` и настроенном ключе провайдера.
- Роутер определяет MAC по `REMOTE_ADDR`; клиент не выбирает идентификатор квоты самостоятельно.
- Неопределённые и находящиеся в `blocklist` устройства отклоняются.
- Квота ведётся отдельно для каждого MAC. Параметры: `ai_rate_limit_requests` и `ai_rate_limit_window_seconds`.
- Обычный клиент может передать только текст вопроса. Выбор провайдера/модели, диагностика, журнал и данные аккаунта считаются привилегированным контекстом и требуют действительного Bearer-токена администратора.
- Индивидуальные журналы дополнительно требуют `ai_individual_logs=1`.
- Перед включением индивидуальных журналов роутер обязан установить или найти OpenSSL, проверить реальное шифрование и инициализировать ключевой материал. При любой ошибке флаг возвращается в `0`.
- API-ключи провайдеров остаются на роутере и не передаются Android-клиентам.

## Приватные архивы и USB

- Ключ архива выводится из двух случайных 32-байтовых компонентов: `/etc/sheepfold/.kp` и `/etc/sheepfold/.app_salt`.
- Оба файла создаются атомарно с правами `0600`; каталог имеет права `0700`.
- Общая соль не встраивается в пакет, поэтому она не одинакова на разных роутерах и не меняется при обновлении.
- OpenSSL обязателен. MD5 и копирование plaintext под расширением `.enc` запрещены.
- Шифрование выполняется во временный файл. Только непустой успешно созданный ciphertext атомарно переименовывается в итоговый `.enc`.
- При ошибке исходный архив остаётся на месте, операция возвращает ошибку, а незашифрованный файл на USB не создаётся.

## Локальный HTTP и целевой HTTPS

Текущая локальная интеграция использует HTTP. Это временное ограничение совместимости, а не желаемая конечная модель. План миграции подробно описан в `docs/development-ideas-security.ru.md`.

До миграции запрещено:

- открывать API на WAN;
- считать сам факт нахождения в LAN аутентификацией администратора;
- передавать root-пароль, LuCI cookie, bot/API keys или постоянный Bearer внутри QR;
- включать глобальный Android cleartext без ограничения назначением и сетью;
- отключать проверку сертификата в будущем HTTPS-клиенте.

## Проверка изменений

Любое изменение pairing, токенов, AI gateway, журналов или USB-шифрования должно сопровождаться:

- позитивным тестом ожидаемого сценария;
- негативными тестами обхода полномочий;
- проверкой повторного запуска установочного hardening;
- проверкой обновления с предыдущей версии без сброса ключевого материала;
- проверкой отсутствия секретов в логах, UCI-выводе и QR-коде.
