## Главная идея продукта

Sheepfold Family Internet Control - self-hosted семейный инструмент для OpenWRT-роутера. Он управляет доступом домашних устройств в интернет через OpenWRT-роутер и его веб-интерфейс LuCI, а не через облако разработчика.

Основные части:

- OpenWRT-пакет `luci-app-sheepfold-family-internet-control`;
- LuCI-интерфейс `Sheepfold Family Internet Control` / `Sheepfold : контроль доступа в интернет для семьи`;
- backend-сервис на роутере;
- Android-приложение `Sheepfold` для телефонов родителей-администраторов;
- отдельное Android-приложение `Sheepfold Child` для ребёнка: без административных функций, только собственный статус доступа и опциональный ИИ-помощник после разрешения родителя;
- один активный мессенджер-бот на роутере: VK по умолчанию при настройке, Telegram как стабильная альтернатива, MAX только экспериментально;
- ИИ-помощник для родителей в Android-приложении, без автоматической передачи чувствительных данных.

## Названия и формулировки

- Основное имя проекта в EN/RU текстах: `Sheepfold`.
- Android-приложение в публичных текстах называть `Sheepfold`. Не писать `Овчарня`, если владелец проекта отдельно не просит обсудить старое/внутреннее имя.
- Всегда уточнять тип списка: `белый список устройств`, `чёрный список устройств`, `белый список сайтов`, `чёрный список сайтов` или `аварийно-полезные сайты`. Чёрный список устройств независимо от доменных исключений всегда закрывает LuCI, SSH и API роутера; чёрный список сайтов такого смысла не имеет (§usrcomm, §84azytj).
- Не писать в русском README "управлять через LuCI" без контекста. Пиши понятно: "управлять доступом домашних устройств в интернет через OpenWRT-роутер и его веб-интерфейс LuCI".
- Не переименовывать GitHub repo, OpenWRT package, Android package и LuCI display names без явного решения владельца проекта.

## Архитектурные инварианты

- Sheepfold стоит первым в цепочке принятия решения по устройству.
<!-- §v7x2k9p -->
- Целевая цепочка трафика: `Sheepfold -> AdGuard Home -> Podkop`.
- Sheepfold решает, можно ли устройству выходить в сеть.
- AdGuard Home фильтрует DNS, если включён.
- Podkop маршрутизирует разрешённый трафик, если включён.
- Проект не должен ломать стандартные OpenWRT firewall/DNS настройки, AdGuard Home, Podkop, Dnsmasq, sing-box и пользовательские конфиги.
- Автоматические изменения чужих конфигов допустимы только после явного подтверждения и желательно после backup/export.
<!-- §k5rf0hb -->
- Полноценное управление через Android и LuCI - только в локальной сети.
- Вне дома - только короткие подтверждаемые команды и уведомления через выбранный мессенджер.
<!-- §q3nj8wd -->
- Не добавлять developer-operated cloud backend для управления Sheepfold. Единственное согласованное исключение — необязательный канал отзывов `§feedback`: он не участвует в правилах доступа и передаёт только явно показанные пользователю поля.
- Не добавлять full remote management через WireGuard, VPN или другие туннели к роутеру.

## OpenWRT-пакет и backend

Цель - современные OpenWRT с `firewall4` / `nftables`.

Не реализовывать поддержку старых OpenWRT с `firewall3` / `iptables`, пока владелец проекта явно не попросит.

Backend должен:

- иметь узкий API/command layer, например `/usr/bin/sheepfold <method>` или эквивалентный безопасный слой;
- возвращать структурированный JSON для LuCI, Android и ботов;
- валидировать MAC, IP, hostname, домены, роли, имена групп и импортируемые данные;
- быть идемпотентным: повторный запуск не плодит дубли nftables/UCI/rpcd/service-записей;
- восстанавливать правила после `fw4 restart`, перезагрузки роутера и restart сервиса;
- не строить shell-команды из пользовательского ввода;
<!-- §b3nw8vp -->
- логировать административные действия без секретов.

Installer должен:

<!-- §n2lp7yt -->
- показать ссылку на пользовательское соглашение и требовать явное `yes`, `y` или `да`;
- проверить OpenWRT-окружение;
- проверить, задан ли root-пароль OpenWRT, или хотя бы обеспечить gate перед открытием Sheepfold в LuCI;
- обнаружить AdGuard Home и Podkop;
- выставить рекомендуемый `integration_mode`, но не переписывать чужие настройки без отдельного подтверждения;
- в режиме автонастройки настроить время роутера: выбрать подходящий timezone, включить NTP-клиент, прописать рекомендуемые NTP-серверы и сделать роутер NTP-сервером для LAN;
- не активировать мессенджер до ввода credentials и привязки администратора.

### Связность, DNS и время роутера

Проверки связности должны учитывать страну роутера. Для России нельзя делать единственным критерием доступность иностранных IP/доменов вроде `1.1.1.1`, `8.8.8.8`, `google.com`, Cloudflare или Google DNS: такие цели могут быть недоступны из-за блокировок, маршрутизации или политики провайдера.

Для российского профиля по умолчанию использовать максимально российские или релевантные России цели:

- DNS-запросы: `ya.ru`, `gosuslugi.ru`, `ntp1.vniiftri.ru`;
- HTTP(S)-проверки: лёгкие запросы к российским/локально релевантным доменам из country-profile;
- иностранные цели держать только как вторичный настраиваемый fallback, а не как обязательный признак живого интернета.

ICMP ping не должен быть единственным методом диагностики: часть нормальных сервисов не отвечает на ping. Правильная цепочка диагностики: статус WAN через `ubus`, линк/кабель, default route/gateway, DNS-запросы country-profile доменов, затем лёгкий HTTP(S)-check.

В настройках Sheepfold нужна группа настроек времени роутера:

- `делать роутер NTP-сервером` — по умолчанию включено в автонастройке;
- `автоматически настраивать NTP-клиент роутера` — по умолчанию включено в автонастройке;
- список NTP-серверов по умолчанию: `ntp1.vniiftri.ru`, `ntp2.ntp-servers.net`, `3.openwrt.pool.ntp.org`;
- `часовой пояс роутера` — выставлять на самый подходящий при установке/автонастройке.

Для России, если пользователь не выбрал регион точнее, использовать `Europe/Moscow` / `MSK-3`.

Реализация должна работать через системный OpenWRT config `system`:

- `system.@system[0].zonename`;
- `system.@system[0].timezone`;
- `system.ntp.enabled`;
- `system.ntp.enable_server`;
- `system.ntp.server`.

Так как это системные настройки OpenWRT, при экспорте/отчёте нужно по возможности показывать, что Sheepfold их менял, и сохранять старые значения там, где это практически реализуемо.

Uninstall должен:

- удалять пакет без очистки Sheepfold-настроек и списков клиентов;
- после удаления показывать отчёт об оставшихся настройках/следах на роутере, которые могут требовать ручного решения;
- не удалять пользовательские данные молча.

Update должен:

- ставить только stable-релизы;
- обновлять LuCI cache-busting version;
- не ломать существующий пользовательский config;
- при необходимости чистить LuCI index/module cache;
- после правок версионности LuCI-ассетов запускать `node --test tests/*.test.mjs` или `npm.cmd test` в Windows PowerShell, если обычный `npm test` заблокирован Execution Policy.

Тестирование с живым OpenWRT-роутером описано отдельно в `docs/live-router-testing.ru.md`. Такие тесты делятся на read-only, write, install/update, LuCI-browser и hardware-in-loop. Write/hardware-тесты нельзя запускать без явных флагов окружения и backup UCI-конфига.

## LuCI-интерфейс

LuCI - основной интерфейс настройки. Он должен быть понятен родителю, а не только сетевому администратору.

Обязательные вкладки:

- все устройства;
- белый список;
- чёрный список;
- расписания;
- доступ к аварийно-полезным сайтам;
- Wi-Fi;
- интеграции;
- администраторы;
- бот/мессенджер;
- журнал;
- настройки;
- диагностика.

Производственная LuCI-архитектура:

- Podkop-подобная структура как основной ориентир для LuCI-реализации;
- небольшой entrypoint;
- `form.Map("sheepfold")`;
- `tabbed = true`;
- отдельный JS/module слой на вкладку;
- общий store/API client;
- строгий `rpcd` ACL;
- backend-вызовы через узкий Sheepfold API;
- не разращивать один огромный `overview.js`.

Что именно брать из подхода Podkop:

- раздельную структуру LuCI frontend/backend вместо монолитной страницы;
- маленький LuCI entrypoint, который собирает вкладки, а не содержит всю бизнес-логику;
- отдельные модули для вкладок и диагностики;
- backend methods/commands, которые возвращают понятный JSON;
- строгий `rpcd` ACL только на нужные файлы, UCI-секции, ubus-объекты и команды;
- диагностическую вкладку с проверками состояния сервиса, firewall, dns/integration mode и версий;
- аккуратную работу с LuCI cache после обновлений;
- понятный путь установки/обновления из GitHub stable releases.

Что не копировать слепо из Podkop:

- не переносить routing/sing-box логику в Sheepfold;
- не менять Podkop-managed правила без отдельного документированного решения;
- не делать UI для сетевых специалистов в ущерб родительскому сценарию;
- не смешивать responsibility: Sheepfold управляет доступом устройств, Podkop управляет маршрутизацией, AdGuard Home управляет DNS-фильтрацией.

UI-правила:

- показывать одну понятную кнопку `Сохранить`;
- на странице `Настройки` не сохранять изменения автоматически при `blur`, `change`, переключении чекбокса/радиокнопки или вводе времени; все настройки должны применяться только после явного нажатия `Сохранить`;
- кнопку `Сохранить` на странице `Настройки` размещать сверху справа и повторять снизу после настроек;
- при несохранённых изменениях показывать понятную подсказку, что изменения ещё не применены;
- не заставлять родителя выбирать между `Сохранить` и `Применить`;
<!-- §m9qe4lk -->
- если LuCI-действие Sheepfold меняет UCI-конфиг сразу, например добавляет/удаляет устройство из белого или чёрного списка, привязывает устройство к администратору, меняет группу или сохраняет настройки, код обязан не только вызвать `uci.save`, но и принять/применить очередь изменений LuCI. Нельзя оставлять стандартную плашку LuCI `Не принятые изменения`: родитель уже нажал понятную кнопку Sheepfold и не должен искать ещё одну системную кнопку `Применить`;
- для этого использовать общий helper сохранения: сохранить нужные configs, затем применить через LuCI changes API (`ui.changes.apply(...)` или эквивалент конкретной версии LuCI), а если API недоступен - fallback на обычный OpenWRT `uci.apply`;
- не плодить отдельные варианты `uci.save()`/`uci.apply()` в разных вкладках: списки пользователей, администраторы, группы, настройки и быстрые действия должны идти через общий helper, чтобы поведение было одинаковым;
- опасные действия требуют подтверждения;
- все строки должны быть локализуемыми;
- русский - основной источник формулировок, английский - обязательный fallback;
- planned languages можно генерировать позже, но код не должен мешать локализации.
- для каждого неочевидного блока, настройки, статуса или опасного действия нужна кнопка справки `?` или эквивалентный help icon;
- справка должна открывать короткую инструкцию прямо в интерфейсе: что делает функция, кого затронет, что будет после нажатия, что может пойти не так и как отменить;
- длинные инструкции выносить в отдельные docs-файлы, а из UI давать ссылку или краткую версию;
- образец подхода: инструкция Telegram. Общее правило описано в `docs/contextual-help.ru.md`.

Опасные действия:

- перезагрузка роутера;
- обновление приложения;
- импорт настроек;
- экспорт полного backup с секретами;
- очистка журнала;
- массовая блокировка;
- смена Wi-Fi SSID/password/security/channel;
- смена активного мессенджера;
- добавление/удаление/понижение администратора;
- включение аварийно-полезных сайтов для чёрного списка;
- автоматическая настройка интеграции с AdGuard Home/Podkop.

Root password gate:

- при первом открытии Sheepfold в LuCI проверить root-пароль OpenWRT;
- если пароль не задан, не пускать в настройки Sheepfold;
- показать понятное предупреждение и кнопку/ссылку на стандартную страницу установки пароля OpenWRT;
- не создавать дефолтные Sheepfold-пароли.

LuCI cache: <!-- §assetv1 -->

- все Sheepfold JS/CSS/static assets используют один suffix `?v=<PKG_VERSION>-<PKG_RELEASE>`;
- source of truth - версия OpenWRT-пакета;
- не хардкодить отдельные версии в отдельных файлах;
- bump package release/version при изменении frontend-файлов.

## Устройства и правила доступа

Определять устройства по MAC-адресу. Искать можно по:

- MAC;
- текущему IP;
- hostname;
- пользовательскому имени.

Данные устройств брать автоматически из:

- DHCP leases;
- static DHCP leases;
- ARP/neighbor data;
- других безопасных OpenWRT-источников, если они есть.

Имена устройств и постоянные IP должны быть синхронизированы с OpenWRT DHCP static leases. Если пользователь меняет имя или постоянный IP в Sheepfold, это должно быть согласовано с разделом DHCP -> "постоянная аренда" на роутере.

Приоритет правил:

1. чёрный список;
2. белый список;
3. временный доступ;
4. расписание;
5. общие правила и политика новых устройств.

Жёсткие правила:

- один MAC не может быть одновременно в белом и чёрном списке;
- backend, LuCI, Android, bot commands и import validation должны это проверять;
<!-- §lstxcl1 -->
- Sheepfold не переносит устройство между белым и чёрным списком автоматически: операция отклоняется с понятным красным сообщением, а родитель сначала сам удаляет устройство из прежнего списка;
<!-- §w8dm3kv -->
- в стандартном порядке чёрный список сильнее временного доступа и расписаний;
<!-- §x1hy6pz -->
- белый список не блокируется глобальной блокировкой и расписаниями;
- глобальная кнопка `Выключить интернет` блокирует всех, кроме белого списка;
<!-- §h6mxq4c -->
- blocklisted devices не должны открывать LuCI, SSH и Sheepfold API;
- emergency-useful access для blocklist допустим только отдельной явной настройкой и всё равно не даёт доступ к LuCI/SSH/API.

Поведение новых устройств:

- настройка `поведение для новых устройств`;
- значения: `пускать в интернет` по умолчанию или `ограничивать до настройки`;
- неизвестные устройства должны быть видны в общем списке и доступны для настройки.

Группы устройств:

- дефолтная пользовательская группа `Ребёнок номер 1`;
- дети;
- родители;
- телевизоры/медиа;
- гости;
- без ограничений;
- пользовательские группы.

Удаление групп:

- если к группе привязано хотя бы одно устройство, группу удалять нельзя;
- кнопку удаления не делать визуально заблокированной: при нажатии показывать понятное всплывающее сообщение, что сначала нужно убрать устройства из группы;
- защищённые системные группы, например `Без ограничений`, удалять нельзя.

Группа `Без ограничений`:

- это не белый список конкретного устройства, а доверенная группа для домашней инфраструктуры: NAS, Home Assistant, AdGuard Home, Proxmox, сервер видеонаблюдения, Zigbee/Matter/Thread-хабы и похожие сервисные устройства;
- устройства этой группы не должны блокироваться расписаниями, временными ограничениями, правилом новых устройств и глобальной кнопкой выключения интернета;
- группа `Без ограничений` обходит глобальное отключение, расписания и политику новых устройств, но никогда не обходит чёрный список устройств; назначение в эту защищённую группу считается чувствительным действием и должно быть явно видно родителю;
- при установке нужно спросить `Применить автонастройку программы?`;
- полный режим автонастройки является режимом по умолчанию, потому что он полезен большинству семей;
- если родитель нажимает Enter или отвечает `yes`, `y` или `да`, включить `auto_configure=1`, `detection_mode=full`, `no_restrictions_auto_assign=1`;
- при включённой автонастройке полная установка может автоматически добавлять уверенно распознанные инфраструктурные устройства в эту группу;
- урезанная установка (`auto_configure=1`, `detection_mode=reduced`, `no_restrictions_auto_assign=1`) выбирается только явным ответом `no`, `n` или `нет` и нужна для роутеров с очень малым запасом места;
- урезанная установка не использует тяжёлые проверки портов, но может автоматически добавлять уверенно распознанные инфраструктурные устройства в эту группу.

Обновления:

- в `Общие` должна быть настройка `Проверка обновления и его установка`;
- значения: раз в день, раз в неделю, раз в месяц, никогда;
- значение по умолчанию: раз в неделю;
- использовать только stable-релизы;
- установка обновления должна требовать подтверждения.

Сильное автоопределение устройств:

- использовать не только MAC/hostname, а несколько router-side сигналов: DHCP/static lease, hostname, vendor/OUI, открытые порты, service banners, mDNS/SSDP/UPnP имена, ранее подтверждённый fingerprint;
- реализовывать сильное автоопределение отдельным backend-компонентом/пакетом, например `sheepfold-device-detector`, а не внутри LuCI-only пакета;
- LuCI должен вызывать Sheepfold backend API и оставаться лёгким интерфейсом;
- в полном режиме можно использовать существующие OpenWRT-инструменты, если они установлены или разрешены как зависимости полного режима: `nmap`/`nmap-ssl` для портов и баннеров, `avahi-utils` или аналоги для mDNS, SSDP/UPnP probes;
- эти инструменты не должны быть обязательными зависимостями урезанной установки;
- не запускать тяжёлые сканирования постоянно: определять переход offline -> online по текущим LAN ARP/neighbour, hostapd association или свежему DHCP hotplug, ждать 20 секунд, затем анализировать только всё ещё online-устройство (§detlife1);
- постоянная DHCP-аренда или старая строка lease сама по себе не является подключением;
- при запуске службы и раз в сутки делать страховочный проход только по подтверждённо online-устройствам; не трогать выключенные устройства;
- обрабатывать ограниченную очередь событий, по умолчанию не более четырёх за 10-секундный тик;
- `nmap` запускать после лёгкой проверки только для нового, изменившегося, неопределённого или явно выбранного устройства; стандартный таймаут 20 секунд на хост, повторный scan портов не чаще раза в сутки (§detload);
- не считать MAC, hostname или открытый порт криптографическим доказательством, потому что ребёнок может подменить простые признаки;
- различать классификационный hash и доверенную identity-базу: первый решает, обновлять ли тип, вторая сравнивает хешированный сильный UUID/serial либо минимум два слабых семейства и никогда не переписывается автоматически вслед за новым клиентом;
- при сильном противоречии прежнего MAC включать отдельный quarantine overlay. В автоматическом режиме он блокирует на уровне чёрного списка устройств, в ручном ограничивает до решения родителя; исходные права сохраняются и возвращаются при совпадении настоящего отпечатка;
- исчезновение одного mDNS/SSDP-сигнала и одно слабое несовпадение не считать подменой. Уже действующий карантин снимать только положительным совпадением либо явным подтверждением родителя;
- устройство из постоянного чёрного списка устройств не классифицировать, не сканировать и не назначать в группы; продолжать только online/offline и журнал запрещённых попыток доступа к роутеру;
- переход 2,4/5 ГГц или между mesh-точками не считать новым анализом, пока MAC и доверенный отпечаток прежние;
- исходить из того, что у ребёнка могут быть “левые телефоны”; APK ставится только родителям-администраторам, а решения по детским устройствам строятся на данных роутера;
- результат автоопределения должен быть объяснимым в UI: почему Sheepfold считает устройство сервером/хабом/приставкой;
- при сомнении показывать рекомендацию родителю, а не автоматически выдавать доступ без ограничений;
- если родитель вручную выбрал тип устройства, дальнейшее автоопределение типа для этого устройства выключается;
- если автоопределение уже дало уверенность `70` или выше и классификационный отпечаток не менялся, повторную классификацию типа можно не запускать; кандидат в `Без ограничений` всё равно должен набрать не менее `80` и два независимых семейства доказательств (§agfix88);
- даже при включённой автонастройке родитель должен видеть, какие устройства были помещены в `Без ограничений`, и иметь простой способ исправить группу.

Гостевая сеть:

- не добавлять отдельный режим гостевой сети сейчас;
- можно иметь группу `гости`;
- отдельные guest Wi-Fi функции не делать, пока владелец проекта снова явно не попросит.

Offline cleanup:

- хранить историю известных offline-устройств;
- чистить неактивные через настраиваемое число дней;
- default: 90 дней.

## Белый список, чёрный список и быстрые действия

Белый список:

- устройства никогда не блокируются глобальной кнопкой и расписаниями;
- нельзя добавить MAC, который уже в чёрном списке;
- при конфликте не удалять MAC из чёрного списка автоматически;
- нужен ручной ввод устройства по MAC;
- нужен поиск по MAC/IP/name/hostname.

Чёрный список:

- устройства всегда блокируются;
- нельзя добавить MAC, который уже в белом списке;
- при конфликте не удалять MAC из белого списка автоматически;
- по умолчанию не имеют доступа к LuCI/SSH/API;
- emergency-useful domains для них - только через отдельный явный переключатель ради безопасности.

Быстрое добавление в белый список:

- кнопка во вкладке белого списка;
- при нажатии открывается модалка;
- слева QR-код Wi-Fi;
- справа список устройств, подключившихся после запуска окна;
- окно активного подключения - 30 секунд;
- кнопка `Разрешено подключение` показывает убывающий statusbar;
- по истечении 30 секунд кнопка сереет, но остаётся кликабельной;
- повторный клик запускает новый 30-секундный период;
- кандидаты не добавляются автоматически;
- у каждого кандидата есть кнопка `Добавить`;
- backend всё равно проверяет конфликт с чёрным списком.

Пользовательское название:

- лучше узкие домены и поддомены, а не широкие родительские домены;
- объяснять ограничения HTTPS/CDN/mobile apps/DoH/cache.

Стартовый список России может включать:

- `esia.gosuslugi.ru`;
- `mos.ru`;
- `school.mos.ru`;
- `dnevnik.ru`;
- `ya.ru`;
- `2gis.ru`.

Не добавлять в default/starter/auto-generated/auto-imported/safe-minimum lists:

- широкий `yandex.ru`;
- маркетплейсы;
- супераппы;
- доставку еды;
- app stores;
- видео, музыка, игры, соцсети, короткие видео, стриминг;
- широкие порталы, которые открывают покупки или развлечения.

Большие белые и чёрные списки сайтов:

- не составлять большой детский белый список вручную внутри репозитория;

- в `Настройки -> Разное` должны быть `Источники белых списков`, `Чёрный список сайтов`, `Источники чёрного списка сайтов` и `Обновление списков сайтов`;
- итоговые белые/чёрные списки сайтов формируются backend-процессом на роутере, нормализуются и обязательно очищаются от дублей до применения;

Yandex:

- для простого поиска использовать `ya.ru`;
- для карт по умолчанию лучше `2gis.ru`;
- широкий `yandex.ru` не добавлять автоматически нигде: ни в defaults, ни в examples, ни в fixtures, ни в screenshots;
- объяснять, что широкий Yandex может открыть видео, музыку, игры, ленты, развлечения и другие сервисы;
- Yandex Go / Taxi / Market / Food / Lavka / Delivery не добавлять по умолчанию.

Такси:

- не включать такси в строгий стартовый список;

- Yandex Go не предлагать по умолчанию из-за широкого суперапп-сценария.

Банки:

- могут быть полезны ребёнку или взрослому при блокировке;
- категория `банки` допустима как ручная/off-by-default;
- не включать в строгий детский starter без решения родителя.

## Администраторы и сопряжение Android

В текущем MVP не показывать пользователю выбор роли администратора.

Если позже понадобятся роли, минимальная модель:

- `owner` управляет администраторами и не может быть удалён или понижен, если он последний owner;
- `admin` управляет семейными интернет-правилами и не удаляет последнего owner.

Не добавлять скрытые child/client роли или детские интерфейсы с административными возможностями. Отдельное приложение `android-child/` допустимо только как явно установленный статусный/помогающий клиент без управления правилами; статус устройства вычисляет роутер, а не само приложение.

Вкладка `Администраторы`:

- один администратор создаётся при first setup;
- дополнительные администраторы имеют уникальное имя, логин и пароль;
- пароли хранить только salted hash;
- дефолтных паролей не должно быть.

Админское устройство:

- админские устройства привязываются только во вкладке `Администраторы`, через действие `Привязать устройства`;
- в общем списке устройств не должно быть кнопки `Сделать админом`;
- устройства из чёрного списка недоступны для привязки;
- устройства, уже привязанные к другому администратору, недоступны для повторной привязки;
- при привязке устройство выходит из обычных групп/расписаний и получает безопасный доступ, чтобы родитель не заблокировал сам себя;
- в таблице показывается специальный локальный значок по смыслу FontAwesome `laptop-mobile`;
- не hotlink внешних CDN и FontAwesome;
- рядом есть действие `Сопряжение`.

Сопряжение:

- открывает модальное окно с QR-кодом;
- рядом показываются ручные настройки;
- QR содержит только минимальный pairing payload;
- payload: router/API address, admin login or id, one-time pairing token/code, token lifetime, device/MAC context;
<!-- §e4bq1mn -->
- QR не содержит root-пароль роутера, LuCI session cookies, bot tokens, AI keys, пароли, full backups или другие секреты;
- token короткоживущий, одноразовый, revocable;
- token хранится на роутере только как hash/non-reusable secret;
- после успешного сопряжения backend роутера немедленно помечает token/code как использованный (`consumed`) и больше никогда не принимает его повторно, даже если QR сфотографировали или ручной код переписали;
<!-- §j9wv3sz -->
- сжигание одноразового token/code реализуется именно на backend роутера. Android state, LuCI frontend state, признак "уже отсканировано" или скрытие кнопки не являются защитой от повторного использования;
- перед выдачей постоянного Android credential backend проверяет hash token, TTL, администратора-владельца, admin-device, consumed state и revocation state;
- ручной временный пароль должен генерироваться backend-частью роутера через криптографически стойкий генератор и быть удобным для переноса руками, но сложным: 10 случайных символов из safe lowercase `abcdefghkmnpqrstuvwxyz`, safe uppercase `ABCDEFGHKMNPQRSTUVWXYZ`, safe digits `2456789`, safe specials `+-*()[]{}<>?@#$%^&:;.,`; не больше 3 спецсимволов в одном пароле;
- pairing actions логируются с masking.

## Android-приложение

Название: `Sheepfold`.

Package: `app.sheepfold.android`.

Минимум: Android 9.0 / API 28.

Стек:

- Kotlin;
- Jetpack Compose;
- Android Keystore для токенов и локальных секретов;
- один Android build root: `android/`;
- модуль приложения: `android/app/`;
- не добавлять root-level Gradle files.

<!-- §z5ck8mv -->
Основное Android-приложение из каталога `android/` ставится только на телефоны родителей-администраторов. Не делать скрытую установку на телефоны детей. Если развивается детское приложение из `android-child/`, оно должно быть отдельным, явно установленным клиентом без административных функций.

Первый запуск:

1. показать пользовательское соглашение и чекбокс согласия;
2. попросить подключиться к домашнему Wi-Fi роутера Sheepfold и дать переход в системные Wi-Fi настройки;
3. проверить/объяснить настоящий MAC для этой Wi-Fi сети: если включён randomized/private MAC, провести пользователя в системные Wi-Fi настройки;
4. не обещать автоматическое переключение MAC;
5. не продолжать pairing, пока router-side проверка не подтвердит реальный MAC выбранного admin-device;
6. выполнить pairing через QR из LuCI, загрузку QR-картинки из файлов или ручной ввод;
7. после успешного подключения к роутеру настроить локальную защиту приложения;
8. рекомендовать пароль/PIN для локальной защиты;
9. биометрию можно предложить, но не как самый безопасный вариант;
10. кратко объяснить: fingerprint/face unlock может быть менее безопасен, потому что ребёнок может попробовать разблокировать приложение, пока родитель спит.

Connectivity:

- полный интерфейс работает локально с роутером;
- вне дома Android не должен обещать full control;
- remote commands идут через выбранный messenger adapter на роутере;
- VPN/WireGuard/tunnel onboarding не делать.

AI tab:

- ИИ-помощник в Android - для родителей;
- в детском APK ИИ-помощник может быть видимым, но вопрос отправляется только если backend подтвердил, что устройство ребёнка зарегистрировано, не в blocklist, не админское, родитель включил детский AI и устройство находится в личной группе;
- если provider на роутере настроен, но устройство не в личной группе, детский APK показывает баннер: попросите родителя поместить устройство в вашу личную группу;
- long prompt хранить отдельно;
- текущий prompt считается draft до проверки владельцем и семейным психологом;
- DeepSeek может быть preferred default только в country profiles, где он разрешён и доступен;
- provider layer должен быть абстрактным;
- нельзя захардкодить один DeepSeek как единственный глобальный вариант.

## Мессенджеры и бот

<!-- §r2at5nq -->
Один роутер - один активный messenger adapter.

Варианты:

- VK - default first-run choice;
- Telegram - стабильная альтернатива;
- MAX - experimental, disabled by default.

Нельзя включать несколько adapters одновременно.

Общие требования:

- bot runs/configured router-side;
- команды принимает только от явно разрешённых admin users/chat ids;
- каждый admin binding настраивается в роутере;
- administrative actions требуют confirmation;
- все adapters используют один Sheepfold API;
- сообщения не должны содержать лишние чувствительные данные.

Команды должны включать как минимум:

- показать все устройства;
- найти устройство;
- временно разрешить;
- заблокировать;
- разблокировать;
- показать статус;
- глобально выключить/включить интернет с подтверждением;
- update/reboot/import/export только с отдельным подтверждением и ролью owner, если нужно.

Telegram:

- prefer outbound HTTPS long polling (`getUpdates`);
- не открывать входящие порты и webhooks на роутер;
- хранить offset локально.

VK:

- использовать тот же adapter interface;
- default choice при первичной настройке;
- если возможно, использовать исходящий long-poll/polling сценарий, не открывая роутер наружу.

MAX:

- экспериментальный;
- не blocking requirement для первого релиза;
- не включать по умолчанию;
- реализацию делать только после проверки актуального публичного API.

## ИИ-помощник и country profiles

<!-- §t4lq9bw -->
ИИ-помощник не управляет роутером сам. Он может советовать, но любое действие применяет только parent/admin после явного подтверждения.

Country profiles управляют:

- доступными AI providers;
- suggested emergency-useful sites;
- domain descriptions and warnings;
- localization defaults.

Manual user entries должны переживать смену country.

Data sharing:

- по умолчанию AI provider получает только текст, который родитель сам написал;
<!-- §u6fj2sr -->
- MAC/IP/device names/child names/family details/logs/device lists/router settings не отправлять автоматически;
- для расширенного контекста нужен отдельный preview screen;
- preview показывает, какие поля уйдут;
- пользователь должен явно подтвердить отправку;
- поля должны быть минимизированы и, где возможно, обезличены.

Запрещено отправлять AI без отдельного подтверждения:

- MAC addresses;
- IP addresses;
- реальные имена детей;
- имена устройств;
- семейные детали;
- logs;
- device lists;
- router settings;
- full AI prompts history;
- banking/medical/private data.

## Интеграции AdGuard Home и Podkop

LuCI setting:

```text
Use together with / Использование совместно с
```

Значения:

- `none`;
- `adguard`;
- `podkop`;
- `adguard_podkop`.

AdGuard Home и Podkop не взаимоисключающие.

Installer должен определить, что уже установлено, и выбрать default mode:

- ничего нет -> `none`;
- только AdGuard Home -> `adguard`;
- только Podkop -> `podkop`;
- оба -> `adguard_podkop`.

При выборе режима UI должен показывать нюансы и предупреждения.

Нельзя:

- blindly overwrite AdGuard Home config;
- blindly overwrite Podkop config;
- менять Podkop-managed Dnsmasq/nftables/sing-box/routing без документированного решения и подтверждения;
- обещать идеальную совместимость без диагностики.

Ссылки для документации:

- Podkop + AdGuard Home: https://podkop.net/docs/adguard/
- очистка browser cache после LuCI updates: https://podkop.net/docs/clear-browser-cache/
- выбор OpenWRT router: https://hattabbi4.github.io/openwrt-router-compare/

## Wi-Fi

Sheepfold может дать family-facing shortcut для Wi-Fi настроек.

Настройки:

- 2.4 GHz SSID;
- 2.4 GHz password;
- 5 GHz SSID;
- 5 GHz password;
- advanced: security mode;
- advanced: channel;
- optional channel width, если безопасно.
- автоматическое включение всего Wi-Fi: `никогда` или ежедневное время;
- автоматическое выключение всего Wi-Fi: `никогда` или ежедневное время.

2.4 GHz и 5 GHz редактируются отдельно.

Любая смена Wi-Fi должна требовать confirmation, потому что администратор может потерять подключение.

Автоматическое включение/выключение применяется ко всем Wi-Fi радиомодулям роутера. Backend должен журналировать фактическое действие и не выполнять его, если настройки роутера не позволяют безопасно определить целевые радиомодули.

<!-- §f8ck5ry -->
Если родитель включает любое автоматическое выключение Wi-Fi, интерфейс обязан показать непропускаемый дисклеймер с обратным отсчётом 10 секунд. Кнопка подтверждения называется `Я понимаю риск, продолжить` и становится активной только после отсчёта. Дисклеймер объясняет, что после отключения Wi-Fi телефон, подключённый только по Wi-Fi, не сможет включить его обратно; заранее нужно настроить мессенджер или действие кнопки WPS для аварийного включения Wi-Fi вне расписания.

Кнопка WPS:

- настройки размещать в `Настройки -> Разное`;
- отдельные действия для короткого и долгого нажатия;
- варианты: `функционал роутера по умолчанию`, `дать подключиться к Wi-Fi`, `дать подключиться к Wi-Fi и добавить устройства в белый список (опасно)`, `отключить Wi-Fi`;
- предупреждение для опасного режима: `Добавление устройств в белый список через WPS кнопку опасно так как при нажатии на неё, в течении 30 секунд - любое устройство сможет подключиться к вифи и попасть в белый список.`;
- когда кнопка WPS нажимается и открывается 30-секундное окно подключения, все светодиоды должны мигать паттерном `1010000`; один тик — полсекунды.

Светодиоды роутера:

- настройки размещать в `Настройки -> Разное`;
- варианты: `функционал роутера по умолчанию`, `отключить навсегда все светодиоды`, `сигнал светодиодами о новом устройстве до входа в LuCI`;
- в интерфейсе пояснение для режима сигнала показывать только при выборе этого режима: `Когда подключилось новое устройство, светодиоды на роутере зажгуться.`;
- после успешного входа в LuCI с паролем сразу возвращать дефолтное поведение светодиодов роутера;
- после просмотра уведомления о новом устройстве на телефоне любым администратором тоже сразу возвращать дефолтное поведение светодиодов роутера;
- простое открытие страницы без успешной авторизации не должно сбрасывать сигнал;
- реализация зависит от модели роутера и OpenWRT LED triggers, поэтому backend обязан показывать понятную ошибку, если режим не поддерживается.

Не скрывать и не заменять стандартные OpenWRT wireless pages.

## Логи и приватность

Sheepfold self-hosted и family-only по умолчанию. Это важно для юридической и privacy модели.

Logging:

- configurable retention;
- default retention: `3d`;
- default max size: `1024 KB`;
- можно выключить;
- кнопка `очистить журнал` с confirmation;
- masked export by default.

Retention values:

```text
6m
3m
1m
14d
7d
3d
1d
12h
3h
1h
off
```

Писать в журнал:

- административные действия;
- кто изменил правило;
- что было изменено;
- результат: success/failure/cancelled/pending confirmation;
- технические ошибки сервиса;
- pairing events с masking.

Маскировать:

- MAC частично;
- IP в экспорте с последним октетом `x`;
- chat/user IDs частично;
- tokens/API keys/passwords/session IDs всегда `[secret]`.

Не писать вообще:

- пароли;
- bot tokens;
- API keys;
- session cookies;
- полные переписки;
- полные AI-запросы;
- историю сайтов;
- raw DNS query log всех устройств;
- банковские данные;
- медицинские данные;
- точные приватные детали детей.

История интернет-активности:

- не смешивать с обычным административным журналом;
- считать отдельной чувствительной функцией, выключенной по умолчанию;
- не собирать для устройств администраторов, белого списка и явно исключённых устройств;
- не хранить полный URL, параметры, поисковые запросы, POST body, cookies, session IDs и содержимое страниц;
- если функция будет добавлена, рядом с настройкой и страницей должна быть кнопка `?` с честным объяснением приватности, срока хранения, очистки и экспорта;
- подробное правило: `docs/site-activity-logs.ru.md`.

## Страница-заглушка для заблокированных устройств

Нужна локальная HTML-страница на роутере, чтобы заблокированное устройство не видело просто бесконечную загрузку.

Правила:

- текст настраивается родителем;
- это не детский интерфейс управления;
- не давать через заглушку доступ к LuCI/SSH/API;
- не показывать чувствительные сведения о правилах семьи.
- POST/PUT/PATCH/DELETE-запросы должны получать завершённый машинно-читаемый ответ, а не зависание; предпочтительно `403` с JSON `{ "blocked": true, "reason": "family_rules" }`;
<!-- §c7ds2xv -->
- игровые консоли, роботы-пылесосы, колонки и другие IoT-устройства должны получать короткий предсказуемый HTTP-ответ или корректный TCP reject/drop-policy, чтобы устройство не висело в бесконечном ожидании;
- HTML-страница нужна только для браузеров; API/IoT-клиентам не навязывать тяжёлую HTML-страницу.

## Репозиторий и сборка

Структура:

- `package/` - OpenWRT/LuCI package;
- `android/` - Android build root родительского/admin-приложения;
- `android-child/` - отдельный Android build root детского приложения без административных функций;
- `android/app/` и `android-child/app/` - application modules;
- `bot/` - messenger adapters;
- `docs/` - продуктовые и технические документы;
- `scripts/` - helper scripts.

Android:

- использовать Gradle Wrapper из соответствующего Android build root;
- Windows commands: `android\gradlew.bat -p android :app:assembleDebug` и `android-child\gradlew.bat -p android-child :app:assembleDebug`;
- локальный `android/local.properties` не коммитить;
- APK/build outputs не коммитить.

OpenWRT:

- package scripts LF-only;
- shell scripts проверять через `bash -n`, если менялись;
- JSON проверять валидатором, если менялся;
- не коммитить `.ipk`, `.apk`, secrets, local env.

## Что делать не надо

Не делать:

- cloud backend разработчика для управления Sheepfold; согласованное исключение `§feedback` не расширять на другие функции;
- full remote Android/LuCI management через WireGuard/VPN/tunnel;
- установку Android-приложения на телефоны детей;
- child-facing control interface без нового явного решения;
- скрытую слежку;
- сбор истории сайтов и raw DNS-query-log всех устройств;
- default broad `yandex.ru`;
- default marketplaces/super-apps/food delivery/app stores/entertainment;
- Yandex Go/Yandex Taxi в default emergency-useful list;
- несколько активных messenger adapters на одном роутере;
- MAX как обязательный first-release blocker;
- default passwords;
- хранение router root password в Android/QR/export/logs;
- QR с LuCI session cookies, bot tokens, AI keys или root password;
- автоматическую отправку MAC/IP/child names/device names/logs/family context в AI;
- blind overwrite чужих AdGuard Home/Podkop configs;
- legacy firewall3/iptables support;
- отдельные root-level Gradle files для Android;
- разрастание production LuCI в один огромный `overview.js`;
- удаление пользовательских настроек при uninstall;
- unencrypted export с секретами по умолчанию;
- обещание автоматического отключения Android randomized/private MAC;
- опасные действия без confirmation.

## Definition of Done для агентов

Изменение можно считать готовым, если:

- оно не нарушает эти инструкции и `AGENTS.md`;
- пользовательские изменения описаны в EN/RU документах, где нужно;
- validations есть не только в UI, но и в backend;
- опасные действия требуют confirmation;
- приватные данные не попали в logs/export/QR/AI/bot messages;
- import/export не ломает existing config;
- allowlist/blocklist conflict невозможен;
- blocklist остаётся самым сильным правилом;
- Android остаётся parent/admin-only;
- remote full-control не появился;
- LuCI strings локализуемы;
- shell scripts/JSON/Gradle сборка проверены по затронутой части.
