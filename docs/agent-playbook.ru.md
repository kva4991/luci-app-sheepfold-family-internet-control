# Подробные инструкции для ИИ-агентов

Этот документ фиксирует решения владельца проекта и служит рабочим playbook для ИИ-разработчиков. Если возникает противоречие между этим документом и старым каркасом кода, сначала проверь `AGENTS.md`, `docs/product-requirements.md` и свежие документы в `docs/`, затем сохраняй продуктовую логику из этих инструкций.

## Главная идея продукта

Sheepfold Family Internet Control - self-hosted семейный инструмент для OpenWRT-роутера. Он управляет доступом домашних устройств в интернет через OpenWRT-роутер и его веб-интерфейс LuCI, а не через облако разработчика.

Основные части:

- OpenWRT-пакет `luci-app-sheepfold-family-internet-control`;
- LuCI-интерфейс `Sheepfold Family Internet Control` / `Sheepfold : контроль доступа в интернет для семьи`;
- backend-сервис на роутере;
- Android-приложение `Овчарня` для телефонов родителей-администраторов;
- один активный мессенджер-бот на роутере: VK по умолчанию при настройке, Telegram как стабильная альтернатива, MAX только экспериментально;
- ИИ-помощник для родителей в Android-приложении, без автоматической передачи чувствительных данных.

## Названия и формулировки

- Основное имя проекта в EN/RU текстах: `Sheepfold`.
- `Овчарня` использовать только как название Android-приложения. В русском LuCI-заголовке писать `Sheepfold`, а не `Овчарня`.
- Не писать в русском README "управлять через LuCI" без контекста. Пиши понятно: "управлять доступом домашних устройств в интернет через OpenWRT-роутер и его веб-интерфейс LuCI".
- Не переименовывать GitHub repo, OpenWRT package, Android package и LuCI display names без явного решения владельца проекта.

## Архитектурные инварианты

- Sheepfold стоит первым в цепочке принятия решения по устройству.
- Целевая цепочка трафика: `Sheepfold -> AdGuard Home -> Podkop`.
- Sheepfold решает, можно ли устройству выходить в сеть.
- AdGuard Home фильтрует DNS, если включён.
- Podkop маршрутизирует разрешённый трафик, если включён.
- Проект не должен ломать стандартные OpenWRT firewall/DNS настройки, AdGuard Home, Podkop, Dnsmasq, sing-box и пользовательские конфиги.
- Автоматические изменения чужих конфигов допустимы только после явного подтверждения и желательно после backup/export.
- Полноценное управление через Android и LuCI - только в локальной сети.
- Вне дома - только короткие подтверждаемые команды и уведомления через выбранный мессенджер.
- Не добавлять developer-operated cloud backend.
- Не добавлять full remote management через WireGuard, VPN или другие туннели к роутеру.

## OpenWRT-пакет и backend

Цель - современные OpenWRT с `firewall4` / `nftables`.

Не реализовывать поддержку старых OpenWRT с `firewall3` / `iptables`, пока владелец проекта явно не попросит.

Backend должен:

- иметь узкий API/command layer, например `/usr/bin/sheepfold <method>` или эквивалентный безопасный слой;
- возвращать структурированный JSON для LuCI, Android и ботов;
- валидировать MAC, IP, hostname, домены, роли, имена групп и импортируемые данные;
- быть идемпотентным: повторный запуск не плодит дубли nftables/UCI/rpcd/service-записей;
- восстанавливать правила после `fw4 restart`, перезагрузки роутера и restart сервиса;
- не строить shell-команды из пользовательского ввода;
- логировать административные действия без секретов.

Installer должен:

- показать ссылку на пользовательское соглашение и требовать явное `yes`, `y` или `да`;
- проверить OpenWRT-окружение;
- проверить, задан ли root-пароль OpenWRT, или хотя бы обеспечить gate перед открытием Sheepfold в LuCI;
- обнаружить AdGuard Home и Podkop;
- выставить рекомендуемый `integration_mode`, но не переписывать чужие настройки без отдельного подтверждения;
- не активировать мессенджер до ввода credentials и привязки администратора.

Uninstall должен:

- удалять пакет без очистки Sheepfold-настроек и списков клиентов;
- после удаления показывать отчёт об оставшихся настройках/следах на роутере, которые могут требовать ручного решения;
- не удалять пользовательские данные молча.

Update должен:

- ставить только stable-релизы;
- обновлять LuCI cache-busting version;
- не ломать существующий пользовательский config;
- при необходимости чистить LuCI index/module cache.

## LuCI-интерфейс

LuCI - основной интерфейс настройки. Он должен быть понятен родителю, а не только сетевому администратору.

Обязательные вкладки:

- все устройства;
- белый список;
- чёрный список;
- расписания;
- доступ к аварийно-полезным сайтам;
- Wi-Fi;
- интеграции;
- администраторы;
- бот/мессенджер;
- журнал;
- настройки;
- диагностика.

Производственная LuCI-архитектура:

- Podkop-подобная структура как основной ориентир для LuCI-реализации;
- небольшой entrypoint;
- `form.Map("sheepfold")`;
- `tabbed = true`;
- отдельный JS/module слой на вкладку;
- общий store/API client;
- строгий `rpcd` ACL;
- backend-вызовы через узкий Sheepfold API;
- не разращивать один огромный `overview.js`.

Что именно брать из подхода Podkop:

- раздельную структуру LuCI frontend/backend вместо монолитной страницы;
- маленький LuCI entrypoint, который собирает вкладки, а не содержит всю бизнес-логику;
- отдельные модули для вкладок и диагностики;
- backend methods/commands, которые возвращают понятный JSON;
- строгий `rpcd` ACL только на нужные файлы, UCI-секции, ubus-объекты и команды;
- диагностическую вкладку с проверками состояния сервиса, firewall, dns/integration mode и версий;
- аккуратную работу с LuCI cache после обновлений;
- понятный путь установки/обновления из GitHub stable releases.

Что не копировать слепо из Podkop:

- не переносить routing/sing-box логику в Sheepfold;
- не менять Podkop-managed правила без отдельного документированного решения;
- не делать UI для сетевых специалистов в ущерб родительскому сценарию;
- не смешивать responsibility: Sheepfold управляет доступом устройств, Podkop управляет маршрутизацией, AdGuard Home управляет DNS-фильтрацией.

UI-правила:

- показывать одну понятную кнопку `Сохранить`;
- не заставлять родителя выбирать между `Сохранить` и `Применить`;
- опасные действия требуют подтверждения;
- все строки должны быть локализуемыми;
- русский - основной источник формулировок, английский - обязательный fallback;
- planned languages можно генерировать позже, но код не должен мешать локализации.

Опасные действия:

- перезагрузка роутера;
- обновление приложения;
- импорт настроек;
- экспорт полного backup с секретами;
- очистка журнала;
- массовая блокировка;
- смена Wi-Fi SSID/password/security/channel;
- смена активного мессенджера;
- добавление/удаление/понижение администратора;
- включение аварийно-полезных сайтов для чёрного списка;
- автоматическая настройка интеграции с AdGuard Home/Podkop.

Root password gate:

- при первом открытии Sheepfold в LuCI проверить root-пароль OpenWRT;
- если пароль не задан, не пускать в настройки Sheepfold;
- показать понятное предупреждение и кнопку/ссылку на стандартную страницу установки пароля OpenWRT;
- не создавать дефолтные Sheepfold-пароли.

LuCI cache:

- все Sheepfold JS/CSS/static assets используют один suffix `?v=<PKG_VERSION>-<PKG_RELEASE>`;
- source of truth - версия OpenWRT-пакета;
- не хардкодить отдельные версии в отдельных файлах;
- bump package release/version при изменении frontend-файлов.

## Устройства и правила доступа

Определять устройства по MAC-адресу. Искать можно по:

- MAC;
- текущему IP;
- hostname;
- пользовательскому имени.

Данные устройств брать автоматически из:

- DHCP leases;
- static DHCP leases;
- ARP/neighbor data;
- других безопасных OpenWRT-источников, если они есть.

Имена устройств и постоянные IP должны быть синхронизированы с OpenWRT DHCP static leases. Если пользователь меняет имя или постоянный IP в Sheepfold, это должно быть согласовано с разделом DHCP -> "постоянная аренда" на роутере.

Приоритет правил:

1. чёрный список;
2. белый список;
3. временный доступ;
4. расписание;
5. общие правила и политика новых устройств.

Жёсткие правила:

- один MAC не может быть одновременно в белом и чёрном списке;
- backend, LuCI, Android, bot commands и import validation должны это проверять;
- чёрный список всегда сильнее временного доступа и расписаний;
- белый список не блокируется глобальной блокировкой и расписаниями;
- глобальная кнопка `Выключить интернет` блокирует всех, кроме белого списка;
- blocklisted devices не должны открывать LuCI, SSH и Sheepfold API;
- emergency-useful access для blocklist допустим только отдельной явной настройкой и всё равно не даёт доступ к LuCI/SSH/API.

Поведение новых устройств:

- настройка `поведение для новых устройств`;
- значения: `пускать в интернет` по умолчанию или `ограничивать до настройки`;
- неизвестные устройства должны быть видны в общем списке и доступны для настройки.

Группы устройств:

- дети;
- родители;
- телевизоры/медиа;
- гости;
- без ограничений;
- пользовательские группы.

Группа `Без ограничений`:

- это не белый список конкретного устройства, а доверенная группа для домашней инфраструктуры: NAS, Home Assistant, AdGuard Home, Proxmox, сервер видеонаблюдения, Zigbee/Matter/Thread-хабы и похожие сервисные устройства;
- устройства этой группы не должны блокироваться расписаниями, временными ограничениями, правилом новых устройств и глобальной кнопкой выключения интернета;
- чёрный список всегда выше группы `Без ограничений`: если родитель вручную добавил устройство в чёрный список, оно блокируется;
- при установке нужно спросить `Применить автонастройку программы?`;
- полный режим автонастройки является режимом по умолчанию, потому что он полезен большинству семей;
- если родитель нажимает Enter или отвечает `yes`, `y` или `да`, включить `auto_configure=1`, `detection_mode=full`, `no_restrictions_auto_assign=1`;
- при включённой автонастройке полная установка может автоматически помещать уверенно распознанные инфраструктурные устройства в эту группу;
- урезанная установка (`auto_configure=0`, `detection_mode=reduced`, `no_restrictions_auto_assign=0`) выбирается только явным ответом `no`, `n` или `нет` и нужна для роутеров с очень малым запасом места;
- урезанная установка не должна автоматически помещать новые устройства в эту группу.

Сильное автоопределение устройств:

- использовать не только MAC/hostname, а несколько router-side сигналов: DHCP/static lease, hostname, vendor/OUI, открытые порты, service banners, mDNS/SSDP/UPnP имена, ранее подтверждённый fingerprint;
- реализовывать сильное автоопределение отдельным backend-компонентом/пакетом, например `sheepfold-device-detector`, а не внутри LuCI-only пакета;
- LuCI должен вызывать Sheepfold backend API и оставаться лёгким интерфейсом;
- в полном режиме можно использовать существующие OpenWRT-инструменты, если они установлены или разрешены как зависимости полного режима: `nmap`/`nmap-ssl` для портов и баннеров, `avahi-utils` или аналоги для mDNS, SSDP/UPnP probes;
- эти инструменты не должны быть обязательными зависимостями урезанной установки;
- не запускать тяжёлые сканирования постоянно: делать ограниченные локальные проверки, кэшировать результат и показывать уверенность/объяснение;
- не считать MAC, hostname или открытый порт криптографическим доказательством, потому что ребёнок может подменить простые признаки;
- результат автоопределения должен быть объяснимым в UI: почему Sheepfold считает устройство сервером/хабом/приставкой;
- при сомнении показывать рекомендацию родителю, а не автоматически выдавать доступ без ограничений;
- даже при включённой автонастройке родитель должен видеть, какие устройства были помещены в `Без ограничений`, и иметь простой способ исправить группу.

Гостевая сеть:

- не добавлять отдельный режим гостевой сети сейчас;
- можно иметь группу `гости`;
- отдельные guest Wi-Fi функции не делать, пока владелец проекта снова явно не попросит.

Offline cleanup:

- хранить историю известных offline-устройств;
- чистить неактивные через настраиваемое число дней;
- default: 90 дней.

## Белый список, чёрный список и быстрые действия

Белый список:

- устройства никогда не блокируются глобальной кнопкой и расписаниями;
- нельзя добавить MAC, который уже в чёрном списке;
- нужен ручной ввод устройства по MAC;
- нужен поиск по MAC/IP/name/hostname.

Чёрный список:

- устройства всегда блокируются;
- нельзя добавить MAC, который уже в белом списке;
- по умолчанию не имеют доступа к LuCI/SSH/API;
- emergency-useful domains для них - только через отдельный явный переключатель ради безопасности.

Быстрое добавление в белый список:

- кнопка во вкладке белого списка;
- при нажатии открывается модалка;
- слева QR-код Wi-Fi;
- справа список устройств, подключившихся после запуска окна;
- окно активного подключения - 30 секунд;
- кнопка `Разрешено подключение` показывает убывающий statusbar;
- по истечении 30 секунд кнопка сереет, но остаётся кликабельной;
- повторный клик запускает новый 30-секундный период;
- кандидаты не добавляются автоматически;
- у каждого кандидата есть кнопка `Добавить`;
- backend всё равно проверяет конфликт с чёрным списком.

## Расписания и временный доступ

Расписания должны поддерживать:

- правила `разрешить` и `заблокировать`;
- дни недели;
- диапазоны времени;
- включено/выключено;
- цели: устройство или группа;
- интервалы через полночь;
- deterministic conflict resolution;
- предупреждения о конфликтующих правилах.

Временный доступ:

- может переопределить расписание;
- никогда не переопределяет чёрный список;
- должен логироваться как административное действие.

Быстрые кнопки:

- +15 минут;
- +30 минут;
- +1 час;
- +2 часа;
- +3 часа;
- +5 часов;
- до конца суток;
- до отбоя.

Время отбоя:

- default: 21:00;
- настраивается в разделе расписаний.

## Аварийно-полезные сайты

Пользовательское название:

- RU: `Доступ к аварийно-полезным сайтам`;
- EN: `Access to emergency-useful sites`.

Это не развлекательный белый список и не замена AdGuard Home presets. Это минимальный редактируемый список доменов, который может помочь в бытовых/экстренных ситуациях, не открывая полноценный интернет.

Правила:

- список редактируемый;
- не делать встроенные пресеты;
- стартовые предложения родитель должен просмотреть и подтвердить;
- у каждой автоматически предложенной записи должны быть domain, display name, понятное описание, optional warning, source;
- лучше узкие домены и поддомены, а не широкие родительские домены;
- объяснять ограничения HTTPS/CDN/mobile apps/DoH/cache.

Стартовый список России может включать:

- `gosuslugi.ru`;
- `esia.gosuslugi.ru`;
- `mos.ru`;
- `school.mos.ru`;
- `dnevnik.ru`;
- `ya.ru`;
- `2gis.ru`.

Не добавлять в default/starter/auto-generated/auto-imported/safe-minimum lists:

- широкий `yandex.ru`;
- маркетплейсы;
- супераппы;
- доставку еды;
- app stores;
- видео, музыка, игры, соцсети, короткие видео, стриминг;
- широкие порталы, которые открывают покупки или развлечения.

Yandex:

- для простого поиска использовать `ya.ru`;
- для карт по умолчанию лучше `2gis.ru`;
- широкий `yandex.ru` не добавлять автоматически нигде: ни в defaults, ни в examples, ни в fixtures, ни в screenshots;
- объяснять, что широкий Yandex может открыть видео, музыку, игры, ленты, развлечения и другие сервисы;
- Yandex Go / Taxi / Market / Food / Lavka / Delivery не добавлять по умолчанию.

Такси:

- такси полезно в экстренной ситуации, но часто является супераппом;
- не включать такси в строгий стартовый список;
- для России ручные варианты для осознанного добавления: `taximaxim.ru`, `city-mobil.ru`, `vezet.ru`, `taxovichkof.ru`;
- Yandex Go не предлагать по умолчанию из-за широкого суперапп-сценария.

Банки:

- могут быть полезны ребёнку или взрослому при блокировке;
- категория `банки` допустима как ручная/off-by-default;
- не включать в строгий детский starter без решения родителя.

## Администраторы и сопряжение Android

Минимальные роли:

- `owner`;
- `admin`.

Owner:

- управляет администраторами;
- не может быть удалён или понижен, если он последний owner.

Admin:

- управляет семейными интернет-правилами;
- не удаляет последнего owner.

Не добавлять child/client роли и интерфейсы для детей, пока владелец проекта явно не попросит.

Вкладка `Администраторы`:

- один owner создаётся при first setup;
- дополнительные администраторы имеют уникальное имя, логин, роль и пароль;
- пароли хранить только salted hash;
- дефолтных паролей не должно быть.

Админское устройство:

- любое обнаруженное устройство можно сделать админским;
- при этом LuCI спрашивает, какому администратору оно принадлежит;
- в таблице показывается специальный локальный значок по смыслу FontAwesome `laptop-mobile`;
- не hotlink внешних CDN и FontAwesome;
- рядом есть действие `Сопряжение`.

Сопряжение:

- открывает модальное окно с QR-кодом;
- рядом показываются ручные настройки;
- QR содержит только минимальный pairing payload;
- payload: router/API address, admin login or id, one-time pairing token/code, token lifetime, device/MAC context;
- QR не содержит root-пароль роутера, LuCI session cookies, bot tokens, AI keys, пароли, full backups или другие секреты;
- token короткоживущий, одноразовый, revocable;
- token хранится на роутере только как hash/non-reusable secret;
- после успешного сопряжения backend роутера немедленно помечает token/code как использованный (`consumed`) и больше никогда не принимает его повторно, даже если QR сфотографировали или ручной код переписали;
- сжигание одноразового token/code реализуется именно на backend роутера. Android state, LuCI frontend state, признак "уже отсканировано" или скрытие кнопки не являются защитой от повторного использования;
- перед выдачей постоянного Android credential backend проверяет hash token, TTL, администратора-владельца, admin-device, consumed state и revocation state;
- ручной временный пароль должен генерироваться backend-частью роутера через криптографически стойкий генератор и быть удобным для переноса руками, но сложным: 10 случайных символов из safe lowercase `abcdefghkmnpqrstuvwxyz`, safe uppercase `ABCDEFGHKMNPQRSTUVWXYZ`, safe digits `2456789`, safe specials `+-*()[]{}<>?@#$%^&:;.,`; не больше 3 спецсимволов в одном пароле;
- pairing actions логируются с masking.

## Android-приложение

Название: `Овчарня`.

Package: `app.sheepfold.android`.

Минимум: Android 9.0 / API 28.

Стек:

- Kotlin;
- Jetpack Compose;
- Android Keystore для токенов и локальных секретов;
- один Android build root: `android/`;
- модуль приложения: `android/app/`;
- не добавлять root-level Gradle files.

Android-приложение ставится только на телефоны родителей-администраторов. Не делать скрытую установку на телефоны детей.

Первый запуск:

1. показать пользовательское соглашение и чекбокс согласия;
2. попросить подключиться к домашнему Wi-Fi роутера Sheepfold и дать переход в системные Wi-Fi настройки;
3. проверить/объяснить настоящий MAC для этой Wi-Fi сети: если включён randomized/private MAC, провести пользователя в системные Wi-Fi настройки;
4. не обещать автоматическое переключение MAC;
5. не продолжать pairing, пока router-side проверка не подтвердит реальный MAC выбранного admin-device;
6. выполнить pairing через QR из LuCI, загрузку QR-картинки из файлов или ручной ввод;
7. после успешного подключения к роутеру настроить локальную защиту приложения;
8. рекомендовать пароль/PIN для локальной защиты;
9. биометрию можно предложить, но не как самый безопасный вариант;
10. кратко объяснить: fingerprint/face unlock может быть менее безопасен, потому что ребёнок может попробовать разблокировать приложение, пока родитель спит.

Connectivity:

- полный интерфейс работает локально с роутером;
- вне дома Android не должен обещать full control;
- remote commands идут через выбранный messenger adapter на роутере;
- VPN/WireGuard/tunnel onboarding не делать.

AI tab:

- ИИ-помощник в Android - для родителей;
- long prompt хранить отдельно;
- текущий prompt считается draft до проверки владельцем и семейным психологом;
- DeepSeek может быть preferred default только в country profiles, где он разрешён и доступен;
- provider layer должен быть абстрактным;
- нельзя захардкодить один DeepSeek как единственный глобальный вариант.

## Мессенджеры и бот

Один роутер - один активный messenger adapter.

Варианты:

- VK - default first-run choice;
- Telegram - стабильная альтернатива;
- MAX - experimental, disabled by default.

Нельзя включать несколько adapters одновременно.

Общие требования:

- bot runs/configured router-side;
- команды принимает только от явно разрешённых admin users/chat ids;
- каждый admin binding настраивается в роутере;
- administrative actions требуют confirmation;
- все adapters используют один Sheepfold API;
- сообщения не должны содержать лишние чувствительные данные.

Команды должны включать как минимум:

- показать все устройства;
- найти устройство;
- временно разрешить;
- заблокировать;
- разблокировать;
- показать статус;
- глобально выключить/включить интернет с подтверждением;
- update/reboot/import/export только с отдельным подтверждением и ролью owner, если нужно.

Telegram:

- prefer outbound HTTPS long polling (`getUpdates`);
- не открывать входящие порты и webhooks на роутер;
- хранить offset локально.

VK:

- использовать тот же adapter interface;
- default choice при первичной настройке;
- если возможно, использовать исходящий long-poll/polling сценарий, не открывая роутер наружу.

MAX:

- экспериментальный;
- не blocking requirement для первого релиза;
- не включать по умолчанию;
- реализацию делать только после проверки актуального публичного API.

## ИИ-помощник и country profiles

ИИ-помощник не управляет роутером сам. Он может советовать, но любое действие применяет только parent/admin после явного подтверждения.

Country profiles управляют:

- доступными AI providers;
- suggested emergency-useful sites;
- domain descriptions and warnings;
- localization defaults.

Manual user entries должны переживать смену country.

Data sharing:

- по умолчанию AI provider получает только текст, который родитель сам написал;
- MAC/IP/device names/child names/family details/logs/device lists/router settings не отправлять автоматически;
- для расширенного контекста нужен отдельный preview screen;
- preview показывает, какие поля уйдут;
- пользователь должен явно подтвердить отправку;
- поля должны быть минимизированы и, где возможно, обезличены.

Запрещено отправлять AI без отдельного подтверждения:

- MAC addresses;
- IP addresses;
- реальные имена детей;
- имена устройств;
- семейные детали;
- logs;
- device lists;
- router settings;
- full AI prompts history;
- banking/medical/private data.

## Интеграции AdGuard Home и Podkop

LuCI setting:

```text
Use together with / Использование совместно с
```

Значения:

- `none`;
- `adguard`;
- `podkop`;
- `adguard_podkop`.

AdGuard Home и Podkop не взаимоисключающие.

Installer должен определить, что уже установлено, и выбрать default mode:

- ничего нет -> `none`;
- только AdGuard Home -> `adguard`;
- только Podkop -> `podkop`;
- оба -> `adguard_podkop`.

При выборе режима UI должен показывать нюансы и предупреждения.

Нельзя:

- blindly overwrite AdGuard Home config;
- blindly overwrite Podkop config;
- менять Podkop-managed Dnsmasq/nftables/sing-box/routing без документированного решения и подтверждения;
- обещать идеальную совместимость без диагностики.

Ссылки для документации:

- Podkop + AdGuard Home: https://podkop.net/docs/adguard/
- очистка browser cache после LuCI updates: https://podkop.net/docs/clear-browser-cache/
- выбор OpenWRT router: https://hattabbi4.github.io/openwrt-router-compare/

## Wi-Fi

Sheepfold может дать family-facing shortcut для Wi-Fi настроек.

Настройки:

- 2.4 GHz SSID;
- 2.4 GHz password;
- 5 GHz SSID;
- 5 GHz password;
- advanced: security mode;
- advanced: channel;

## Логи и приватность

Sheepfold self-hosted и family-only по умолчанию. Это важно для юридической и privacy модели.

Logging:

- configurable retention;
- default retention: `3d`;
- default max size: `1024 KB`;
- можно выключить;
- кнопка `очистить журнал` с confirmation;
- masked export by default.

Retention values:

```text
6m
3m
1m
14d
7d
3d
1d
12h
3h
1h
off
```

Писать в журнал:

- административные действия;
- кто изменил правило;
- что было изменено;
- результат: success/failure/cancelled/pending confirmation;
- технические ошибки сервиса;
- pairing events с masking.

Маскировать:

- MAC частично;
- IP в экспорте с последним октетом `x`;
- chat/user IDs частично;
- tokens/API keys/passwords/session IDs всегда `[secret]`.

Не писать вообще:

- пароли;
- bot tokens;
- API keys;
- session cookies;
- полные переписки;
- полные AI-запросы;
- историю сайтов;
- raw DNS query log всех устройств;
- банковские данные;
- медицинские данные;
- точные приватные детали детей.
