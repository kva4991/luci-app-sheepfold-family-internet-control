# Контракт UX-ревью Sheepfold

<!-- §uxrev01 -->

Интерфейс рассчитан на занятого родителя, который может слабо разбираться в компьютерах. Понятность действия и одинаковый контекст в одинаковых местах важнее декоративной оригинальности.

## Основные критерии

1. **Задача видна сразу.** Название кнопки описывает результат для семьи, а не внутренний backend-метод.
2. **Состояние замкнуто.** Для загрузки, успеха, пустого результата, ошибки, disabled и повторной попытки есть явное поведение.
3. **Одинаковое действие выглядит одинаково.** Шестерёнка означает настройку, корзина — удаление, зелёный action — разрешение/добавление, красный — отключение/опасное действие.
4. **Опасность предупреждается до команды.** Дисклеймер не заменяет безопасный default и backend validation.
5. **Технический текст не перекладывается на пользователя.** В интерфейсе нет UCI, token-burning, internal endpoint и инструкций для программиста.
6. **Сохранение честное.** Успех показывается после подтверждения backend; немедленная команда не перезагружает всю страницу без необходимости.
7. **Размер стабилен.** Текст, таблица, QR, модалка и кнопки не выходят за viewport и не перекрывают друг друга.

## Приоритет замечаний

- `P0`: можно потерять доступ, раскрыть секрет, выдать неверные права или невозможно завершить основной сценарий.
- `P1`: действие работает неверно, состояние вводит в заблуждение, mobile layout ломается, элемент недоступен без мыши/зрения текста.
- `P2`: непоследовательная формулировка, слабая иерархия или лишнее действие, не блокирующее сценарий.

## Автоматическая browser-проверка

Read-only Playwright на живом роутере проверяет desktop и mobile, основные вкладки, router info, console/request errors, горизонтальное переполнение, доступные имена видимых кнопок/ссылок, ненулевой размер интерактивных элементов и делает screenshot. Он не нажимает `Сохранить`, не меняет UCI и не является визуальной baseline-проверкой пикселей.

## Ручная проверка

Для изменённого сценария пройти его с чистым и заполненным состоянием, длинным русским текстом, mobile viewport, keyboard back/focus и реальной ошибкой backend. Проверить, что пользователь понимает: что произошло, что сохранилось, что делать дальше и какое действие опасно.
