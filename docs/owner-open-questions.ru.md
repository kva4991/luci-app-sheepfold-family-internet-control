# Открытые вопросы владельцу Sheepfold

<!-- §ownques -->

Этот файл хранит решения, без которых разработчик рискует сделать внешне убедительную, но неверную реализацию. После ответа вопрос нужно перенести в каноническое требование, добавить тест и убрать отсюда.

## Сначала решить

### 1. Кто реально применяет белые и чёрные списки сайтов

Нужна единая матрица:

- без AdGuard Home и без Podkop;
- с AdGuard Home, без Podkop;
- без AdGuard Home, с Podkop;
- одновременно с AdGuard Home и Podkop.

**Рекомендация:** Sheepfold всегда владеет правилами устройств; AdGuard Home при наличии владеет DNS-фильтрацией; Podkop владеет только своей маршрутизацией. В режиме без AdGuard Home Sheepfold использует собственные dnsmasq/nftables-объекты. Нельзя объявлять эту схему готовой до проверки всех четырёх вариантов на живом роутере.

### 2. Включать ли чёрный список сайтов по умолчанию

Сейчас заводское значение режима — `для всех кроме белого списка устройств и админских устройств`.

**Рекомендация:** для первого стабильного релиза сделать `выключен` по умолчанию и предложить включение в мастере настройки. Причина: внешние категории допускают ложные срабатывания, а совместная работа с AdGuard Home ещё не завершена.

### 3. Где хранить последнюю рабочую копию списков после перезагрузки

Сейчас большой кэш находится в `/tmp` и исчезает после перезагрузки.

**Рекомендация:** RAM остаётся рабочим местом; при доступной USB-флешке разрешить необязательный постоянный кэш. Во внутреннюю flash роутера большие списки регулярно не писать. Без USB после перезагрузки временно использовать только успешно скачанные заново данные.

### 4. Какой ресурсный класс роутера поддерживать

Нужно определить минимальные RAM и свободное хранилище и допустимое общее число доменов.

**Рекомендация:** базовый профиль рассчитан на современные роутеры уровня Xiaomi AX3000T; не более 250 000 строк на источник и не более примерно 500 000 уникальных доменов суммарно. Многомиллионные категории отдавать AdGuard Home на более мощном устройстве или отдельному серверу.

### 5. Что делать с правдоподобным, но внезапно уменьшившимся файлом

Файл может содержать корректные домены, но быть обрезанным до 1% прежнего размера.

Backend уже считает подозрительным сокращение более чем на 75% для источника, в старой копии которого было не менее 1000 доменов, сохраняет рабочую копию и разрешает отдельное ручное подтверждение.

**Открытое решение:** подтвердить эти два порога и вид LuCI-подтверждения. Рекомендуется показывать название источника, было/стало доменов и отдельную кнопку `Принять уменьшившийся список`; обычное обновление не должно обходить защиту.

### 6. Сколько хранить устаревшую рабочую копию

**Рекомендация:** продолжать применять её без автоматического отключения, показывать возраст постоянно, усилить уведомление через 7 и 30 дней. Автоматически снять защиту из-за возраста хуже, чем оставить известный старый список.

### 7. Как блокировать обход через DoH, DoT, QUIC и внешний DNS

Без этого ребёнок или приложение может обойти доменные правила.

**Рекомендация:** отдельная явно описанная настройка защиты от обхода для ограничиваемых групп; не включать агрессивную блокировку скрыто, потому что она может ломать корпоративные, банковские и VPN-приложения.

## Frontend

### 8. Как показывать состояние источников

**Рекомендация:** карточка на источник: название, включён/выключен, домены, последнее успешное обновление, текущая ошибка, старая копия и следующая попытка. Технические коды ошибок пользователю не показывать.

### 9. Когда запускать обновление после сохранения URL

**Рекомендация:** кнопка `Сохранить настройки` сохраняет UCI и ставит фоновую задачу; LuCI остаётся открытым. Рядом показывается `Проверяется`, затем результат без перезагрузки страницы.

### 10. Нужна ли ручная кнопка «Обновить списки сейчас»

**Рекомендация:** да, одна общая кнопка и отдельная кнопка у проблемного источника. Повторное нажатие во время работы показывает уже запущенную задачу, а не создаёт вторую.

### 11. Насколько подробно объяснять ложные срабатывания

**Рекомендация:** короткое понятное предупреждение всегда, подробная инструкция под значком `?`: как временно выключить источник, добавить исключение и сообщить автору списка.

## Backend и настройки

### 12. Какие заводские категории обязательны

Сейчас предложены UT1 child, HaGeZi NSFW/Gambling mini/TIF mini и URLhaus.

**Рекомендация:** считать их стартовыми предложениями, а не обещанием безопасности. Нужно решить, включать ли отдельно игры, социальные сети, VPN/DoH, наркотики и опасные материалы.

### 13. Разрешать ли источники по обычному HTTP

**Рекомендация:** HTTPS по умолчанию. HTTP разрешать только вручную с предупреждением, потому что список можно подменить по дороге.

### 14. Кого уведомлять после трёх ошибок

Сейчас событие доступно всем привязанным родительским APK, а Telegram получает немедленное сообщение, если настроен.

**Рекомендация:** уведомлять всех администраторов, но дедуплицировать событие. Нужен ли режим «только владелец»?

### 15. Нужно ли уведомление о восстановлении

Backend уже отправляет одно короткое `Источник снова обновляется`, только если до этого успел предупредить о трёх ошибках. Обычные успешные еженедельные обновления не уведомляются.

**Открытое решение:** должно ли восстановление, как и ошибка, быть видно всем администраторам, или только владельцу уведомления. До отдельной модели прочтения используется общая очередь для всех администраторов.

### 16. На каком языке хранить уведомление на роутере

Сейчас текст создаётся по-русски.

**Рекомендация:** API хранит код причины и параметры, а каждый телефон переводит сообщение своим языком. Telegram использует язык приложения роутера.

### 17. Нужны ли контрольные суммы или подписи источников

**Рекомендация:** HTTPS и проверка структуры обязательны сейчас; опциональную SHA-256 добавить для собственных/семейных источников. Подпись имеет смысл только когда поставщик публикует проверяемую подпись отдельно.

### 18. Как учитывать страну роутера

**Рекомендация:** общий безопасный набор угроз плюс отдельные выключенные по умолчанию региональные предложения. Не считать одну страну доказательством того, что конкретный домен полезен или вреден.

## Проверка реализации

### 19. Можно ли использовать живой тестовый роутер для матрицы DNS

Нужны консольный доступ и отдельное тестовое устройство, чтобы проверить dnsmasq, nftables, AdGuard Home и Podkop без риска потерять доступ к домашнему роутеру.

### 20. Какой результат считать достаточным для первого релиза

**Рекомендация:** автоматические тесты загрузчика плюс живые проверки: частично испорченный файл, HTML вместо файла, обрыв, перезагрузка, ошибка одного из нескольких источников, shared CDN, IPv4/IPv6, DoH-обход и все четыре режима интеграции.

## Общая архитектура Standard IPK

### 21. Кого указывать исполнителем действия из LuCI

Telegram и родительский APK знают конкретного администратора, а обычная LuCI-сессия обычно подтверждает только пользователя OpenWrt `root`.

**Рекомендация:** пока писать `LuCI (root)` и не угадывать имя администратора Sheepfold по MAC. Если нужен персональный аудит, добавить отдельный вход администратора Sheepfold поверх LuCI или явный выбор профиля с повторной проверкой пароля.

### 22. Что делать при потере администраторского телефона

**Рекомендация:** в LuCI дать отзыв конкретного устройства и всех его токенов одной командой. Отзыв не удаляет самого администратора и не меняет другие его устройства. Нужна ли также кнопка `Завершить все сеансы администратора`?

### 23. Как обрабатывать смену MAC или замену телефона

Автоматически склеивать записи опасно: ребёнок может имитировать имя, IP или часть сетевых признаков.

**Рекомендация:** создавать новое устройство с новым числовым ID; предложить родителю ручное объединение с показом обеих карточек. При объединении переносить только явно подтверждённые настройки, а исходную запись оставлять в журнале как алиас.

### 24. Что именно удаляет очистка старых офлайн-устройств

**Рекомендация:** автоматически удалять только не настроенные устройства без группы, списка, расписания, постоянной DHCP-аренды и административной привязки. Настроенные карточки не удалять по таймеру. Устраивает ли такой консервативный вариант?

### 25. Как расписания ведут себя при смене часового пояса и летнего времени

**Рекомендация:** расписание хранит локальные дни и часы семьи, вычисляется по текущему часовому поясу роутера, а после его смены интерфейс показывает предупреждение. Для перехода летнего времени не создавать второй скрытый UTC-график.

### 26. Как честно показывать страницу блокировки

Роутер может вернуть понятную страницу для обычного HTTP, но не способен безопасно подменять HTTPS, ответы приложений, POST-запросы, игровые консоли и бытовые устройства без перехвата TLS.

**Рекомендация:** не обещать заглушку вместо каждого сайта. Для HTTPS и приложений возвращать корректный сетевой отказ/DNS-ответ, а локальную страницу использовать там, где клиент действительно может открыть её без подмены сертификатов. MITM-сертификат в Sheepfold не внедрять.

### 27. Как восстанавливать Wi-Fi после ошибочного расписания

**Рекомендация:** кроме десятисекундного предупреждения требовать хотя бы один проверенный путь восстановления: проводной LuCI, работающий мессенджер или настроенная кнопка роутера. Нужно ли запрещать сохранение автоматического выключения, если ни один путь не подтверждён?

### 28. Что делать с WPS и светодиодами на неподдерживаемом роутере

Названия GPIO, кнопок и LED различаются даже внутри одной марки.

**Рекомендация:** сначала обнаруживать аппаратные объекты через OpenWrt, показывать только реально доступные действия и всегда иметь режим `Функционал роутера по умолчанию`. Не пытаться применять универсальный номер GPIO.

## API и Android

### 29. Какой минимум LuCI должен быть продублирован в родительском APK первого релиза

Сейчас часть экранов Android остаётся заготовками.

**Рекомендация:** до стабильного релиза обязательно завершить устройства и списки устройств, временный доступ, расписания, группы, администраторов, Wi-Fi, уведомления и информацию о роутере. Сложные сервисные настройки можно оставить только в LuCI. Подтвердите этот состав.

### 30. Какие данные разрешено отдавать детскому `/client-status`

Endpoint не требует администраторской авторизации и идентифицирует устройство по наблюдаемому роутером соединению.

**Рекомендация:** только собственный числовой ID, собственное эффективное состояние доступа, понятную причину ограничения, доступность запроса времени и факт доступности детского ИИ. Не отдавать списки устройств, имена администраторов, чужие группы, расписания или сетевую диагностику.

### 31. Что делать после замены HTTPS-сертификата роутера

Закрепление сертификата защищает приложение от подмены, но законная замена разрывает соединение.

**Рекомендация:** автоматическую замену отпечатка не разрешать. Показывать понятную ошибку и требовать нового локального QR-сопряжения либо ручного подтверждения нового отпечатка рядом со старым.

### 32. Нужен ли аварийный откат обновления IPK

**Рекомендация:** перед обновлением сохранять версию пакета, конфиг и последний успешно установленный IPK при наличии места. При провале установки автоматически восстанавливать конфиг; откат бинарного пакета предлагать только когда архив предыдущей версии действительно сохранён. Никогда не обещать откат без проверенного файла.

### 33. Как сообщать о неполной работе AdGuard Home или Podkop

**Рекомендация:** различать `не установлен`, `остановлен`, `API недоступен`, `версия не поддержана` и `работает`. Ошибка интеграции не должна молча переключать владельца правил или перезаписывать конфиги чужого приложения.

## Поведение интерфейса и данных

### 34. Какие действия выполняются сразу, а какие ждут общей кнопки сохранения

**Рекомендация:** команды `Интернет включён/отключён`, `+30 минут`, тест мессенджера, перезагрузка и ручное обновление выполняются отдельно с подтверждением. Редактирование настроек, групп, расписаний и источников остаётся черновиком до `Сохранить настройки`. Добавление в белый/чёрный список устройств сейчас выглядит как отдельное действие; подтвердите, должно ли оно оставаться мгновенным.

### 35. Что показывать при частичной недоступности backend

**Рекомендация:** не заменять реальные данные пустыми таблицами. Показывать последнюю успешно загруженную картину с временем и заметной плашкой `Не удалось обновить`; опасные кнопки блокировать, если текущий результат неизвестен.

### 36. Какой источник главнее при расхождении данных устройства

DHCP-аренда, neighbor-таблица, статическая аренда и ручная карточка могут содержать разные имена и IP.

**Рекомендация:** MAC и числовой ID являются идентичностью; ручное имя родителя главнее DHCP-имени; текущий IP берётся из живого состояния; постоянный IP и имя синхронизируются с DHCP static lease только после явного сохранения.

### 37. Как закрывать административные уведомления

**Рекомендация:** уведомление имеет общий ID события и отдельную отметку просмотра для каждого привязанного телефона. Просмотр одним родителем не должен скрывать событие у второго. На роутере очередь ограничена возрастом и количеством, без хранения текста переписок.

### 38. Что сохранять в журнале отказов доступа к роутеру

**Рекомендация:** время, ID устройства, MAC, текущий IP, назначенное имя, протокол/порт и причина отказа; не сохранять пароль, тело запроса, URL с параметрами и содержимое пакета. Экспорт маскирует MAC/IP, локальный журнал показывает их полностью.

### 39. Какие версии OpenWrt считать поддерживаемыми

**Рекомендация:** зафиксировать актуальную stable-ветку и одну следующую stable-ветку OpenWrt с `firewall4`; старые `iptables`-ветки не поддерживать. Для выпуска нужна хотя бы одна реальная установка на каждой заявленной ветке.

### 40. Какой живой роутер использовать как эталон

**Рекомендация:** Xiaomi Mi Router AX3000T оставить основной моделью, но добавить вторую модель с иным Wi-Fi/LED/button стеком. Это рано выявит аппаратные предположения, которые незаметны в shell-тестах.
