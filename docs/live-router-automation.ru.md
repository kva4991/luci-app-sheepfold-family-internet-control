# Автоматические тесты на живом роутере

<!-- §routerharness -->

Этот контур позволяет агенту определить менеджер пакетов роутера, сделать резервную копию конфигурации, установить подходящий OpenWrt-пакет, проверить backend, временно проверить белый и чёрный списки устройств и открыть LuCI в настоящем браузере. После первоначальной настройки участие пользователя в обычных прогонах не требуется (§pkgmgr1).

Расширенная матрица будущих сетевых, Android- и hardware-in-loop сценариев находится в [`live-router-testing.ru.md`](live-router-testing.ru.md). Этот файл описывает уже исполняемую безопасную основу.

## Первоначальная настройка

Для автоматизации нужен включённый SSH/Dropbear. Скрипт создаёт отдельный ключ `%USERPROFILE%\.ssh\sheepfold_test_router_ed25519`, устанавливает только его открытое содержимое на роутер и проверяет вход без пароля.

```powershell
$testRouterIp = '192.168.X.Y'
npm.cmd run router:setup -- `
  -RouterHost $testRouterIp `
  -InstallPublicKey `
  -SaveLuciCredential
```

Первый запуск может один раз спросить root-пароль роутера для добавления открытого SSH-ключа и отдельно пароль LuCI. Пароль LuCI сохраняется вне репозитория в `%LOCALAPPDATA%\Sheepfold\routerTestCredential.xml`, зашифрованном Windows DPAPI для текущего пользователя. Профиль без секретов лежит рядом в `routerTest.json`.

Если SSH-порт закрыт, скрипт остановится до любых изменений. Один раз включите SSH/Dropbear в LuCI и повторите команду. Пароли нельзя передавать аргументами командной строки, записывать в `.env`, Git или логи.

Другой адрес или порт задаются явно:

```powershell
npm.cmd run router:setup -- `
  -RouterHost $testRouterIp `
  -SshPort 22 `
  -HttpsPort 443 `
  -AppPort 5201 `
  -InstallPublicKey `
  -SaveLuciCredential
```

Сценарий принимает только буквальный частный IPv4-адрес. Это защищает от случайной установки тестового пакета на публичный сервер или ошибочный hostname.

## План первого прогона после включения роутера

1. Выполнить `router:setup` с явным IP тестового роутера, `-InstallPublicKey -SaveLuciCredential`. Критерий: повторный SSH-вход проходит в `BatchMode` без prompt.
2. Запустить `router:readOnly`. Критерий: определены OpenWrt/model/package, обязательные UCI-секции, executable-права, `router-info`, IPv6 status и `fw4 check`; состояние не изменено.
3. Просмотреть `run.log`. В нём не должно быть токенов, паролей, полного UCI, Wi-Fi credentials или клиентских MAC из конфигурации.
4. Подготовить пакет требуемого формата и запустить `router:install`. Критерий: внешний backup получен до системного менеджера пакетов, SHA-256 совпал, новая версия установлена, существующий `/etc/config/sheepfold` не потерян, post-install read-only проверки зелёные.
5. Запустить `router:writeSafe`. Критерий: фиктивный MAC по очереди попадает в белый и чёрный списки устройств, противоположный список отклоняется, после теста MAC отсутствует и UCI совпадает с backup.
6. Запустить `router:frontend`. Критерий: LuCI открывается без `RPCError`/`TypeError`; все верхние вкладки и обе строки вкладок настроек переключаются; модель и прошивка на странице «Информация» не пусты и не равны `unknown`; desktop/mobile не имеют горизонтального переполнения; запросы и console чистые.
7. После успешных отдельных этапов использовать `router:allSafe` как обычную автономную проверку перед изменениями IPK/backend/LuCI.

При любой ошибке следующий этап не запускается. Сначала изучаются `run.log`, `result.json` и скриншоты. Hardware-сценарии не добавляются в первый прогон: даже тестовый роутер не следует перезагружать или лишать сети, пока backup/restore основного контура не подтверждён практикой.

## Команды

```powershell
# Только чтение состояния роутера и Sheepfold
npm.cmd run router:readOnly

# Сборка/выбор пакета, backup, установка и read-only проверки
npm.cmd run router:install

# Временная проверка списков устройств с восстановлением UCI
npm.cmd run router:writeSafe

# Сборка, установка, read-only и восстанавливаемые backend-проверки
npm.cmd run router:fullSafe

# LuCI в desktop/mobile размерах, без изменения настроек
npm.cmd run router:frontend

# Полный безопасный backend + frontend проход
npm.cmd run router:allSafe
```

Для AI Support:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File tools\router-testing\runAllRouterTests.ps1 `
  -Variant sheepfoldAi
```

На роутере с `opkg` готовый IPK можно передать через `-IpkPath`; без него тестовый IPK собирается автоматически в `.build/live-router-ipk`. На OpenWrt 25.12 с `apk` нужно передать через `-ApkPath` настоящий OpenWrt APK, собранный официальным SDK. Переименование IPK в APK и Android APK не принимаются. Артефакты не копируются в `Downloads`.

## Что выполняется автоматически

1. Проверяется SSH-ключ, частный адрес, `ubus`, `opkg` либо `apk` и свободное место в `/overlay`.
2. Перед любыми изменениями отдельный `routerState.sh` копирует `/etc/config/sheepfold`, `dhcp`, `wireless`, `firewall` и сведения об установленном пакете. Сложная логика backup/restore не передаётся аргументом SSH: это защищает Windows-прогон от ошибок CRLF, кавычек и внешней блокировки длинной команды.
3. Backup скачивается в `.build/live-router/<run>/private`, которому ограничиваются Windows ACL.
4. Перед изменяющим профилем проверяется только количество заполненных чувствительных полей, без чтения их значений в журнал. Если секреты настроены, проход останавливается. Для специально подготовленного тестового роутера это ограничение можно осознанно снять параметром `-AllowConfiguredSecrets`.
5. Пакет передаётся legacy SCP в уникальный каталог `/tmp`, а его SHA-256 сверяется до `opkg install` либо `apk add --allow-untrusted`.
6. Выполняются проверки UCI, прав executable-файлов, `router-info`, `ui_asset_version`, root-пароля, IPv6 helper и `fw4 check`.
7. Write-проверка использует только фиктивный locally administered MAC `02:53:48:45:45:50`. При любом выходе точный конфиг и RAM-журнал восстанавливаются.
8. Playwright Core запускает уже установленный Chrome или Edge в отдельном headless-контексте. Проверяются desktop/mobile, все вкладки, модель/прошивка роутера, `RPCError`, `TypeError`, ошибки запросов/консоли и горизонтальное переполнение; сохраняются обычные и failure-скриншоты.
9. Временные файлы на роутере удаляются после прохода.

`playwright-core@1.61.1` устанавливается автоматически в игнорируемый `tools/local/routerFrontend`. Отдельный браузер не скачивается. Самоподписанный сертификат LuCI принимается только тестовым browser context через `ignoreHTTPSErrors`; системные настройки Chrome/Edge не ослабляются.

## Что безопасный контур не делает

- не выполняет массовые `opkg upgrade`/`apk upgrade`, `sysupgrade`, `firstboot` или сброс overlay;
- не перезагружает роутер и сетевые службы;
- не выключает Wi-Fi, WAN или Ethernet;
- не запускает WPS и не управляет светодиодами;
- не меняет SSID, пароли Wi-Fi, Telegram/VK/AI-токены;
- не тестирует правила на реальном семейном MAC;
- не делает вид, что backup UCI является бинарным rollback старой версии пакета.

Если post-install проверка не проходит, конфиги восстанавливаются. Для полноценного отката исполняемых файлов нужен пакет предыдущей версии того же формата; поэтому обновление бинарников и hardware-тесты остаются отдельными явно запускаемыми уровнями.

## Артефакты и приватность

Отчёты находятся в `.build/live-router/<run>/`. `run.log` не должен содержать секреты. Каталог `private` может содержать конфигурационный backup с паролями и токенами. Browser-каталог может содержать failure-скриншот с SSID или заполненным полем. Оба каталога игнорируются Git и получают ограниченные Windows ACL. Их нельзя прикладывать к issue или отправлять разработчику без ручной проверки и очистки.

## Работа агента

Новый агент на Windows сначала читает этот файл и запускает `npm.cmd run router:readOnly`. Перед изменением backend/UCI/пакетирования он использует `router:fullSafe`; перед завершением пользовательской LuCI-правки дополнительно запускает `router:frontend`. После однократной установки ключа и DPAPI-учётных данных агент не просит пользователя вручную копировать пакет, выполнять SSH-команды или делать скриншоты.
