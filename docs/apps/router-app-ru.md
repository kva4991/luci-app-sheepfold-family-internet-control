# Sheepfold на роутере (LuCI / OpenWrt)

## 1. Назначение

Sheepfold на роутере — это основное серверное приложение семейного контроля интернета, устанавливаемое как пакет `luci-app-sheepfold-family-internet-control` на роутеры OpenWrt.[cite:80] Оно управляет правилами доступа к сети для домашних устройств, собирает телеметрию, предоставляет веб-интерфейс (LuCI) и API для мобильных приложений и ботов.

## 2. Целевая аудитория

- Родители и опекуны, управляющие доступом детей к интернету.
- Администраторы домашней сети, которым нужно контролировать устройства, сайты и расписания.
- Дизайнеры интерфейсов LuCI, работающие над UX панели управления.
- Тестировщики, проверяющие корректность работы правил, интерфейса и интеграций.

## 3. Основные роли пользователей

- **Владелец роутера (SuperParent/Owner)** — имеет полный доступ, настраивает глобальные правила, pairing и токены.[cite:58]
- **Другие администраторы** — могут управлять устройствами и списками, но с ограничениями.
- **Детские устройства** — не имеют прямого доступа к LuCI, но их поведение контролируется.

## 4. Ключевые подсистемы

1. **Router Control** — центральный диспетчер команд (`sheepfold-router-control` и `sheepfold-router-control-legacy`):
   - управление списками устройств (allowlist/blocklist);
   - временный доступ и глобальная блокировка интернета;
   - Wi-Fi и WPS-окна;
   - pairing администраторов и токены.[cite:58]
2. **Service-оркестратор** (`sheepfold-service`):
   - реагирует на изменения DHCP/ARP;
   - запускает `device-detector`, `device-presence`;
   - обрабатывает запросы на обновление и перезагрузку.[cite:57]
3. **Device-пайплайн**:
   - `sheepfold-device-detector` — сбор сигналов о устройствах, скан портов, mDNS, формирование TSV и UCI-конфигов.[cite:61]
   - `sheepfold-device-classifier` — определение типа устройства (сетевое оборудование, смарт-гейтвей, обычный клиент).[cite:57]
   - `sheepfold-device-presence` — онлайн/офлайн статус устройств.[cite:61]
4. **API и CGI**:
   - `sheepfold-api-legacy` — старый API для панели и внешних клиентов.[cite:60]
   - `sheepfold-api-client-status` — лёгкий endpoint для детского apk.[cite:60]
   - `sheepfold-api-pair` — endpoint для pairing администраторов.[cite:60]
5. **Интеграции**:
   - Telegram/VK-боты (см. отдельный файл);[cite:57]
   - списки сайтов (`sheepfold-site-lists`);
   - USB-хранилище (`sheepfold-usb-storage`);
   - обновления (`sheepfold-updater`).[cite:57]

## 5. Для дизайнеров

### 5.1. Основные экраны LuCI

- Экран обзора сети: список устройств, статус онлайн/офлайн, группы и политики.
- Экран правил: глобальная блокировка, расписания, списки сайтов.
- Экран pairing и токенов: привязка администратора к устройству, управление ключами.
- Экран настроек AI и журналов (если включены соответствующие функции).[cite:80]

При проектировании важно:

- Чётко визуализировать статус устройств (доверенные, ограниченные, заблокированные).
- Показывать причину блокировки (правило, расписание, категория сайта).
- Обозначать безопасные действия (например, временный доступ) отдельными цветами/иконками.

### 5.2. Визуальные паттерны

- Цветовая дифференциация групп (например, "Без ограничений" как отдельная категория).[cite:61]
- Иконки для типов устройств (роутер, смартфон, ТВ, смарт-гейтвей).
- Логичное расположение элементов для быстрого доступа к критичным действиям (глобальная блокировка, снятие временного доступа).

### 5.3. Карта экранов и функций

1. **Главный экран (Dashboard)**
   - Карточка общего состояния интернета (онлайн/limited/offline).
   - Счётчик подключённых устройств и новых устройств с алертами.
   - Быстрые действия: глобальная блокировка, включение/выключение Wi-Fi, открытие WPS-окна.

2. **Экран устройств (Devices)**
   - Таблица/список устройств с колонками: имя, тип, группа, статус онлайн, политика.[cite:61]
   - Фильтры: по группе (дети, родители, "Без ограничений"), по статусу (онлайн/офлайн), по блокировке.
   - Детальная карточка устройства: история, текущие ограничения, временный доступ, кнопки allow/block/temp-access.[cite:58]

3. **Экран правил (Policies)**
   - Глобальные настройки интернета (block_on_boot, расписания).[cite:58]
   - Списки сайтов: allowlist/blocklist доменов с возможностью поиска и импорта.[cite:57]
   - Настройки расписаний по группам/устройствам (если реализовано).

4. **Экран pairing и токенов (Pairing / Tokens)**
   - Форма выдачи pairing-кода для нового администратора.[cite:58]
   - Список действующих токенов с информацией об устройствах и сроках.
   - Кнопки revoke для токенов.

5. **Экран журналов и AI (Logs / AI)**
   - Просмотр событий (подключение устройств, блокировки, изменения правил).
   - Настройки интеграции с AI (если включено): включение, режимы, предупреждения в UI.[cite:80]

## 6. Для тестировщиков

### 6.1. Основные сценарии

- Добавление нового устройства и автоматическое создание UCI-секции.[cite:61]
- Переключение устройства между allowlist/blocklist и проверка эффектов.
- Настройка временного доступа и проверка его окончания.
- Проверка WAN-диагностики (онлайн/limited/offline) при разных состояниях кабеля и провайдера.[cite:58]
- Проверка работы API-эндпоинтов (legacy, client-status, pairing) на корректный JSON/HTTP.[cite:60]

### 6.2. Негативные кейсы

- Неизвестный MAC; MAC из blocklist; истёкший pairing-код; ошибка записи токена.[cite:58]
- Отсутствие OpenSSL, readonly filesystem, неверный block device для USB.[cite:57]

### 6.3. Карта тестовых сценариев по экранам

1. **Dashboard**
   - Проверка корректной индикации WAN-состояния при смене статуса (поднят, нет линка, нет связи вовне).[cite:58]
   - Проверка работы быстрых действий: глобальная блокировка, Wi-Fi on/off, WPS-окно.

2. **Devices**
   - Добавление/удаление устройства через реальные DHCP/ARP события и через UI.
   - Перевод устройства между группами, проверка изменений в UCI и фактического поведения.
   - Проверка временного доступа: начало, продолжение, автоматическое окончание.

3. **Policies**
   - Настройка списков сайтов и проверка их применения (блокировка/разрешение доменов).
   - Настройка расписаний, проверка смены политики по времени.

4. **Pairing / Tokens**
   - Issuance и использование pairing-кодов; проверка отказов для неправильных/истёкших/повторных кодов.[cite:58]
   - Создание/удаление токенов и проверка доступа через API.

5. **Logs / AI**
   - Проверка корректного протоколирования событий.
   - Проверка UI-отображения предупреждений AI и реакций на отключение/включение.

## 7. Связь с другими приложениями

- Android-приложения используют роутерный API для получения статусов и управления.[cite:80]
- Telegram/VK-боты вызывают команды контроллера для управления из мессенджеров.[cite:57]
