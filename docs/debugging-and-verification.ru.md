# Диагностика и доказательное завершение

<!-- §debug01 -->

Этот контракт применяется к дефектам, нестабильным тестам и сообщениям вида «в интерфейсе сохранено, но после обновления исчезло». Его цель — исправлять первопричину и не выдавать предположение за проверенный результат.

## Минимальная карточка дефекта

До правки зафиксировать:

1. наблюдаемое поведение и точные шаги воспроизведения;
2. ожидаемое поведение со ссылкой на ADR, профильный документ или подтверждение владельца;
3. слой, где расхождение уже доказано: LuCI, Android, HTTP/API, router helper, UCI, runtime firewall/DNS или пакет;
4. свежий артефакт: error code, JSON, UCI после reload, journal line, screenshot либо test failure;
5. последнее известное рабочее поведение или близкий рабочий путь в проекте.

## Последовательность расследования

1. **Воспроизвести.** Если воспроизведение невозможно, сначала добавить безопасную диагностику, а не менять поведение наугад.
2. **Проследить данные через границы.** Сравнить, что отправил UI, что принял API, что записано в UCI и что применил runtime.
3. **Сформулировать одну проверяемую гипотезу.** Например: «backend пишет anonymous section, а reader ожидает named section».
4. **Проверить гипотезу минимальным наблюдением.** Не смешивать одновременно несколько возможных исправлений.
5. **Исправить первопричину и добавить regression test** на самом низком уровне, который действительно воспроизводит риск.
6. **Перепроверить соседний рабочий сценарий** и затем применимую тестовую категорию.

## Полезные границы Sheepfold

| Визуальный симптом | Что проверить следующим |
|---|---|
| Toast «сохранено», после reload старое значение | HTTP response, named UCI section, `uci commit`, повторное чтение backend |
| UCI изменён, сеть ведёт себя по-старому | runtime apply, lock, nftables/dnsmasq/AdGuard state |
| Локальный код исправлен, роутер показывает старое | версия установленного пакета и `?v=PKG_VERSION-PKG_RELEASE` |
| Android пишет timeout | достигнут ли endpoint, истёк ли CGI timeout, был ли commit завершён до фонового firewall sync |
| Статический тест зелёный, реальный роутер падает | BusyBox/UCI/uhttpd semantics и наличие необязательных утилит |
| Browser test падает только на mobile | overflow, размер цели, доступное имя icon-button, текст в контейнере |

## Доказательство завершения

Нельзя писать «исправлено» только потому, что код выглядит правильно или один старый тест когда-то проходил. Перед завершением нужны свежие результаты:

- ближайший regression test;
- все категории из [`change-impact-review.ru.md`](change-impact-review.ru.md);
- `npm.cmd run lint:js`, если изменён JS;
- `git diff --check`;
- live-router/Android/browser проверка, если дефект зависит от соответствующего runtime.

В отчёте отдельно перечислять, что проверено и что осталось непроверенным. Успешная UI-всплывашка не доказывает UCI commit, зелёный статический тест не доказывает OpenWrt runtime, а screenshot не доказывает backend.

## Запрещённые сокращения пути

- Не ослаблять validation только ради прохождения корявого входа.
- Не добавлять silent fallback с привилегированным или разрешающим значением.
- Не чинить timeout увеличением ожидания, пока неизвестно, где задержка.
- Не создавать второй helper с почти той же ответственностью вместо исправления общего.
- Не объявлять внешний инструмент обязательным, пока его уникальная польза не подтверждена на Sheepfold.
