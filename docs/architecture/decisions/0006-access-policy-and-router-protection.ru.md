# ADR-0006: Приоритет доступа отделён от безусловной защиты роутера

- Статус: Принято
- Дата: 2026-07-20
- Теги: `§84azytj`, `§h6mxq4c`, `§w8dm3kv`, `§x1hy6pz`
- Связанные документы: [Backend](../../backend-design.ru.md), [Расписания](../../schedules.ru.md)

## Контекст

Белый/чёрный список устройств, группы, расписания, временный доступ и глобальное отключение могут действовать одновременно. Перестановка бытовых правил не должна случайно открыть LuCI, SSH или API подозрительному устройству.

## Рассмотренные варианты

1. Набор независимых firewall-исключений — трудно объяснять и легко получить противоречие.
2. Один редактируемый список всех правил, включая защиту роутера — опасно.
3. Явный internet-приоритет и отдельный защитный overlay — выбранный вариант.

## Решение

Internet-политика вычисляется единым evaluator в согласованном порядке. Чёрный список устройств и нерешённое несовпадение baseline всегда закрывают управление роутером. Настройка ручного/автоматического мониторинга влияет на интернет подозрительного подключения, но не ослабляет эту защиту. Аварийно-полезные сайты не открывают административные интерфейсы.

## Последствия

- Любой редактор приоритета обязан проверять полный список без дублей.
- Status API и nftables должны объяснять один и тот же итог.
- В документации всегда уточняется: список устройств или список сайтов.

## Проверка

`firewallIntegration.test.mjs`, `schedulePriorityUi.test.mjs`, `routerBackendAccessRules.test.mjs` и live-router nftables matrix.

## Когда пересматривать

При добавлении нового вида доступа, который нельзя выразить существующим evaluator и overlay без скрытого исключения.
