# ADR-0011: Локальные детерминированные quality gates

- Статус: Принято
- Дата: 2026-07-20
- Теги: `§qassist`, `§impact1`, `§testcat`
- Связанные документы: [`quality-assistants/README.ru.md`](../../quality-assistants/README.ru.md), [`test-strategy.ru.md`](../../test-strategy.ru.md)

## Контекст

Sheepfold объединяет LuCI, OpenWrt shell/ucode, UCI, firewall/DNS, два Android-приложения, package formats и живой тестовый роутер. Полный suite дорог, а выбор тестов на глаз пропускает соседние границы. Внешние AI-сканеры и большие skill-коллекции требуют сети, аккаунта, непроверяемых эвристик либо добавляют ещё один источник правил.

## Рассмотренные варианты

1. Установить один или несколько внешних vibe/AI-slop scanners и следовать их score. Отклонено: уникальная польза для этого смешанного OpenWrt-проекта не доказана, правила непрозрачны, а результаты могут меняться вне репозитория.
2. Всегда запускать только полный suite. Отклонено для рабочего цикла: сетевые и package-тесты идут долго, а частая обратная связь становится слишком дорогой.
3. Хранить в репозитории точную карту влияния, быстрый объединённый запуск и отдельный строгий gate. Принято.

## Решение

Sheepfold использует локальный детерминированный контур:

- Git diff сохраняет status, untracked и обе стороны rename;
- точные path rules назначают риск, тестовые категории и внешние проверки;
- изменённые test-файлы запускаются напрямую, категории объединяются без дублей;
- доказанная правка только `PKG_RELEASE` получает узкий package-контур, но любая другая или нераспознанная правка `Makefile` остаётся в полном security-контуре;
- документация, whitespace и структура проверяются отдельными лёгкими модулями;
- `quality:changed` даёт быструю обратную связь;
- `quality:gate` перед push запускает строгий полный локальный suite;
- живой роутер, Android build и GitHub SDK остаются явными отдельными действиями;
- all-pairs сокращает только обычные конфигурационные сочетания;
- внешняя LLM не получает исходники и не определяет зелёный/красный статус.

## Последствия

- Новый агент получает одну точку входа и машиночитаемый отчёт.
- Ложные совпадения и случайный запуск тяжёлых тестов должны иметь regression test.
- Content-aware исключение допустимо только с чистой функцией сравнения, консервативным fallback и сохранением обязательного полного gate.
- Карта требует сопровождения при появлении нового каталога или контрактной границы.
- `passed` локального gate не заменяет ручные проверки из `pendingManualChecks`.
- Advisory структурного аудита нельзя превращать в механический запрет больших файлов.

## Проверка

`tests/changeImpact.test.mjs`, `tests/qualityAssistants.test.mjs`, `tests/testCategories.test.mjs`, `tests/runtimeCompatibilityMatrix.test.mjs` и `tests/liveRouterHarness.test.mjs` защищают отдельные модули. `quality:docs:all`, `quality:changed`, ESLint, полный suite и живой `router:frontend` проверяют совместную работу. Они не доказывают hardware, release SDK и Android lifecycle без соответствующих внешних прогонов.

## Когда пересматривать

Если закреплённый внешний инструмент стабильно обнаружит важный класс дефектов, который нельзя разумно закрыть ESLint, тестом или текущим harness, его можно рассмотреть отдельно. Нужны воспроизводимый пример, измеренная стоимость, pinned version, режим без отправки исходников и понятный план удаления.
