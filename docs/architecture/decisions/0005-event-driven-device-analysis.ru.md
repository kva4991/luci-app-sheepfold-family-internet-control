# ADR-0005: Анализ устройств событийный и ограниченный по ресурсам

- Статус: Принято
- Дата: 2026-07-20
- Теги: `§detlife1`, `§d7vqm4s`, `§e8kxtjc`
- Связанные документы: [Паспорт устройства](../../device-passport-and-control.ru.md), [Автоопределение](../../device-detection.ru.md)

## Контекст

Домашний роутер слабее сервера, а постоянный nmap/mDNS/SSDP-поиск расходует CPU, RAM и эфир. Только DHCP lease не доказывает, что устройство сейчас online.

## Рассмотренные варианты

1. Полный проход каждые 15 минут — лишняя нагрузка и повторные ложные события.
2. Только одно событие DHCP — пропускает статические IP и потерянные события.
3. Offline→online плюс редкий online-only проход — выбранный вариант.

## Решение

Анализ запускается после подтверждённого offline→online с задержкой, один раз после старта, раз в сутки только для online-устройств и вручную. Лёгкие признаки собираются первыми; bounded port scan нужен только новому или неопределённому устройству. Offline и чёрный список устройств не сканируются. Все входы сериализованы одним lock.

## Последствия

- Flapping подавляется grace-окном.
- Низкая уверенность повторяется не чаще раза в сутки или при следующем подключении.
- Живой роутер нужен для оценки нагрузки и пропущенных событий.

## Проверка

`deviceDetectionService.test.mjs`, `devicePresence.test.mjs`, `deviceDetectorSafety.test.mjs` и live-router наблюдение CPU/RAM.

## Когда пересматривать

Если целевые роутеры заметно изменятся по мощности либо появится надёжный OpenWrt event API, устраняющий страховочный проход.
