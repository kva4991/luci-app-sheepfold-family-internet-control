# ADR-0003: UCI хранит настройки, а немедленные команды применяются атомарно

- Статус: Принято
- Дата: 2026-07-20
- Теги: `§m9qe4lk`, `§listnoref`, `§lstxcl1`
- Связанные документы: [Миграции UCI](../../uci-config-migration.ru.md), [Backend](../../backend-design.ru.md)

## Контекст

LuCI умеет копить неприменённые изменения, но обычному родителю непонятна разница между «Сохранить» и «Применить». Одновременно добавление устройства в список является командой с немедленным сетевым эффектом, а не черновиком большой формы.

## Рассмотренные варианты

1. Полагаться на системную плашку LuCI — оставляет конфиг и runtime в разных состояниях.
2. Перезагружать всю страницу после каждой команды — скрывает ошибки и ухудшает UX.
3. Явное сохранение настроек и отдельные атомарные команды — выбранный вариант.

## Решение

Настройки сохраняются общей кнопкой, проходят backend validation, UCI commit и runtime apply без системного хвоста. Немедленные действия, включая белый/чёрный список устройств, выполняются защищённой backend-командой и обновляют только нужную frontend-модель. Конфликт списков отклоняется; автоматического переноса нет.

## Последствия

- Toast об успехе допустим только после перечитывания результата.
- Миграции дописывают недостающие поля и никогда не заменяют живой конфиг целиком.
- Команда и настройка должны быть явно различимы в API и тестах.

## Проверка

`allowlistUi.test.mjs`, `routerBackendAccessRules.test.mjs`, `settingsBackup.test.mjs` и проверка UCI на живом роутере.

## Когда пересматривать

Если LuCI предоставит единый транзакционный API, который гарантированно возвращает подтверждённое runtime-состояние без pending changes.
