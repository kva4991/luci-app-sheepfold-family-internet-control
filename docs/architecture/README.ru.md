# Архитектура Sheepfold

<!-- §adrproc -->

Эта папка помогает отличать три уровня документации:

- [`decisions/`](decisions/README.ru.md) отвечает на вопрос «почему выбрали именно так»;
- профильные документы в `docs/` подробно описывают текущее устройство подсистемы;
- [`../dev/tag-map.md`](../dev/tag-map.md) связывает одно решение с кодом, тестами и другими документами через §-теги.

ADR не заменяет техническое задание и не пересказывает весь код. Он фиксирует устойчивое решение, рассмотренные альтернативы, последствия и условие пересмотра. Если реализация расходится с принятым ADR, нельзя молча исправлять только документ или только код: сначала надо подтвердить фактическое поведение тестами и решить, что именно устарело.

## Порядок изменения архитектуры

1. Найти профильный документ и связанные §-теги.
2. Проверить, существует ли ADR и действует ли он.
3. Если решение меняется, создать новый ADR со статусом `Предложено` либо пометить старый как заменённый новым. Не переписывать историю так, будто прежнего решения не было.
4. После согласования обновить код, профильные документы, карту тегов и проверки в одной логической правке.
5. Выполнить советник влияния и применимые тестовые категории.
