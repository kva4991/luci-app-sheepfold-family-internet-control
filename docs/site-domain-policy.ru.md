# Применение белых и чёрных списков сайтов

<!-- §dompol -->

Этот документ описывает уже реализованный runtime-путь от загруженных доменов до правил DNS и firewall. Белый и чёрный списки **сайтов** нельзя смешивать с белым и чёрным списками **устройств**: это разные уровни политики.

## Компоненты

- `sheepfold-site-lists` скачивает, проверяет, нормализует и объединяет источники;
- `/tmp/sheepfold/site-lists/allowlist.domains` содержит итоговый белый список сайтов;
- `/tmp/sheepfold/site-lists/blocklist.domains` содержит итоговый чёрный список сайтов;
- `sheepfold-domain-policy` выбирает исполнителя, управляет fallback и хранит подтверждённый runtime-статус;
- `sheepfold-adguard` формирует одну принадлежащую Sheepfold ленту и управляет ею через API AdGuard Home;
- `/cgi-bin/sheepfold-adguard-list` отдаёт эту ленту только локальному AdGuard Home по случайному токену;
- `sheepfold-firewall` заполняет только собственные наборы `sheepfold_*`;
- `30-sheepfold.nft` применяет строгий белый список сайтов групп и режим чёрного списка сайтов.

## Топология и исполнитель фильтрации

`integration_mode` описывает установленную цепочку и не заменяет выбор исполнителя списков:

| `integration_mode` | Цепочка после решения Sheepfold по устройству |
| --- | --- |
| `none` | Обычная сеть OpenWrt. |
| `podkop` | Podkop маршрутизирует уже разрешённый трафик; его таблицы, маршруты, метки и sing-box не изменяются. |
| `adguard` | DNS-запросы проходят через AdGuard Home. |
| `adguard_podkop` | AdGuard Home фильтрует DNS, затем Podkop маршрутизирует разрешённый трафик. |

Настройка `site_filter_backend` в `Настройки -> Интеграции -> Фильтрация сайтов` имеет три значения:

| Значение | Поведение |
| --- | --- |
| `auto` | Рекомендуемый режим. Если цепочка содержит AdGuard Home, автоматическое управление включено и API подтвердил фильтр Sheepfold, используется AdGuard Home. В остальных случаях включается встроенная фильтрация Sheepfold. |
| `adguard` | Запрашивается AdGuard Home. Если он не выбран в цепочке, API недоступен или фильтр не подтверждён, Sheepfold применяет встроенный безопасный fallback и показывает причину. |
| `sheepfold` | Всегда используются dnsmasq `nftset` и собственные nftables-наборы Sheepfold. |

`adguard_auto_manage=1` включено по умолчанию. Sheepfold создаёт и обновляет только URL-фильтр `Sheepfold family site policy`; пользовательские фильтры, custom rules, YAML и общие настройки AdGuard Home не перезаписываются. При выключенном автоматическом управлении режим AdGuard считается ручным и непроверенным: LuCI показывает жёлтый статус, а пользователь сам отвечает за настройку списков в AdGuard Home.

## Безопасное применение

`sheepfold-domain-policy` сериализует параллельные запуски. Встроенное применение выполняется как одна транзакция:

1. Проверяет настройки, наличие подготовленных списков, поддержку `nftset` установленным dnsmasq и ограничения размера.
2. Создаёт новый конфиг во временном файле и запускает `dnsmasq --test`.
3. Атомарно заменяет рабочий файл и перезапускает dnsmasq.
4. Записывает состояние готовности, после чего синхронизирует доменные наборы firewall.
5. Если перезапуск DNS или синхронизация firewall не удались, восстанавливает прежний dnsmasq-конфиг, прежнее состояние готовности и прежние nftables-наборы.

Для AdGuard Home адаптер сначала строит временную ленту, проверяет размер и наличие текущих IPv4-адресов клиентов, читает `/control/filtering/status`, добавляет или включает только свой URL-фильтр, обновляет его и повторно подтверждает состояние API. Активная лента заменяется атомарно; при ошибке предыдущая рабочая лента возвращается. Локальный dnsmasq-конфиг удаляется только после подтверждения AdGuard Home. При ошибке API включается встроенный fallback.

Файл общего состояния находится в `/tmp/sheepfold/domain-policy/status`, подтверждённое локальное состояние — в `/tmp/sheepfold/domain-policy/active`, состояние адаптера — в `/tmp/sheepfold/adguard/status`.

## Последняя рабочая копия и отказ без ложной блокировки

- Ошибка отдельного внешнего источника не заменяет его последнюю рабочую копию (§slstres).
- Пока ни одного проверенного белого списка сайтов нет, строгая группа остаётся в интернете. Sheepfold не должен случайно закрыть весь доступ пустым или повреждённым файлом.
- Если dnsmasq не поддерживает `nftset`, локальная фильтрация списков сайтов остаётся выключенной, а администратор получает одно понятное уведомление.
- Ошибка применения не должна оставлять новый DNS-конфиг со старыми nftables-наборами или наоборот.
- Ошибка AdGuard Home не оставляет интерфейс с ложной зелёной плашкой: LuCI показывает причину и фактически выбранный fallback.
- При переключении обратно на встроенный backend Sheepfold отключает только собственный фильтр AdGuard Home. Если API не позволил это подтвердить, локальная фильтрация включается, но LuCI предупреждает о возможном двойном применении.
- Чёрный список устройств остаётся сильнее автоматических назначений. Аварийно-полезные сайты могут открыть только разрешённый web-трафик и не открывают LuCI, SSH или Sheepfold API (§emerg1).
- Исключение устройства из чёрного списка сайтов не является исключением из глобального отключения интернета.

## Что именно фильтруется

Строгий белый список сайтов для группы разрешает web-доступ к IPv4/IPv6-адресам, которые локальный DNS получил для разрешённых доменов, и блокирует остальной пересылаемый трафик этой группы. Чёрный список сайтов блокирует TCP `80/443` и UDP `443` к адресам запрещённых доменов в зависимости от выбранного режима.

Технические домены входа, изображений, карт, CDN и API часто отличаются от видимого адреса сайта. Родителю может потребоваться разрешить несколько узких доменов, чтобы нужный сервис работал полностью.

## Честные ограничения

- Это доменная DNS-фильтрация, а не анализ URL, пути страницы, текста, названия видео, комментариев или содержимого приложения.
- Внешний DNS, DNS-over-HTTPS, VPN, прокси и прямое подключение по IP могут обходить доменное решение. Их отдельная блокировка сейчас не заявляется.
- Один IP может обслуживать много доменов. Блокировка общего CDN-адреса способна затронуть соседний сайт, а смена адресов требует нового DNS-ответа.
- Чёрный список сайтов сейчас предназначен для web-трафика. Он не обещает блокировку любого собственного протокола приложения на произвольном порту.
- Sheepfold не заменяет dnsmasq на `dnsmasq-full` автоматически: смена базового DNS-пакета может нарушить сеть конкретного роутера.
- Клиентские правила AdGuard Home привязаны к наблюдаемому роутером IPv4-адресу. MAC как идентификатор доступен AdGuard Home только когда он сам является DHCP-сервером, поэтому Sheepfold обновляет свою ленту после изменения DHCP-аренд. Устройство без актуального IPv4 не считается успешно защищённым через этот backend.
- Для AdGuard Home на этом же роутере разрешён локальный HTTP loopback. Для API на другом устройстве требуется HTTPS, чтобы логин и пароль не уходили по сети открытым текстом.
- Локальная лента AdGuard Home зависит от HTTP CGI uhttpd. Если AdGuard Home не может получить её с loopback-адреса, Sheepfold сохраняет встроенный fallback.
- Локальные тесты проверяют транзакции и формируемые наборы, но реальное поведение dnsmasq, IPv4/IPv6, CDN и всех четырёх режимов обязательно проверяется на тестовом OpenWrt-роутере.

## Состояние в LuCI

В `Настройки -> Интеграции -> Фильтрация сайтов` справа от выбора показывается лампа и фактическое состояние: выключено, ожидается первая загрузка, подтверждено AdGuard Home, применено Sheepfold, применён fallback, требуется ручная проверка либо ошибка с сохранением прежней рабочей конфигурации. В `Настройки -> Разное -> Источники списков сайтов` остаются сами URL источников и период их обновления. Количество доменов и ошибки скачивания не подменяют runtime-статус применения.

API и синтаксис сверяются с официальными материалами: [OpenAPI AdGuard Home](https://github.com/AdguardTeam/AdGuardHome/blob/master/openapi/openapi.yaml), [правила `$client`](https://github.com/AdguardTeam/AdGuardHome/wiki/Hosts-Blocklists) и [идентификация клиентов](https://github.com/AdguardTeam/AdGuardHome/wiki/Clients).
