# Контракт API Sheepfold

<!-- §apicon1 -->

Этот документ задаёт общие правила router API. Точный список endpoint и полей находится в [`backend-design.ru.md`](backend-design.ru.md) и [`android-openwrt-api.ru.md`](android-openwrt-api.ru.md).

## Владелец истины

- Router backend принимает решение и перечитывает сохранённое состояние до ответа об успехе.
- LuCI, Android и бот являются клиентами: они не имитируют UCI commit, сжигание кода, firewall apply или проверку прав локальным состоянием.
- `client-status` является узким исключением без Bearer-токена и возвращает только статус текущего LAN-клиента, определённого роутером.

## Ответы и ошибки

- Успех и ошибка возвращаются структурированным JSON, а не человекочитаемой строкой, которую клиент должен разбирать.
- Стабильный машинный `errorCode` отделён от локализуемого сообщения. Новые клиенты показывают понятный текст; старые получают безопасный общий fallback.
- HTTP status соответствует классу ошибки: authentication не маскируется под `200`, временная server error не выглядит как окончательно отозванный token.
- Отсутствующее новое необязательное поле трактуется безопасно. Неизвестное поле игнорируется. Изменение смысла существующего поля требует новой версии либо capability.
- Повтор запроса разрешён только для доказанно идемпотентной операции. Pairing code, reboot и неоднозначно завершившаяся команда автоматически не повторяются.

## Безопасные defaults

- Неизвестная capability равна `false`.
- Неизвестная роль не получает права.
- Ошибка проверки TLS identity останавливает запрос.
- Временный timeout не удаляет рабочий token и не превращается в «доступ отозван».
- Ответ не содержит root-пароль, provider keys, bot token, session cookie или полный UCI.

## Совместимость

При изменении endpoint сначала обновляются backend-design, Android contract и ADR при изменении архитектуры. Затем добавляются contract/regression tests для старого и нового клиента. Удаление legacy path допускается только после доказанного отсутствия поддерживаемых клиентов.

## Ревью endpoint

Перед merge проверить: authentication, authorization, validation, size/time limits, rate limit, idempotency, atomic persistence, runtime apply, stable error codes, secret-free logs и поведение старого клиента.
