# Приватные логи активности устройств

## Статус реализации

В текущей ветке реализована **низкоуровневая утилита хранения и агрегации**:

```text
/usr/libexec/sheepfold/sheepfold-activity-log
```

Автоматический перехват DNS/HTTP-событий, публичный CGI endpoint `device-insights` и пользовательская форма LuCI пока **не реализованы**. Их нельзя считать доступными функциями пакета до появления отдельного кода, ACL, тестов и явного согласия пользователя.

Функция по умолчанию выключена:

```text
sheepfold.global.private_logs=0
```

## Что уже умеет backend

Утилита поддерживает команды:

```bash
sheepfold-activity-log write <mac> <event_type> <data>
sheepfold-activity-log rotate
sheepfold-activity-log clean <mac>
sheepfold-activity-log clean-all
sheepfold-activity-log enrich <mac>
sheepfold-activity-log status
```

Допустимые типы события:

- `dns`;
- `http_host`;
- `access_change`;
- `connect`;
- `disconnect`.

MAC проверяется в строгом формате `AA:BB:CC:DD:EE:FF`. Непроверенное значение не участвует в имени файла, поэтому команды `write`, `clean` и `enrich` не допускают path traversal.

## Хранение

Текущие данные располагаются в tmpfs:

```text
/tmp/sheepfold/.private/
    aabbccddeeff.log
    archive/
        YYYY-MM-DD.tar.gz
```

Права:

- каталог — `0700`;
- файлы — `0600`.

Данные в RAM исчезают после перезагрузки, если архивы не были отдельно перенесены на настроенный USB-носитель.

### Ограничения размера

Лимиты рассчитываются по доступной памяти роутера:

| Доступная память | Общий лимит | На устройство |
|---|---:|---:|
| от 128 МБ | 12 МБ | 2 МБ |
| 64–127 МБ | 8 МБ | 1,5 МБ |
| меньше 64 МБ | 2 МБ | 256 КБ |

Дополнительно общий лимит не превышает 15% `MemAvailable`. Значения можно переопределить через:

```text
log_max_dir_kb
log_max_device_kb
```

## Формат и агрегация

Строка сырого файла:

```text
<ISO8601>\t<MAC>\t<event_type>\t<data>
```

Команда `enrich` не возвращает весь сырой файл. Она формирует ограниченную сводку:

- до 50 доменов;
- количество событий;
- первый и последний час наблюдения;
- агрегированные изменения доступа;
- число исходных строк.

Кавычки, обратные слеши и управляющие переводы строк удаляются до записи, чтобы содержимое события не могло сломать выдаваемый JSON.

## Архивация

`rotate` упаковывает текущие `.log` в дневной архив и удаляет исходные файлы только после успешного создания архива.

В репозитории добавлены задания в `root/etc/cron.d/`, однако для целевых версий OpenWrt ещё необходимо подтвердить, что выбранный `crond` читает этот каталог. До такой проверки автоматический ночной запуск нельзя считать гарантированным. Надёжный вариант установки должен создавать управляемый блок в `/etc/crontabs/root` и удалять его при деинсталляции.

## Безопасность и приватность

- сбор выключен по умолчанию;
- включение должно требовать явного действия администратора;
- сырой журнал нельзя выдавать обычному LAN-клиенту;
- будущий API обязан требовать действительный Bearer-токен администратора;
- нельзя использовать логи для скрытого наблюдения или автоматически строить «психологический профиль»;
- перед передачей любой сводки внешнему AI требуется отдельное информированное подтверждение;
- очистка данных должна быть доступна без обращения к облачному сервису.

## Что ещё требуется реализовать

1. Источник событий с чёткой моделью согласия и фильтрацией до записи.
2. LuCI-настройка включения, срока хранения и очистки.
3. Аутентифицированный backend endpoint, если он действительно нужен продукту.
4. Тесты на маскирование, JSON-экранирование, path traversal и ограничения размера.
5. Проверенная интеграция с cron OpenWrt.
6. Пользовательское объяснение того, какие данные собираются и зачем.

Документ не является обещанием наличия DNS-перехвата или API в текущем пакете.
