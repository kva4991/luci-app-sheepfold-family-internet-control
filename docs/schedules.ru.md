# Расписания доступа

Расписания — один из основных сценариев Sheepfold.

Интерфейс должен быть понятным родителям и похожим по духу на простые правила доступа: выбрать устройство, дни недели и время.

## Основные возможности

Правило расписания должно включать:

- название правила;
- список устройств или группу устройств;
- дни недели;
- время начала;
- время окончания;
- действие: разрешить или заблокировать;
- состояние: включено/выключено;
- короткое описание для родителя.

## Приоритеты

Расписание применяется, только если ни одно правило с более высоким приоритетом не определило доступ.

Приоритет:

1. Аварийно-полезные сайты.
2. Группа «Без ограничений».
3. Чёрный список.
4. Белый список.
5. Глобальное отключение интернета.
6. Временное разрешение.
7. Расписание отдельного устройства.
8. Расписание группы.
9. Обычный доступ.

В стандартном порядке временное разрешение перекрывает расписания, но находится ниже чёрного и белого списков устройств.

Нужны и правила блокировки, и правила разрешения.

## Временный доступ

Быстрые кнопки:

```text
+15 минут
+30 минут
+1 час
+2 часа
+3 часа
+5 часов
до конца суток
до отбоя
```

Время отбоя по умолчанию:

```text
21:00
```

Время отбоя должно настраиваться внутри раздела расписаний.

## Время

Нужно поддержать интервалы, пересекающие полночь.

Пример:

```text
22:00 -> 07:00
```

Такое правило означает вечер текущего дня и утро следующего дня.

## Конфликты

Если два расписания конфликтуют, интерфейс должен показать предупреждение.

Backend должен иметь детерминированное поведение:

- результат конфликта расписаний одного уровня задаёт `sheepfold.global.schedule_conflict_internet`: `off` по умолчанию оставляет интернет выключенным, `on` — включённым;
- более конкретное правило устройства сильнее группового правила;
- настройка результата не скрывает конфликт: LuCI всё равно показывает предупреждение, а backend записывает изменение конфликта в журнал;
- вычисление выполняется после загрузки роутера, сохранения правил, изменения группы, времени/часового пояса и перезагрузки firewall;
- интервалы через полночь и переходы летнего времени не должны оставлять старое состояние;
- в журнал записывается изменение результата, а не каждая минутная проверка;
- Сейчас UI только показывает фиксированный порядок, который одинаково воспроизводят status API и nftables. Редактирование `access_priority` должно оставаться выключенным до появления единого runtime-вычислителя для всех трёх частей.

В `Настройки → Разное` настройка показывается радиогруппой «При конфликте правил включения и отключения интернета между расписаниями интернет будет: Выключен / Включен». Она сохраняется только общей кнопкой `Сохранить настройки`.

## UI

В LuCI и Android нужны:

- список правил;
- быстрый переключатель правила;
- редактор дней недели;
- редактор времени;
- предпросмотр "что будет сейчас";
- подсказка, почему устройство сейчас заблокировано или разрешено.

## Логи

В журнал писать только факт срабатывания расписания:

```text
schedule applied: rule name, target device display name, action, result
```

Не писать историю сайтов, DNS-запросы или лишние технические данные.
