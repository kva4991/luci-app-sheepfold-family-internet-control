# Сообщения и уведомления

Sheepfold должен поддерживать двухстороннее управление через мессенджер.

Поддерживаемые адаптеры:

- Telegram;
- VK;
- MAX, экспериментально.

Отдельное направление для голосовых помощников описано в `docs/voice-assistants.ru.md`. Алиса, Маруся и похожие помощники должны использовать тот же безопасный слой команд Sheepfold, но не должны открывать роутер в интернет напрямую.

На одном роутере одновременно может быть активен только один мессенджер (§r2vy5mk).

## Правило активного мессенджера (§r2vy5mk)

- VK — вариант по умолчанию при первичной настройке мессенджера.
- Установленный конфиг должен держать мессенджер выключенным, пока не введены данные доступа и не привязан хотя бы один разрешённый администратор.
- На роутере может быть включён только один адаптер: `telegram`, `vk` или экспериментальный `max`.
- Переключение активного мессенджера должно требовать подтверждения администратора (§m1fqo8a).
- Отключение текущего мессенджера не должно сразу удалять его настройки; их можно оставить для быстрого повторного включения, пока пользователь явно не удалит конфигурацию.
- Все действия мессенджера должны проходить через тот же Sheepfold API, который используют LuCI и Android.
- Опасные действия должны требовать явного подтверждения в выбранном мессенджере (§m1fqo8a).

## Двухсторонний чат в Telegram

Telegram должен поддерживать и уведомления, и интерактивное управление родителем.

Пошаговая пользовательская инструкция по созданию Telegram-бота лежит в `docs/telegram-bot-setup.ru.md`.

Минимальная текущая реализация:

- локальный `procd`-процесс `/usr/libexec/sheepfold/sheepfold-telegram-bot`;
- исходящий long polling через Telegram Bot API, без webhook и без публичного адреса роутера (§u4hnqle);
- настройки читаются из `sheepfold.global.active_messenger`, `telegram_bot_token`, `telegram_admin_chat_id`;
- если Telegram включён и токен задан, но chat ID ещё не указан, бот отвечает написавшему пользователю его chat ID для ручного ввода в LuCI;
- команды принимаются только от сохранённого `telegram_admin_chat_id`;
- Telegram-бот сам регистрирует меню команд через Bot API `setMyCommands` при запуске адаптера и при тестовом сообщении;
- команды, которые Telegram показывает в меню, должны быть slash-командами на латинице: `/start`, `/help`, `/status`, `/devices`, `/internet_on`, `/internet_off`, `/wifi_on`, `/wifi_off`, `/support`, `/grant_time`, `/block_device`, `/unblock_device`, `/allowlist_add`, `/blocklist_add`, `/logs`, `/clear_logs`, `/update`, `/reboot`, `/emergency_sites`, `/wifi_status`;
- русские сообщения тоже должны работать как алиасы: `помощь`, `статус`, `показать все устройства`, `включить интернет`, `отключить интернет`, `выключить интернет`, `включить вайфай`, `выключить вайфай`, `дать #3 30 минут`, `заблокировать #3`, `разблокировать #3`, `добавить #3 в белый список`, `добавить #3 в чёрный список`, `журнал`, `очистить журнал`, `обновить приложение`, `перезагрузить роутер`, `аварийно-полезные сайты`, `саппорт`, `сапорт`, `поддержка`;
- новые строки журнала могут отправляться разрешённому администратору как уведомления;
- опасные команды вроде перезагрузки, обновления и очистки журнала требуют отдельного текстового подтверждения с коротким кодом (§m1fqo8a).

Планируемые события:

- включена глобальная блокировка;
- выключена глобальная блокировка;
- найдено новое неизвестное устройство;
- устройство добавлено в белый список;
- устройство добавлено в чёрный список;
- выдан временный доступ;
- сработало расписание;
- импорт/экспорт завершён;
- запрошена перезагрузка роутера;
- обновление приложения завершилось или упало;
- предупреждение интеграции с AdGuard Home;
- предупреждение интеграции с Podkop;
- ошибка firewall/rpcd/service;
- запрос ребёнка на доступ, если такой сценарий будет добавлен позже.

Если включён LED-сигнал о новом устройстве, просмотр уведомления о новом устройстве любым разрешённым администратором в мессенджере должен отправлять backend-событие подтверждения просмотра и сбрасывать LED-сигнал на дефолтное поведение роутера.

Интерактивные возможности:

- показать текущий статус;
- показать все устройства;
- искать устройства по имени, IP или MAC;
- включить/выключить глобальную блокировку с подтверждением (§m1fqo8a);
- выдать временный доступ;
- добавить/удалить устройство из белого или чёрного списка;
- показать состояние Wi-Fi;
- включить Wi-Fi;
- выключить Wi-Fi;
- изменить автоматическое включение/выключение Wi-Fi;
- одобрить или отклонить запрос ребёнка на доступ;
- перед перезагрузкой, обновлением или импортом настроек задавать вопрос с подтверждением (§m1fqo8a);
- принимать короткие естественные команды на русском языке.

## MAX Experimental

MAX можно оставить как экспериментальный адаптер.

Требования:

- не включать по умолчанию;
- не считать блокером первого стабильного релиза;
- использовать тот же внутренний интерфейс адаптеров;
- показывать в UI пометку `experimental`;
- не открывать роутер в интернет.

Архитектура:

- Telegram и VK реализовать отдельными адаптерами;
- код мессенджера не смешивать с firewall-логикой OpenWRT;
- все действия проводить через тот же Sheepfold API, который используют LuCI и Android;
- права команд явно ограничивать списком разрешённых пользователей.
- разрешённых администраторов назначать в настройках роутера;
- у каждого администратора должен быть привязанный Telegram/VK/MAX user ID или chat ID; роль в текущем MVP не обязательна;
- логировать административные действия: кто выполнил команду, что изменил, когда и с каким результатом.

## Реализация на OpenWRT

Бот должен работать как локальный сервис на роутере через `procd`.

Предпочтительная схема для Telegram (§u4hnqle):

- не открывать входящий порт на роутере;
- не требовать белый IP;
- использовать Telegram Bot API через исходящие HTTPS-запросы;
- получать сообщения через long polling `getUpdates`;
- отправлять ответы через HTTPS API;
- хранить offset последнего обработанного update локально;
- пропускать команды от неизвестных user ID / chat ID.

Эта схема подходит для домашнего роутера лучше, чем webhook, потому что webhook потребовал бы публичный HTTPS endpoint.

Для VK использовать тот же интерфейс адаптера. VK более обоснован как второй стабильный адаптер, чем MAX, потому что экосистема VK-ботов зрелее. MAX можно оставить экспериментальным.

## Голосовые помощники

Голосовые помощники вроде Алисы и Маруси можно добавить как отдельные коннекторы поверх Sheepfold API.

Принципы:

- не считать голосовой помощник полноценным активным мессенджером: он не должен вытеснять Telegram/VK как канал уведомлений;
- использовать общий обработчик команд, чтобы `отключить интернет`, `статус`, `дать #3 30 минут` работали одинаково в боте, Android и голосовом коннекторе;
- не открывать входящий порт роутера для webhook из интернета (§u4hnqle);
- опасные действия подтверждать в Android, LuCI или активном мессенджере (§m1fqo8a);
- учитывать, что голосовую команду могут услышать дети или гости, поэтому добавление в белый список, WPS, перезагрузка, обновление и очистка журнала не должны выполняться только по голосу.

Минимальный набор зависимостей для OpenWRT:

```text
uclient-fetch
ca-bundle
jsonfilter
```

Если позже shell-реализация станет слишком хрупкой, бот лучше перенести на `ucode`/`rpcd`-backend, но внешний API Sheepfold менять не нужно.

Полезные ссылки:

- Telegram Bot API: https://core.telegram.org/bots/api
- Telegram `getUpdates`: https://core.telegram.org/bots/api#getupdates
- VK Developers: https://dev.vk.com/

## Общие требования

- интеграция с мессенджером должна быть необязательной;
- токены ботов хранить безопасно (§pw7z6nt);
- разрешить доверенные chat ID / user ID;
- не отправлять пароли роутера, API-токены, session ID, полные MAC-адреса, приватные заметки, полные списки устройств и другие чувствительные сетевые данные без явного разрешения (§pw7z6nt);
- по умолчанию в сообщениях показывать понятное имя устройства и статус, а технические детали раскрывать только по запросу администратора;
- опасные действия должны требовать подтверждения: перезагрузка роутера, обновление, импорт настроек, очистка журнала, массовая блокировка, удаление правил, смена активного мессенджера, добавление нового администратора, включение/выключение всего Wi-Fi, изменение расписания Wi-Fi, изменение поведения WPS-кнопки (§m1fqo8a);
- учитывать настройку журнала: если журнал отключён, живые уведомления можно отправлять, но локальную историю событий не писать без явного включения.
